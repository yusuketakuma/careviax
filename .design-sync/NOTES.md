# design-sync NOTES (careviax / PH-OS Design System)

careviax is a **Next.js application, not a packaged component library** — so this
sync runs the package shape in **synth-entry mode** with several repo-specific
workarounds. Read this before any re-sync.

## Setup facts

- Project: `PH-OS Design System` (claude.ai/design), id pinned in config.json.
- Shape: `package`. No Storybook, no `dist/`, `package.json` is private with no `main`/`exports`.
- Scope: the 33 UI primitives in `src/components/ui` (pinned in `cfg.componentSrcMap`).
  `createRouteErrorBoundary` (a factory, not a component) is intentionally not listed;
  `sonner.tsx`'s real export is `Toaster`.
- `cfg.srcDir` = `src/components/ui` — **must stay scoped**. Pointing it at `src`
  makes the synth entry `export *` the whole 1200-file app → esbuild OOM.

## Build invocation (NOT the default from the SKILL)

The default `package-build.mjs ... --entry ./dist/...` does NOT apply. Use:

```
npx --yes @tailwindcss/cli@4.2.2 -i src/app/globals.css -o .design-sync/.cache/compiled-tailwind.css   # cfg.buildCmd
NODE_OPTIONS=--max-old-space-size=6144 node .ds-sync/package-build.mjs \
  --config .design-sync/config.json --node-modules node_modules \
  --entry .design-sync/.cache/ds-entry.tsx --out ./ds-bundle
```

- **`--entry .design-sync/.cache/ds-entry.tsx`** is a hand-written synthetic entry
  (`export *` from every `src/components/ui/*.tsx` non-test file). Regenerate it on
  re-sync if components are added/removed (the generator one-liner is in git history /
  re-derivable: list non-test `.tsx` in `src/components/ui`, `export *` each by absolute path).
  Using `--entry` (not bare synth) is REQUIRED so PKG_DIR resolves via walk-up to the repo
  root WITHOUT a `node_modules/ph-os` self-symlink.
- **Do NOT create a `node_modules/ph-os` self-symlink.** It makes PKG_DIR = repo root but
  causes the dts `.d.ts` glob (and any repo-root walk) to follow `node_modules/ph-os → ..`
  in an infinite cycle (`node_modules/ph-os/node_modules/ph-os/...`) → OOM. The `--entry`
  approach avoids it.
- Heap: `--max-old-space-size=6144` (synth bundling + ts-morph).

## CSS (Tailwind 4 CSS-first)

- `cfg.cssEntry` = `.design-sync/.cache/compiled-tailwind.css` — a Tailwind 4 CLI compile of
  `src/app/globals.css` (theme tokens + utilities). It is gitignored (regenerated by `cfg.buildCmd`).
  This becomes `_ds_bundle.css` in the bundle; `styles.css` @imports it. Verified: tokens (oklch)
  + utilities present, CSS closure resolves.

## Stubs (esbuild path aliases via cfg.tsconfig = .design-sync/tsconfig.dssync.json)

The scoped tsconfig stubs the **only two boundary packages** that drag Next/server code into
the browser bundle (confirmed via esbuild metafile boundary-edge trace — these are the complete set):

- `next/link` → `.design-sync/next-link-stub.tsx` (renders `<a>`). Used by error-state, empty-state.
  Real next/link pulls next internals → gzip-size → fs/stream/zlib.
- `@sentry/nextjs` → `.design-sync/sentry-stub.tsx` (no-op captureException etc.). Imported by
  `route-error-boundary.tsx` (a non-component factory, but the synth entry `export *`s it).
  Real @sentry/nextjs pulls next/router + next/constants → same node-builtin failure.
- The scoped tsconfig deliberately has **only these two paths, NOT `@/*`** — adding `@/*` makes the
  converter's tsconfig-paths plugin intercept `@/lib/utils` and resolve it to the `src/lib/utils/`
  DIRECTORY (a `utils.ts` file AND a `utils/` dir coexist; the plugin's empty-extension check matches
  the dir first). With only the stubs in the plugin, `@/*` falls through to esbuild's native tsconfig
  auto-detection (repo `tsconfig.json`), which prefers `utils.ts` correctly.
- **Never put a `"//"` comment key (or any string containing `//`) in tsconfig.dssync.json** — the
  converter's comment-stripping regex corrupts it → JSON.parse fails → plugin silently disabled.

## Fonts

- `[FONT_MISSING] "Yu Gothic Medium"` is EXPECTED and suppressed via
  `cfg.runtimeFontPrefixes = ["Meiryo","Hiragino","Yu Gothic"]`. The DS font stack is
  Meiryo/Hiragino/Yu Gothic (OS-provided Japanese system fonts, per docs/ui-ux-design-guidelines.md) +
  Noto Sans JP (via next/font) + system-ui. These are intentionally host-provided, never bundled.

## Re-sync risks (watch-list)

- `compiled-tailwind.css` and `ds-entry.tsx` live in gitignored `.cache/` — re-sync MUST regenerate
  both (cssEntry via cfg.buildCmd; ds-entry by re-listing src/components/ui). If a component is added,
  it won't appear until ds-entry + componentSrcMap are updated.
- If a NEW ui component imports another Next-coupled or node-builtin package (beyond next/link,
  @sentry/nextjs), the build will fail with `Could not resolve "fs"`-style errors. Re-run the
  metafile boundary-edge trace (mark node builtins external, find non-next→next edges) to find it,
  then add a browser-safe stub to tsconfig.dssync.json.
- Render check needs playwright@1.60.0 (installed in .ds-sync, browser DL skipped) + cached
  chromium-1223 (revision matches 1.60.0). No 200MB install needed on this machine.

## Known render warns (triaged, not new)

- 6 simple components (Alert, Button, Checkbox, Input, LoadingButton, Textarea) flagged
  `[RENDER_BLANK]` as FLOOR CARDS only because an empty/childless instance renders tiny.
  Authoring their previews (real content) resolves this — not a defect.

## Preview authoring learnings (fanout, 30 components)

- **Import shim**: previews import from `'ph-os'` → live bundle (window.PhOs). Layout glue via inline `style`; component uses real props.
- **base-ui overlays render CLOSED by default.** To render OPEN in a static card: pass `defaultOpen` on the Root, and `modal={false}` (so it doesn't inert/focus-trap the page during capture). Applies to Dialog, AlertDialog, Sheet, ConfirmDialog (wraps AlertDialog), Select.
  - **Sheet**: also set `showOverlay={false}` on SheetContent so the dimming backdrop doesn't gray the card.
  - **Select**: pass an `items={{value: label}}` map on the Root (real base-ui prop) so the closed trigger shows the human label not the raw value; give the trigger `className="w-full"` inside a fixed-width wrapper so the `--anchor-width` popup doesn't clip long labels; `alignItemWithTrigger={false}` stacks the popup below.
- **Description sub-components are HelpPopover ("?") not inline text**: CardDescription, DialogDescription, SheetDescription. (AlertDialogDescription IS plain inline text.) Keep body copy in separate elements.
- **Token-driven (no `variant` enum)**: StateBadge/StatusDot take `role` (StatusRole from src/lib/constants/status-tokens.ts); SegmentedProgressBar uses value/max + filledClassName + markerValue. LoadingButton: pass `loading` for a static spinner+label state. Switch: stateless, pass `checked`. Checkbox: `defaultChecked`/`checked`.
- **Tabs**: `defaultValue` renders the panel open statically; variant axis = TabsList `variant` ("default" pills vs "line" underline).

### Capture limitations (cfg.overrides / known)

- **DropdownMenu** → `cfg.overrides.DropdownMenu = {cardMode:'single', viewport:'480x560', primaryStory:'ActionsMenu'}`. base-ui Menu uses Floating-UI absolute positioning portaled to document.body anchored to the trigger; the default per-cell crop excludes it (blank). Unlike Dialog/Sheet/AlertDialog which fixed-center to the viewport. Same treatment will be needed for any base-ui Menu/Combobox/Popover anchored-portal component. No repo usages of DropdownMenu exist, so previews authored from the API.
- **HelpPopover**: no controllable open prop (click/hover only, fixed panel from getBoundingClientRect). Authored as the inline "?" trigger next to field labels (its real usage) — the open panel can't render statically without interaction scripting. Trigger is fully DS-styled. Left as trigger-only (good).
- **Toaster (sonner)**: needs imperative `toast()` calls to show anything; static capture is empty. Authored a faithful recreation of the 4 toast tones (success/info/warning/error) using DS tokens + lucide icons, alongside the real `<Toaster/>`. Depends on next-themes useTheme (falls back to 'system' fine in the harness).
