## 確定した判断 (ユーザー承認, 2026-06-29)

1. **服薬期限 = MIN** — 最早に切れる継続薬の end_date が拘束。頓服/as-needed(PRN) は除外。refill/split/next_visit_suggestion_date も MIN で畳む。`datetime-1` の二重 -1 は撤去(期限=その MIN 日, inclusive)。(P1/meddate-1)
2. **算定の週 = 日曜〜土曜** — 診療報酬標準。weekStartsOn:0 へ。現行 ISO 月曜始まりから変更、UI/test の月曜前提も更新。(P3/caps-4)
3. **override 再申請 = 部分ユニーク index migration を承認** — `VisitScheduleOverride.source_schedule_id` の @unique を `WHERE status IN ('pending','approved')` の partial unique へ(raw-SQL migration)。cancelled 履歴保持。**唯一の承認済みスキーマ変更**。(P9/emergency-2)
4. **未ジオコーディング = 警告付き暫定配置**(fail-closed ではない) — 座標欠落は直線距離で配置し「推定/座標未設定」を明示。明らかに不正な座標(0,0/範囲外/swap)は app-layer zod で reject(migration 不要、Residence CHECK 制約は不要)。(P5/dataquality-1/2)

> Residence lat/lng の CHECK 制約(任意スキーマ変更)は **不採用**(app-layer zod で代替)。承認済みスキーマ変更は #3 の partial-unique のみ。

## Executive Summary
Refactor (not rewrite) the careviax visit-scheduling/route subsystem so it (a) stops silently corrupting dates/times under JST, (b) derives medication-driven deadlines correctly, (c) enforces caps/constraints transactionally, and (d) turns the route engine into a single time-window + dwell-aware ordering authority that both the planner (computeRouteInsertion) and /api/visit-routes share. The dominant systemic root cause is timezone mismatch: @db.Date/@db.Time values are written/derived with LOCAL Date constructors but read with getUTC*; on the Tokyo-region (ap-northeast-1) hosts and JST dev/CI this off-by-one corrupts rrule occurrences, stored visit time windows, and week bucketing (datetime-1..4, rrule-1/5/6, caps-4). The second root cause is constraint/cap checks that run OUTSIDE the serializable write transaction and only over the current request's candidates (concurrency-2/5, rls-1), letting concurrent generates overbook vehicles and exceed billing caps. The third is constraint enforcement asymmetry: max_stops is HARD in the planner but operating-hours-time, vehicle duration, conflicting preference windows, and travel-minute units are silently SOFT/wrong (constraints-1/2/4/5, dataquality-2/3). Ordering: P0 datetime foundation underpins everything; P1 deadline/meddate correctness and P2 rrule correctness are pure-correctness; P3 caps + P4 transaction/authz integrity close billing/compliance holes; P5 hardens planner constraints; P6 is the target route-engine refactor (timeWindow+dwell+2-opt+priority-bands+shared scoreInsertion) that P10 FE depends on; P7 perf, P8 audit, P9 override lifecycle; P10 FE triage screens (the only stage Claude owns end-to-end). Only TWO DB schema changes are required (both flagged, both need user approval): a partial-unique on VisitScheduleOverride.source_schedule_id to unblock re-reschedule (emergency-2/5), and optional Residence lat/lng CHECK bounds (dataquality-2, app-layer zod preferred instead). All VisitRouteWaypoint/VisitRoutePlan enrichments are TypeScript-only (no migration). Lane split: backend (planner/engine/rrule/api/validator/audit) = Codex; FE triage screens = Claude; shared src/types/visit-route.ts is the BE+FE coordination seam. Mutual review per slice per .agent-loop maker/checker discipline.
