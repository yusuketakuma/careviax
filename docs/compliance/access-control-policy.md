# C-01 — アクセス制御ポリシー

## 概要

PH-OS における論理アクセス制御・物理アクセス制御・特権アクセス管理の方針を定義する。
3省2ガイドライン（MHLW v6.0 §6 + METI/MIC v1.1 §5）に準拠。

---

## 1. アカウント種別 / ロール定義

PH-OS は、薬局テナント内の `MemberRole` と、PH-OS 運営者用の
`PlatformOperatorRole` を分離して権限管理する。実装上の正本は以下。

- テナント内ロール enum: `prisma/schema/organization.prisma` の `MemberRole`
- capability matrix: `src/lib/auth/permission-matrix.ts`
- 表示名・運用分類: `src/lib/auth/member-roles.ts`
- PH-OS 運営者ロール enum: `prisma/schema/platform.prisma` の `PlatformOperatorRole`

### 1.1 テナント内 `MemberRole`

| ロール       | 識別子               | 概要                                                                                                               | 主な利用者                 |
| ------------ | -------------------- | ------------------------------------------------------------------------------------------------------------------ | -------------------------- |
| 責任者       | `owner`              | テナント内の全操作権限。ユーザー管理、設定、調剤/監査/訪問/報告/請求/共有を実行できる。                            | 薬局責任者                 |
| 管理者       | `admin`              | `owner` と同等の業務権限。運用管理を担当する。                                                                     | 薬局管理者                 |
| 薬剤師       | `pharmacist`         | 患者管理、処方受付、調剤、監査、訪問記録、報告書作成・送付、請求、患者共有を実行できる。                           | 管理薬剤師・勤務薬剤師     |
| 研修薬剤師   | `pharmacist_trainee` | 調剤/セット/訪問/報告作成は可能。監査、報告書送付、請求管理、患者共有管理は不可。                                  | 研修中薬剤師               |
| 事務スタッフ | `clerk`              | ダッシュボード閲覧と報告/連携/事務系の参照・補助が可能。臨床報告書作成、調剤、訪問、請求管理、患者共有管理は不可。 | 事務担当者                 |
| 配送担当     | `driver`             | 現行 capability matrix では業務機能への権限を持たない。配送専用 UI/assignment は別途設計が必要。                   | 配送担当者                 |
| 外部連携者   | `external_viewer`    | テナントメンバーとしての業務機能権限は持たない。外部共有リンク等の token/scope ベース閲覧は別経路で制御する。      | 医師・訪問看護師等の連携先 |

### 1.2 PH-OS 運営者 `PlatformOperatorRole`

PH-OS 運営者はテナント内 `MemberRole` ではない。全テナント横断操作は
`PlatformOperator` + `BreakGlassSession` によって、対象テナント、理由、MFA、
期限、scope、監査ログを必須にして実行する。

| ロール   | 識別子             | 概要                                                                   |
| -------- | ------------------ | ---------------------------------------------------------------------- |
| サポート | `platform_support` | time-boxed break-glass による read-only 横断支援。                     |
| 管理者   | `platform_admin`   | read + 限定 write の運用支援。高リスク操作は追加確認を要する。         |
| 所有者   | `platform_owner`   | platform operator 管理を含む最上位運営者権限。日常業務では使用しない。 |

## 2. Capability matrix（ロール × 機能）

PH-OS の実装はリソース単位 CRUD ではなく capability key で認可する。下表は
`src/lib/auth/permission-matrix.ts` の現行値に合わせる。個別 API はこの
capability に加えて org/RLS、case assignment、patient/case scope、consent、
support session、purpose、precondition、audit を必要に応じて重ねる。

凡例: ✅=許可、—=不可。

| Capability                                     | owner | admin | pharmacist | pharmacist_trainee | clerk | driver | external_viewer |
| ---------------------------------------------- | ----: | ----: | ---------: | -----------------: | ----: | -----: | --------------: |
| `canViewDashboard` ダッシュボード閲覧          |    ✅ |    ✅ |         ✅ |                 ✅ |    ✅ |      — |               — |
| `canDispense` 調剤                             |    ✅ |    ✅ |         ✅ |                 ✅ |     — |      — |               — |
| `canAuditDispense` 調剤監査                    |    ✅ |    ✅ |         ✅ |                  — |     — |      — |               — |

### 2.1 ダッシュボード / 業務コックピットの表示方針

`canViewDashboard=true` のロールは、認証済み業務画面で自分の権限範囲に属する
患者名、薬剤名、残数、MCS/電話本文、連絡先、添付導線、訪問内容、処方内容、
報告/請求の具体情報を閲覧できる。情報を隠すかどうかではなく、role、
assignment、case scope、consent、support session、purpose により閲覧範囲を
決める。

一覧密度は UI で制御する。カードや行では要約し、展開行、右プレビュー、Drawer、
詳細画面では権限内情報を全て表示してよい。OS 通知、SSE payload、監査差分、
server log、外部共有、CSV/PDF export、public URL は別の出力境界として最小化・
マスクする。

### 2.2 外部共有 / token scope

`external_viewer` はテナントメンバーの capability を持たない。医師、訪問看護師、
ケアマネジャー、家族等への外部共有は `external_viewer` のテナントログイン権限
ではなく、外部共有 link / token / OTP / scope / consent / expiry / audit による
別経路で制御する。

### 2.3 未実装または計画中の横断アクセス

フリーランス薬剤師、複数薬局所属、PH-OS 運営者 support mode は、
`Membership`、`CaseAssignment`、`CrossTenantAccessGrant`、`BreakGlassSession`
を組み合わせて実装する。これらは `MemberRole` だけで許可しない。導入時は本書、
`src/lib/auth/permission-matrix.ts`、route tests、RLS proof、audit log を同じ
slice で更新する。

## 3. 物理アクセス制御

### 3.1 AWS データセンター（ISMAP 準拠）

AWS の東京リージョン（ap-northeast-1）データセンターは AWS が物理セキュリティを管理し、ISMAP 認定を取得している。PH-OS のインフラ（RDS/S3/Cognito/SES）はすべて AWS データセンター内に存在するため、物理アクセス制御は AWS の責任分担（AWS Shared Responsibility Model）に基づく。

PH-OS が確認すべき事項:

- AWS SOC 2 Type II レポート（年次取得・確認）
- ISMAP クラウドサービスリスト（AWS 掲載確認）
- AWS Artifact でのコンプライアンスレポート取得（年 1 回）

### 3.2 薬局オフィス（エンドポイント物理セキュリティ）

| 管理項目   | 要件                                                                           |
| ---------- | ------------------------------------------------------------------------------ |
| 入退室管理 | 薬局施設規定に従う（調剤室は薬剤師以外の立入を制限）                           |
| 端末管理   | 業務端末は施錠可能な保管場所に格納。離席時は画面ロック必須（5 分タイムアウト） |
| 不審者対策 | 調剤室・サーバー機器への外部者立入禁止。入館記録の保持                         |
| 廃棄       | 記録媒体の廃棄はデータ消去後に実施（NSA/CSS EPL 準拠またはベンダー証明書取得） |

## 4. 特権アクセス管理（PAM）

### 4.1 対象ロール

- `platform_owner` / `platform_admin` / `platform_support`: PH-OS 全テナントに対する運営者権限。`BreakGlassSession` なしに tenant data を横断閲覧しない。
- AWS IAM 管理者: インフラ変更権限（CDK デプロイ、RDS 設定変更等）
- RDS 管理ユーザー: `rds_superuser` 相当の DB 直接アクセス

### 4.2 PAM 要件

| 要件                   | 詳細                                                                                                                       |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| MFA 必須               | platform operator ロールへのログイン時および AWS IAM Console アクセス時は TOTP/FIDO2 必須                                  |
| 最小権限               | 日常業務では `owner` / `admin` / `pharmacist` 以下のテナントロールを使用。platform operator は運営支援・障害対応時のみ使用 |
| セッションログ         | platform operator 操作は監査ログ（AuditLog / BreakGlassSession）に全記録。AWS CloudTrail で IAM 操作を全記録               |
| セッションタイムアウト | platform operator / break-glass session: 30 分以下。通常ユーザーセッション: 8 時間（業務時間帯）                           |
| アクセス経路           | RDS 直接アクセスは AWS Systems Manager Session Manager 経由のみ（踏み台 SSH 禁止）                                         |
| 作業記録               | 特権作業実施時は作業内容・目的・時間を変更管理チケットに記録                                                               |
| 緊急アクセス           | Break Glass 手順（BREAK_GLASS_PROCEDURE.md 参照）に従い、事後レビューを必須とする                                          |

### 4.3 AWS IAM ポリシー原則

- ルートアカウントの日常利用禁止（MFA 設定後、認証情報を安全管理）
- IAM ユーザーではなく IAM ロールを使用（EC2/Lambda/Amplify への権限付与）
- 管理者 IAM ユーザーは Secrets Manager に認証情報を保管
- `AdministratorAccess` ポリシーの付与は最小限のユーザーに制限
- CloudTrail による全 API 操作の記録（改ざん防止: S3 Object Lock 有効）

## 5. アカウントライフサイクル

### 5.1 作成

| フロー   | 手順                                                                                                               |
| -------- | ------------------------------------------------------------------------------------------------------------------ |
| 申請     | 組織管理者が PH-OS 管理画面からユーザー招待を実施                                                                  |
| 承認     | 組織管理者が招待送信（メール経由）                                                                                 |
| 初期設定 | 招待メールのリンクから Cognito 経由でパスワード設定、MFA 設定（薬剤師・研修薬剤師・platform operator は MFA 必須） |
| 記録     | AuditLog に `user_created` イベントを記録                                                                          |

### 5.2 変更

| フロー     | 手順                                                                                                                                      |
| ---------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| ロール変更 | 組織管理者が管理画面から変更。変更理由をメモとして記録                                                                                    |
| 所属変更   | 組織管理者が実施。複数組織への兼務、assignment、cross-tenant grant には platform operator または `owner` / `admin` の承認と監査記録が必要 |
| 記録       | AuditLog に `user_role_changed` イベントを記録                                                                                            |

### 5.3 停止・退職

| フロー           | 手順                                                               |
| ---------------- | ------------------------------------------------------------------ |
| 停止             | 組織管理者が管理画面でアカウントを無効化（Cognito: `DisableUser`） |
| タイミング       | 退職日当日または連絡受領後 24 時間以内に停止                       |
| セッション無効化 | Cognito グローバルサインアウトを実行（既存セッションを全て無効化） |
| 記録             | AuditLog に `user_disabled` イベントを記録                         |

### 5.4 削除

| フロー         | 手順                                                                               |
| -------------- | ---------------------------------------------------------------------------------- |
| 削除実施       | アカウント停止から 30 日後、組織管理者が削除申請                                   |
| 操作記録の保持 | ユーザー削除後も AuditLog の過去操作記録は保持（5 年間）                           |
| 個人情報削除   | 退職者の個人情報（メールアドレス等）は APPI 対応で削除、操作ログとの紐付けは匿名化 |

## 6. 定期レビュー

### 6.1 権限棚卸し（四半期ごと）

実施頻度: **四半期ごと（1月・4月・7月・10月）**

棚卸し手順:

1. 組織管理者がユーザー一覧を管理画面からエクスポート
2. 各ユーザーの現在のロールと業務上の必要性を照合
3. 不要な権限の削除または降格を実施
4. 棚卸し結果を記録（実施日・担当者・変更内容）

棚卸しの確認ポイント:

- 異動・退職者のアカウントが停止済みであること
- ロールが現在の業務に適切であること
- 長期未ログインアカウント（90 日以上）の確認
- `owner` / `admin` 以上の権限を持つユーザー数の妥当性確認

### 6.2 特権アクセス棚卸し（年 1 回）

- `platform_owner` / `platform_admin` / `platform_support` ロールを持つユーザーの必要性確認
- AWS IAM ロール・ポリシーの棚卸し
- Secrets Manager に保管されている認証情報の棚卸し

## 7. Cognito 設定要件

| 設定項目                     | 要件値                                                                                                                  |
| ---------------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| パスワードポリシー           | 12 文字以上、大文字・小文字・数字・記号を各 1 文字以上                                                                  |
| MFA                          | 薬剤師・研修薬剤師・platform operator: 必須（TOTP/FIDO2）。事務スタッフ・配送担当・外部連携者: 任意または組織設定に従う |
| Advanced Security            | `ENFORCED`（不審なサインインの自動ブロック）                                                                            |
| アクセストークン有効期限     | 1 時間                                                                                                                  |
| リフレッシュトークン有効期限 | 30 日                                                                                                                   |
| パスワード有効期限           | 90 日（期限切れ時は強制変更）                                                                                           |
| ログイン試行失敗上限         | 5 回（以降 30 分ロックアウト）                                                                                          |
