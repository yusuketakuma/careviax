# C-02 — 変更管理・リリース手順書

## 概要

PH-OS のソフトウェア変更（コード変更・インフラ変更・設定変更）に関する管理手順を定義する。
3省2ガイドライン（MHLW v6.0 §7 + METI/MIC v1.1 §6）が要求する変更管理プロセスへの準拠を確保する。

---

## 1. 変更の分類

| 分類 | 定義 | 例 | 承認フロー |
|---|---|---|---|
| **標準変更** | 既定手順で安全に実施できる反復的変更 | 依存パッケージのマイナーアップデート、テスト追加 | PR レビュー 1 名 |
| **通常変更** | 機能追加・修正・改善 | 新機能実装、バグ修正、UI 変更 | PR レビュー 1 名 + ステージング検証 |
| **重要変更** | リスクの高い変更 | DBスキーマ変更、認証・認可変更、API 破壊的変更 | PR レビュー 2 名 + ステージング検証 + 本番前承認 |
| **緊急変更** | 本番障害対応 | 重大バグ修正、セキュリティパッチ | 事後レビュー（実施後 24 時間以内） |

---

## 2. コードレビュー要件

### 2.1 基本要件

| 要件 | 内容 |
|---|---|
| レビュー人数 | 通常変更: 1 名以上。重要変更: 2 名以上 |
| セルフマージ禁止 | `main` ブランチへの直接 push は禁止（Branch Protection Rule で強制） |
| CI 必須 | GitHub Actions による自動チェック（Lint + Test + Build）が全て通過すること |
| レビュアー資格 | 変更対象領域の知識を持つチームメンバー（コード変更の場合）|

### 2.2 コードレビューチェックリスト

レビュアーは以下を確認する:

**機能・品質**
- [ ] 要件・仕様と整合している
- [ ] エラーハンドリングが適切
- [ ] テストが追加・更新されている

**セキュリティ**
- [ ] 入力バリデーション（Zod スキーマ）が実装されている
- [ ] 認証・認可チェックが Route Handler に実装されている
- [ ] 機密情報がハードコードされていない
- [ ] RLS セッション変数 (`app.current_org_id`) が適切に設定されている
- [ ] SQL インジェクション対策（Prisma の型付きクエリを使用）

**データ整合性**
- [ ] Prisma スキーマ変更は migration ファイルとセットで提出
- [ ] 既存データへの破壊的影響がない（または移行スクリプトがある）
- [ ] 監査ログ（AuditLog）への記録が必要な操作に記録処理が実装されている

## 3. 承認フロー（開発 → ステージング → 本番）

```
開発ブランチ (feature/xxx)
        ↓
     PR 作成
        ↓
  自動 CI チェック
  (Lint / Test / Build)
        ↓
  コードレビュー
  (1〜2名承認)
        ↓
  main ブランチへマージ
        ↓
  Amplify: ステージング自動デプロイ
        ↓
  ステージング検証
  (機能確認・回帰テスト)
        ↓
  本番デプロイ承認
  (重要変更は承認者のサインオフ)
        ↓
  Amplify: 本番デプロイ
        ↓
  本番確認・監視
```

### 3.1 ブランチ戦略

| ブランチ | 用途 | デプロイ先 |
|---|---|---|
| `main` | 本番リリース済みコード | 本番環境（自動） |
| `develop` | 統合テスト・ステージング検証 | ステージング環境（自動） |
| `feature/xxx` | 機能開発・バグ修正 | なし（PR マージで develop へ） |
| `hotfix/xxx` | 緊急修正 | 本番環境（事後レビュー） |

## 4. デプロイ手順（AWS Amplify）

### 4.1 通常デプロイ

Amplify は `main` ブランチへのマージで自動デプロイをトリガーする。

```
git push origin main
  ↓
Amplify Hosting が自動検知
  ↓
ビルド実行 (pnpm build)
  ↓
デプロイ（ゼロダウンタイム: 新旧バージョンをトラフィック切替）
  ↓
ヘルスチェック (/api/health) と通常 API 経路の rate-limit backend 確認
```

### 4.2 Amplify ビルド設定（amplify.yml）

```yaml
version: 1
frontend:
  phases:
    preBuild:
      commands:
        - corepack enable
        - pnpm install --frozen-lockfile
    build:
      commands:
        - pnpm run build
  artifacts:
    baseDirectory: .next
    files:
      - '**/*'
  cache:
    paths:
      - node_modules/**/*
      - .next/cache/**/*
```

### 4.3 環境変数管理

- 本番環境変数: AWS Amplify Console → 環境変数 で管理
- シークレット（DB パスワード・API キー等）: AWS Secrets Manager に格納し、Amplify ビルド時に取得
- `.env` ファイルはリポジトリにコミットしない（`.gitignore` で除外）
- 本番 rate-limit backend は `docs/operations/rate-limit-production-runbook.md` に従い、DynamoDB table / TTL / IAM / 通常 API 経路をデプロイ前後に確認する

## 5. ロールバック手順

### 5.1 Amplify コンソール経由のロールバック

1. AWS Amplify Console にログイン
2. 対象アプリケーション → 「Hosting」→ 「ビルド履歴」を開く
3. 正常動作していた前回ビルドを特定
4. 該当ビルドの「再デプロイ」ボタンをクリック
5. デプロイ完了後、ヘルスチェック（`/api/health`）で正常性確認
6. インシデント対応手順（incident-response.md）に従い、原因調査を開始

### 5.2 DBスキーマ変更のロールバック

Prisma マイグレーションのロールバックは破壊的操作を伴う可能性があるため、以下の手順で実施する:

1. RDS スナップショット（デプロイ直前取得）からリストア
2. 代替案: ロールバック用マイグレーションファイルを作成して適用
3. アプリケーションコードを旧バージョンに戻す（Amplify ロールバック）

**事前対策**: スキーマ変更デプロイ前に必ず RDS スナップショットを手動取得する。

### 5.3 ロールバック判断基準

| 状況 | 対応 |
|---|---|
| エラー率 > 5%（5 分間） | 即時ロールバック |
| 主要機能の停止 | 即時ロールバック |
| パフォーマンス劣化（P99 > 10 秒）| 調査後にロールバック判断 |
| 軽微な不具合 | hotfix で対応（ロールバックしない） |

## 6. 本番デプロイ前チェックリスト

本番デプロイ前に以下を全て確認する。

### 6.1 テスト

- [ ] `pnpm test` が全テスト通過（0 failures）
- [ ] `pnpm test:e2e` が主要フローで通過
- [ ] ステージング環境での手動確認完了

### 6.2 セキュリティ

- [ ] `pnpm audit` で Critical/High 脆弱性がないこと
- [ ] 機密情報のハードコードがないこと（`git grep` で確認）
- [ ] 新規 Route Handler に認証チェックが実装されていること

### 6.3 データベース

- [ ] Prisma マイグレーションファイルが正しく生成されていること
- [ ] ステージング DB でマイグレーション検証済みであること
- [ ] 本番 RDS の直前スナップショットを取得済みであること

### 6.4 インフラ・設定

- [ ] 環境変数が本番環境に設定されていること
- [ ] Amplify ビルドがステージングで成功していること
- [ ] CloudWatch アラートが正常に機能していること

### 6.5 ドキュメント・コミュニケーション

- [ ] CHANGELOG または PR の変更内容が適切に記載されていること
- [ ] 破壊的変更の場合、影響先（連携薬局等）への事前通知完了

## 7. 緊急変更手順（Hotfix）

### 7.1 フロー

```
本番障害検知
    ↓
インシデントレベル判定（Critical/High）
    ↓
hotfix/xxx ブランチを main から作成
    ↓
修正実装（最小限の変更に絞る）
    ↓
CI チェック通過
    ↓
緊急レビュー（最低 1 名、電話・チャットで実施可）
    ↓
本番デプロイ（Amplify ブランチデプロイ or main マージ）
    ↓
本番確認
    ↓
事後レビュー（24 時間以内に通常 PR として再提出・レビュー記録）
```

### 7.2 事後レビュー記録

緊急変更実施後 24 時間以内に以下を記録する:

- 変更内容の説明
- 実施理由・緊急度の根拠
- 本来のレビュープロセスを省略した理由
- 事後レビュー実施者と確認結果
- 再発防止策

---

## 8. 変更記録の保持

| 記録媒体 | 保持内容 | 保持期間 |
|---|---|---|
| Git リポジトリ | 全コミット履歴、PR・コードレビューコメント | 無期限（GitHub 上） |
| Amplify デプロイ履歴 | ビルドログ、デプロイ日時、コミット SHA | 25 ビルド分（Amplify 自動管理） |
| AuditLog テーブル | アプリケーションレベルの設定変更操作 | 5 年 |
| AWS CloudTrail | AWS リソースへの全 API 操作 | 5 年（S3 + Object Lock） |
| インシデントチケット | 緊急変更・障害対応の記録 | 5 年 |

---

## 9. インフラ変更管理

AWS インフラの変更（RDS 設定変更・Cognito 設定変更・S3 バケットポリシー変更等）も同じ変更管理フローに従う。

| 変更手段 | 承認要件 |
|---|---|
| AWS CDK / CloudFormation | PR レビュー + ステージング検証 + 本番承認 |
| AWS Console 手動操作 | 原則禁止。緊急時のみ実施し、事後 IaC に反映 |
| Secrets Manager 更新 | セキュリティ担当者の承認 + 変更記録 |
| RDS パラメータグループ変更 | 重要変更フロー（再起動が伴う場合は事前告知） |
