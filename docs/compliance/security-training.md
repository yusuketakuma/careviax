# C-06 — セキュリティ教育計画（Security Training Plan）

文書番号: C-06
バージョン: 1.0
作成日: 2026-03-29
分類: 内部管理用
準拠: MHLW ガイドライン v6.0 第7章 / METI/MIC ガイドライン v1.1 第4.3条 / 個人情報保護法 第24条

---

## 1. 目的

PH-OS に関わる全従業員および業務委託者が、医療情報の取扱いに関する適切な知識とセキュリティ意識を持ち、3省2ガイドライン（MHLW v6.0 + METI/MIC v1.1）に準拠した安全管理措置を実施できる体制を整備することを目的とする。

---

## 2. 適用範囲

| 対象者 | 適用される研修 |
|---|---|
| 全従業員（正社員・契約社員・パートタイム） | 新入社員研修（入社時）+ 年次研修 |
| 業務委託者（業務上 PHI に接触する者） | 新入社員研修相当（委託開始時）+ 年次研修 |
| ソフトウェア開発者（社内・委託） | 上記に加えて開発者研修 |
| システム管理者・セキュリティ担当者 | 上記に加えて管理者研修 |
| 薬局オーナー・管理者職 | 上記に加えて管理者研修 |

---

## 3. 研修カリキュラム

### 3.1 新入社員研修（入社後30日以内に完了）

**目的**: 業務に先立ち、医療情報の適切な取扱いとPH-OS システムの安全な利用方法を習得する。

#### Module 1: 個人情報保護の基礎（所要時間: 90分）

| トピック | 内容 |
|---|---|
| 個人情報保護法（APPI）の概要 | 個人情報・要配慮個人情報の定義、取得・利用・管理・提供の原則、違反時のリスク（刑事罰・民事責任） |
| 医療情報の特別性 | 患者情報が「要配慮個人情報」である理由、守秘義務の法的根拠、薬剤師法における秘密保持義務 |
| 3省2ガイドラインの概要 | MHLW ガイドライン v6.0 と METI/MIC ガイドライン v1.1 の位置づけと主要要件 |
| PH-OS の取扱い方針 | プライバシーポリシー、情報取扱い規程、日常業務での注意事項 |

確認テスト: 正誤問題10問（80点以上で合格）

#### Module 2: パスワード管理とアクセス制御（所要時間: 60分）

| トピック | 内容 |
|---|---|
| 強固なパスワードの設定 | 最低12文字・複雑性要件、パスワードの使い回し禁止、パスワードマネージャーの活用 |
| 多要素認証（MFA）の設定と利用 | TOTP アプリのセットアップ手順（Google Authenticator / Authy 等）、代替手段としての FIDO2 |
| セッション管理 | 無操作時の自動ロック（推奨15分）、共有端末での作業後のログアウト徹底 |
| 最小権限の原則 | 自身に付与されたロールと権限の確認、不要なデータアクセスの回避 |
| 端末紛失・盗難時の対応 | 即時報告手順、Cognito によるリモートセッション無効化の依頼方法 |

確認テスト: 実技（MFA 設定の実施確認）

#### Module 3: フィッシング対策とソーシャルエンジニアリング（所要時間: 60分）

| トピック | 内容 |
|---|---|
| フィッシングメールの見分け方 | 送信元ドメインの確認、リンク先 URL の確認、不審な添付ファイルの取扱い |
| スミッシング・ビッシング | SMS・電話を使った詐欺の手口と対処法 |
| ソーシャルエンジニアリング | なりすまし、口頭での情報引き出し、物理的な不正侵入手口 |
| 不審なアクセス要求への対応 | 上長への即時報告、口頭でのパスワード提供を絶対に行わない原則 |

確認テスト: 模擬フィッシングメール識別演習

#### Module 4: PH-OS 安全利用規則（所要時間: 60分）

| トピック | 内容 |
|---|---|
| 業務端末の管理 | 私的利用禁止範囲、スクリーンロック設定、持ち出し時の暗号化要件 |
| 患者情報の取扱い | 業務上必要な患者情報へのアクセス範囲、無関係な患者情報への好奇心アクセスの禁止 |
| オフライン利用 | PWA オフライン同期データの取扱い、IndexedDB 暗号化の仕組みと注意事項 |
| 情報の持ち出しと印刷 | 原則ペーパーレス、印刷物の取扱いと廃棄（シュレッダー必須） |
| インシデント報告義務 | 疑わしい事象の即時報告先と報告様式 |

### 3.2 年次研修（毎年4〜5月に実施、所要時間: 120分）

**目的**: セキュリティ意識の維持・更新と、最新の脅威への対応力強化。

#### Part 1: 年次振り返りと最新動向（30分）

| トピック | 内容 |
|---|---|
| 前年度のインシデント事例 | 医療分野でのサイバー攻撃事例（国内・海外）、PH-OS での発生事例（ある場合） |
| 規制・ガイドラインの改定内容 | MHLW・METI/MIC ガイドラインの最新改定ポイント |
| 新しいセキュリティ脅威 | 生成 AI を悪用した攻撃、新しいランサムウェア手口等 |

#### Part 2: ソーシャルエンジニアリング事例演習（30分）

| シナリオ | 演習内容 |
|---|---|
| シナリオA: 内部関係者を装った電話 | 「IT部門ですが緊急でパスワードをリセットしてください」への対応 |
| シナリオB: 患者家族を装ったアクセス依頼 | 「父の処方情報を確認したい」という電話対応のロールプレイ |
| シナリオC: 不審な USB | 会議室に放置された USB デバイスの発見時の対応 |

#### Part 3: 物理セキュリティ（30分）

| トピック | 内容 |
|---|---|
| 入退室管理 | ID カード未携帯者の入室防止、共連れ防止、来訪者管理 |
| クリアデスクポリシー | 患者情報を含む書類の保管、ホワイトボード消去 |
| モバイル端末のセキュリティ | 公共 Wi-Fi 利用禁止（業務端末）、VPN の必須化 |
| 廃棄物のセキュリティ | 書類のシュレッダー、電子機器の適切な廃棄手順 |

#### Part 4: 知識確認テストと振り返り（30分）

- 総合テスト: 正誤問題20問（80点以上で合格）
- 合格した場合: 受講完了を記録
- 不合格の場合: 不正解箇所の解説後、再テスト（2週間以内）

### 3.3 開発者研修（年1回、7月に実施、所要時間: 180分）

**対象**: ソフトウェアエンジニア、フロントエンドエンジニア、インフラエンジニア

**目的**: セキュアコーディングの実践と、医療情報システム特有のセキュリティ要件の習得。

#### Module 1: OWASP Top 10 と PH-OS への適用（60分）

| 脅威 | PH-OS における具体的対策 |
|---|---|
| A01: Broken Access Control | RLS による org_id テナント分離、7ロール RBAC 設計、API ルートの認可チェック |
| A02: Cryptographic Failures | TLS 1.3 強制、AES-256 (KMS) 保存暗号化、IndexedDB の AES-GCM 暗号化 |
| A03: Injection | Prisma ORM によるパラメータ化クエリ、`$executeRaw` での入力サニタイズ |
| A04: Insecure Design | 設計フェーズでの脅威モデリング実施、最小権限原則の徹底 |
| A05: Security Misconfiguration | IaC（CDK）でのセキュリティ設定、環境変数管理（Secrets Manager） |
| A06: Vulnerable Components | `pnpm audit` の CI 実行、依存関係の定期更新 |
| A07: Identification and Authentication Failures | Cognito MFA 強制、セッションタイムアウト設定 |
| A08: Software and Data Integrity Failures | コードレビュー必須、署名付きコミット、CI/CD パイプラインのインテグリティ |
| A09: Security Logging and Monitoring Failures | 全 API 操作の AuditLog 記録、CloudTrail + CloudWatch アラート |
| A10: Server-Side Request Forgery | 外部 URL へのフェッチ制限、許可リストによる外部接続制御 |

#### Module 2: セキュアコーディング実践（60分）

| トピック | 実習内容 |
|---|---|
| 入力バリデーション | Zod スキーマの設計（医療データ特有のバリデーション例）、Server Actions での検証パターン |
| 認証・認可の実装 | NextAuth + Cognito セッション検証、Route Handler でのロールチェック実装パターン |
| 機密情報の取扱い | `.env` ファイルの管理規則、Secrets Manager への移行、ログへの PHI 出力禁止 |
| XSS 対策 | Next.js の自動エスケープ、`dangerouslySetInnerHTML` 使用禁止、CSP ヘッダー設定 |
| CSRF 対策 | Next.js の組み込み CSRF 保護、SameSite Cookie 設定 |
| SQL インジェクション | Prisma Raw クエリ使用時の注意点、`$queryRaw` / `$executeRaw` の安全な使い方 |

#### Module 3: コードレビューのセキュリティ観点（30分）

| チェック項目 | 確認方法 |
|---|---|
| ハードコードされた機密情報 | `grep` / Semgrep ルールによる自動検出 |
| 認証チェックの漏れ | 全 Route Handler の `getServerSession` 呼び出し確認 |
| 不適切なエラーハンドリング | スタックトレースや DB 情報を含むエラーレスポンス禁止 |
| 依存関係の脆弱性 | `pnpm audit --audit-level moderate` の通過確認 |
| テストの回避 | `it.skip` / `describe.skip` の本番コードへの混入禁止 |

#### Module 4: インシデント発生時の開発者対応（30分）

- セキュリティパッチの緊急適用手順
- 本番環境での hotfix フロー（承認プロセスを短縮しつつ記録を維持する方法）
- 脆弱性報告の受付と優先対応基準

### 3.4 管理者研修（年1回、7月に実施、所要時間: 120分）

**対象**: システム管理者、セキュリティ担当者、薬局管理者・オーナー職

**目的**: 特権アクセスの適切な管理と、インシデント発生時の指揮能力の習得。

#### Module 1: 特権アクセス管理（PAM）（60分）

| トピック | 内容 |
|---|---|
| 特権アカウントの種別と管理 | AWS ルートアカウント・管理者 IAM ユーザーの管理方針、日常業務での使用禁止 |
| Just-in-Time 特権昇格 | AWS IAM Roles による一時的な特権付与、使用後の権限失効確認 |
| 特権操作の承認フロー | 本番 DB への直接アクセス・設定変更時の二者承認（4-eyes principle） |
| 特権セッションの記録 | CloudTrail + Session Manager による特権操作の完全ログ取得 |
| 定期的な権限棚卸し | 四半期ごとの IAM ユーザー・ロールの棚卸し手順（不要な権限の即時削除） |
| 退職者・異動時の対応 | 24時間以内のアカウント無効化・権限剥奪手順 |

#### Module 2: インシデント対応指揮（60分）

| トピック | 内容 |
|---|---|
| インシデント指揮官の役割 | 対応の全体調整、情報収集と意思決定、エスカレーション判断 |
| 報告義務の確認 | 個人情報保護委員会への報告（72時間/30日ルール）、患者・関係医療機関への通知 |
| 媒体・メディア対応 | 公式発表前の情報統制、プレスリリースの承認フロー（C-04 危機管理参照） |
| 法的対応 | 証拠保全の重要性、司法・行政機関との連絡 |
| 訓練の実施・評価 | 机上演習のファシリテーション方法、演習後のレビューと改善 |

## 4. インシデント対応机上演習（Tabletop Exercise）

### 4.1 実施スケジュール

| 開催時期 | 対象 | 演習シナリオ区分 |
|---|---|---|
| 毎年6月（第2週） | 管理者・セキュリティ担当者・開発者代表 | サイバー攻撃シナリオ |
| 毎年12月（第2週） | 全従業員代表（部門横断） | 情報漏洩・業務継続シナリオ |

### 4.2 演習シナリオ一覧

#### シナリオ A: 不正アクセス検知（6月演習用）

**概要**: 深夜にCognito ログインが普段と異なる IP アドレス（海外）から成功し、短時間で大量の患者情報にアクセスが行われた。

**演習ポイント**:
- CloudTrail / CloudWatch アラートの確認手順
- 不正ユーザーのセッション強制無効化（Cognito API）
- 影響を受けた患者データの特定
- 個人情報保護委員会への速報判断（3〜5日以内）
- 患者への通知要否の判断基準

**確認事項**:
- [ ] アラートから30分以内にインシデント指揮官が判断を開始できるか
- [ ] 影響範囲の特定に必要なログが揃っているか
- [ ] 報告義務の判断フローが機能するか

---

#### シナリオ B: ランサムウェア感染（6月演習用・上級編）

**概要**: 業務端末の1台がフィッシングメール経由でマルウェアに感染。VPN 経由で社内システムへの接続が試みられている。

**演習ポイント**:
- 感染端末の即時ネットワーク隔離手順
- AWS 側の予防的遮断（WAF・セキュリティグループの設定変更）
- RDS・S3 の感染有無確認（CloudTrail によるバックアップ操作の確認）
- IT-BCP 発動判断（C-03 IT-BCP 参照）
- 業務継続：オフライン PWA への切り替え手順

**確認事項**:
- [ ] 端末隔離から AWS 側の遮断までの所要時間
- [ ] バックアップからのリストア手順が文書化されているか
- [ ] 訪問業務を継続するための代替手順が周知されているか

---

#### シナリオ C: 患者情報の誤送信（12月演習用）

**概要**: 薬剤師が患者Aの報告書PDFを誤って患者Bの主治医に送信してしまった。SES の送信ログで翌日に判明。

**演習ポイント**:
- 誤送信の即時確認（SES 配信ログ、受信確認）
- 患者Aへの通知（誰が、いつ、何を通知するか）
- 誤送信先医師への削除依頼手順
- 内部インシデントチケットへの記録
- 再発防止策の策定（送信前確認ダイアログの改善等）

**確認事項**:
- [ ] 誤送信発覚から患者への通知までの目標時間（24時間以内）
- [ ] 報告義務の要否判断（個人情報保護委員会への報告は必要か）

---

#### シナリオ D: AWS 障害による業務継続（12月演習用）

**概要**: ap-northeast-1 の RDS で障害が発生し、PH-OS のバックエンド API が全て応答不能になった。本日は訪問予定が10件ある。

**演習ポイント**:
- 訪問業務チームへの迅速な状況通知
- PWA オフラインモードへの切り替えと注意事項の周知
- AWS サポートへの連絡・RTO 4時間の目標管理
- 紙・電話による代替連絡体制の発動
- 復旧後の差分データ同期手順

**確認事項**:
- [ ] 従業員全員がオフライン PWA の使い方を理解しているか
- [ ] 代替連絡手段（緊急連絡先リスト）が更新されているか

### 4.3 演習実施手順

1. **事前準備**（演習の2週間前）
   - シナリオの確定と参加者への事前通知
   - 演習用ファシリテーターの指定（セキュリティ責任者 または 訓練担当者）
   - 必要な手順書・参照文書のリストアップ

2. **演習実施**（2〜3時間）
   - イントロダクション（15分）: 目的・ルール・タイムライン説明
   - シナリオ展開（90〜120分）: インジェクション方式で状況を段階的に追加
   - 対応判断の記録: 参加者の判断根拠・時間・使用した手順書を記録

3. **ホットウォッシュ**（演習直後、30分）
   - 対応の良かった点・改善が必要な点をその場で洗い出し
   - 即時対応が必要な改善項目の特定

4. **演習後レポート**（演習後5営業日以内）
   - 演習結果のサマリー
   - 手順書・システム・体制の改善項目リスト
   - 次回演習への反映事項

## 5. 研修記録の管理

### 5.1 記録すべき情報

| 項目 | 内容 |
|---|---|
| 受講者識別 | 従業員 ID・氏名・部署・役職 |
| 研修情報 | 研修名・コード（例: SEC-NE-2026）・バージョン |
| 実施日時 | 受講日・受講形式（集合/オンライン） |
| 受講結果 | テストスコア・合否・合格日 |
| 次回受講期限 | 年次研修の場合は翌年の同月末 |
| 証跡 | 受講完了証（PDF または電子署名） |

### 5.2 記録の保管

- 研修記録は PH-OS の管理システム（HR システムまたは専用管理台帳）に保管する。
- 保管期間: 在職中 + 退職後5年間
- アクセス権: HR 担当者・セキュリティ責任者・外部監査人

### 5.3 記録フォーマット（例）

```
研修記録 ID: TRN-2026-0001
従業員 ID: EMP-00123
氏名: [氏名]
部署: システム開発部
役職: ソフトウェアエンジニア
研修コード: SEC-DE-2026（開発者研修 2026年度）
受講形式: 集合研修
受講日: 2026-07-10
テストスコア: 92 / 100
合否: 合格
次回受講期限: 2027-07-31
担当者確認: [担当者署名/記名押印]
```

## 6. 未受講者へのフォローアップ手順

### 6.1 フォローアップのフロー

```
研修実施後10営業日
    ↓
未受講者リスト抽出
    ↓
[Step 1] 本人および直属上長へのリマインドメール送信（追加受講機会の案内）
    ↓
さらに10営業日後も未受講の場合
    ↓
[Step 2] 直属上長を通じた個別フォロー（業務上の都合の確認）
    ↓
さらに10営業日後も未受講の場合
    ↓
[Step 3] セキュリティ責任者・HR へのエスカレーション
    ↓
[Step 4] 業務停止措置の検討（PHI へのアクセス権限を一時的に制限）
    ↓
受講完了後
    ↓
記録の更新・アクセス権の復旧
```

### 6.2 未受講者の定義

| 対象研修 | 未受講とみなす基準 |
|---|---|
| 新入社員研修 | 入社後30日以内に未完了 |
| 年次研修 | 研修実施月の末日（5月末）までに未完了 |
| 開発者研修 | 研修実施月の末日（7月末）までに未完了 |
| 管理者研修 | 研修実施月の末日（7月末）までに未完了 |

### 6.3 業務上の例外処理

以下の場合は、合理的な期間延長を認める（セキュリティ責任者の承認必要）:

| 事由 | 延長可能期間 |
|---|---|
| 産育休・介護休暇 | 復職後30日以内 |
| 長期入院・病気療養 | 復職後30日以内 |
| 長期出張（海外赴任等） | 帰国後30日以内 |
| 災害・自然災害 | 状況に応じて個別判断 |

## 8. 評価と改善

### 8.1 研修プログラムの有効性評価

| 評価指標 | 測定方法 | 目標値 |
|---|---|---|
| 受講完了率 | 研修記録 | 100%（期限内） |
| テスト合格率 | テストスコア集計 | 95%以上 |
| フィッシングシミュレーション引っかかり率 | 年次比較 | 前年比20%以上改善 |

| 机上演習の手順遵守率 | 演習評価シート | 重大手順ミスゼロ |

### 8.2 研修内容の年次見直し

- 毎年3月に、前年度の研修内容・事例・規制変更を踏まえた見直しを実施する。
- MHLW・METI/MIC ガイドラインの改定時は、速やかに研修コンテンツを更新する。
- インシデント発生時は、当該事例を次回研修に追加する。
