# C-05 — 委託先リスク評価（Vendor Risk Assessment）

文書番号: C-05
バージョン: 1.0
作成日: 2026-03-29
分類: 機密（内部管理用）
準拠: MHLW ガイドライン v6.0 第6章 / METI/MIC ガイドライン v1.1 第4.4条

---

## 1. 目的

CareViaX が利用する外部委託先（クラウドサービス事業者およびサブプロセッサ）のセキュリティリスクを評価・管理し、3省2ガイドライン（MHLW v6.0 + METI/MIC v1.1）および個人情報保護法（APPI）に基づく要配慮個人情報（医療情報）の安全管理措置を確保することを目的とする。

---

## 2. 評価対象の基本方針

### 2.1 委託の種別

| 種別 | 内容 | 本文書での扱い |
|---|---|---|
| **主委託先** | 医療情報を直接処理するクラウドサービス | 本章で詳細評価 |
| **サブプロセッサ** | 主委託先が再委託する事業者 | Section 4 でリスト化 |
| **間接委託** | ソフトウェアライセンス等、PHI 非接触 | 評価対象外 |

### 2.2 評価方針

- 全ての主委託先は年次でリスク評価を実施する。
- ISMAP（情報システムムセキュリティ管理及び評価制度）認定済みのサービスについては、その認定を一次評価として活用する。
- ISMAP 未取得のサービスについては、独自のセキュリティ評価を実施する。

---

## 3. AWS サービス別セキュリティ評価

### 3.0 AWS ISMAP 認定ステータス

| 項目 | 内容 |
|---|---|
| 認定取得日 | 取得済み（最新の更新確認: 政府情報システムのためのセキュリティ評価制度 ISMAP クラウドサービスリスト参照） |
| 対象サービス数 | 171+ サービス（ap-northeast-1 リージョン対象） |
| 評価機関 | 第三者認定機関によるISO/IEC 27001 等に基づく審査 |
| CareViaX の利用リージョン | ap-northeast-1（東京リージョン）固定 |
| リスト参照先 | https://www.ismap.go.jp/csm |

ISMAP 認定により、AWS はMHTW ガイドライン v6.0 に定める「外部保存事業者の選定基準（第6.11条）」を満たすと判断する。

### 3.1 Amazon RDS for PostgreSQL

**役割**: 患者情報・処方情報・ワークフローデータ・監査ログ等、CareViaX の全業務データを保存するプライマリデータベース。

| 評価項目 | 実装内容 | 充足状況 |
|---|---|---|
| **可用性** | Multi-AZ 配置（スタンバイレプリカあり）、自動フェイルオーバー（通常60秒以内） | 充足 |
| **バックアップ** | 自動バックアップ（保持期間: 7〜35日、要設定）、手動スナップショット、特定時点への復元（PITR） | 充足 |
| **暗号化（保存時）** | AWS KMS による AES-256 暗号化（ストレージ、スナップショット、ログすべて暗号化） | 充足 |
| **暗号化（転送中）** | TLS 1.3 による接続暗号化（`require_ssl` パラメータで強制） | 充足 |
| **アクセス制御** | VPC 内のプライベートサブネット配置、セキュリティグループによるポート制限（5432/tcp はアプリサーバーのみ許可） | 充足 |
| **監査ログ** | PostgreSQL のログ機能（pgaudit 推奨）+ CloudWatch Logs への転送 | 要設定 |
| **テナント分離** | PostgreSQL Row Level Security (RLS) による `org_id` ベースのテナント分離 | 充足 |
| **パフォーマンス** | Performance Insights によるクエリ分析、Enhanced Monitoring | 充足 |
| **パッチ管理** | AWS マネージドサービスとしてのセキュリティパッチ自動適用（メンテナンスウィンドウ設定） | 充足 |

**残存リスク**:
- pgaudit の有効化設定が必要（設定漏れの場合、細粒度の DB 監査が不完全）。
- バックアップ保持期間の最終設定値を確認・文書化すること。

---

### 3.2 Amazon Cognito

**役割**: ユーザー認証、MFA、セッション管理、ユーザープール管理。

| 評価項目 | 実装内容 | 充足状況 |
|---|---|---|
| **多要素認証（MFA）** | TOTP（Time-based One-Time Password）および FIDO2/WebAuthn 対応、全ユーザーに MFA 強制 | 充足 |
| **パスワードポリシー** | 最低12文字、大文字/小文字/数字/記号を含む複雑性要件、90日更新推奨 | 充足 |
| **ブルートフォース対策** | アカウントロック（連続失敗5回でロック）、Advanced Security Features によるリスクベース認証 | 充足 |
| **ユーザープール暗号化** | Cognito 内部での暗号化、KMS 統合によるカスタム暗号化キーの利用可能 | 充足 |
| **セッション管理** | JWT アクセストークン（有効期間: 1時間）、リフレッシュトークン（有効期間: 要設定、推奨30日）、明示的な無効化 API | 充足 |
| **ログ・監査** | CloudTrail による全 Cognito API 操作の記録、Advanced Security Events ログ | 充足 |
| **コンプライアンス** | SOC 1/2/3、PCI DSS、ISO 27001 等の認定取得済み（ISMAP 対象） | 充足 |
| **フェデレーション** | SAML 2.0 / OIDC フェデレーション対応（将来の SSO 拡張に備え） | 対応可能 |

**残存リスク**:
- MFA 強制設定が全ユーザープールで有効になっていること（設定確認が必要）。
- 長期間不活性ユーザーアカウントの自動無効化ポリシーの設定を推奨。

### 3.3 Amazon S3

**役割**: 処方箋原本PDF・報告書・同意書・患者関連ファイルの保存。

| 評価項目 | 実装内容 | 充足状況 |
|---|---|---|
| **Object Lock** | WORM（Write Once Read Many）モード有効化、医療文書の不正改ざん防止、保持期間5年以上 | 充足 |
| **バケットポリシー** | パブリックアクセスブロック（全4設定を無効化）、最小権限原則に基づく IAM ポリシー | 充足 |
| **暗号化（保存時）** | SSE-KMS による AES-256 暗号化、KMS カスタムキー使用 | 充足 |
| **暗号化（転送中）** | TLS 1.3 強制（HTTP アクセスを拒否するバケットポリシー） | 充足 |
| **VPC Endpoint** | S3 Gateway Endpoint を通じた VPC 内プライベート通信（インターネット経由を不要とする） | 充足 |
| **バージョニング** | バージョン管理有効、誤削除時の復元対応 | 充足 |
| **アクセスログ** | S3 サーバーアクセスログ有効化、CloudTrail データイベントによる全 API 操作記録 | 充足 |
| **署名付き URL** | ファイルアクセスは Presigned URL（有効期間15分以内）経由のみ、直接公開なし | 充足 |
| **ライフサイクル** | 保持期間経過後の Glacier 移行ルール（医療記録5年 + アーカイブ）、自動削除は設定しない | 充足 |

**残存リスク**:
- Object Lock の保持期間設定値を文書化し、法令上の最低保存期間との整合性を定期確認する必要がある。

---

### 3.4 Amazon SES（Simple Email Service）

**役割**: 患者報告書PDF送付、通知メール、システムアラートメール。

| 評価項目 | 実装内容 | 充足状況 |
|---|---|---|
| **DKIM** | DomainKeys Identified Mail 署名による送信元ドメイン認証 | 充足 |
| **SPF** | Sender Policy Framework による送信元 IP 認証 | 充足 |
| **DMARC** | Domain-based Message Authentication 設定（`p=quarantine` 以上推奨） | 要設定 |
| **TLS** | 送信時の STARTTLS 暗号化 | 充足 |
| **送信ログ** | SES Event Publishing による送信・配信・バウンス・クレームの記録（CloudWatch / S3 転送） | 充足 |
| **バウンス処理** | バウンスおよびクレームの自動処理、高リスク送信先の除外 | 充足 |
| **送信制限** | SES Sending Limits の設定、意図しない大量送信防止 | 充足 |
| **個人情報保護** | 報告書本文に PHI を含む添付 PDF は暗号化 ZIP または Presigned URL 参照方式を検討 | 要対応 |

**残存リスク**:
- DMARC ポリシーの設定状況を確認し、`p=reject` への段階的移行を計画する。
- 報告書 PDF をメール添付する際、PHI を含む PDF のパスワード保護またはリンク参照への移行を検討する。

---

### 3.5 AWS CloudTrail

**役割**: AWS 全サービスの API 操作記録、インシデント調査の基盤。

| 評価項目 | 実装内容 | 充足状況 |
|---|---|---|
| **全リージョン記録** | マルチリージョントレイル有効（ap-northeast-1 メイン） | 充足 |
| **全 API 操作記録** | Management Events（読み取り/書き込み両方）+ Data Events（S3 オブジェクト操作）を記録 | 充足 |
| **ログの完全性検証** | CloudTrail ログファイルの整合性検証（Log File Integrity Validation）有効化 | 充足 |
| **S3 への保存** | CloudTrail ログを専用バケット（Object Lock + アクセス制限）に保存 | 充足 |
| **保存期間** | 5年間保存（MHLW ガイドライン準拠） | 充足 |
| **リアルタイムアラート** | CloudWatch Events との連携による不審操作のリアルタイム検知・通知 | 充足 |
| **アクセス制限** | CloudTrail 設定変更・ログ削除を IAM で厳格に制限 | 充足 |

**残存リスク**:
- Data Events（S3・Lambda）は追加コストがかかるため、記録対象バケットの選定と設定状況を確認する。

---

### 3.6 AWS KMS（Key Management Service）

**役割**: RDS・S3・その他サービスの暗号鍵管理。

| 評価項目 | 実装内容 | 充足状況 |
|---|---|---|
| **カスタマー管理キー（CMK）** | CareViaX 専用の KMS カスタムキーを用途別（DB / ファイル / オフライン）に分離 | 充足 |
| **自動ローテーション** | 年次の自動鍵ローテーション有効化（CMK は1年ごと） | 充足 |
| **鍵ポリシー** | 最小権限原則に基づくキーポリシー、使用者・管理者の分離 | 充足 |
| **キー利用監査** | CloudTrail による全 KMS API 操作（暗号化・復号・ローテーション）の記録 | 充足 |
| **HSM バックエンド** | KMS は FIPS 140-2 認定の HSM（Hardware Security Module）で保護 | 充足 |
| **マルチリージョンキー** | DR シナリオに備えたマルチリージョンキーの設計（将来対応） | 計画中 |
| **待機期間付き削除** | 鍵削除は7〜30日の待機期間必須（誤削除防止） | 充足 |

**残存リスク**:
- 用途別 CMK の設計書（キーエイリアス・用途・管理者・使用者）を文書化し、四半期棚卸しを実施する。

## 5. データ処理場所

### 5.1 リージョン固定ポリシー

CareViaX は、全ての患者情報・医療情報の処理および保存を **ap-northeast-1（東京リージョン）固定** とする。

| データ種別 | 処理場所 | 保存場所 | 国外移転 |
|---|---|---|---|
| 患者基本情報 | ap-northeast-1 | RDS (ap-northeast-1) | なし |
| 処方・薬歴情報 | ap-northeast-1 | RDS (ap-northeast-1) | なし |
| 訪問記録・報告書 | ap-northeast-1 | RDS + S3 (ap-northeast-1) | なし |
| 画像・PDF ファイル | ap-northeast-1 | S3 (ap-northeast-1) | なし |
| 監査ログ | ap-northeast-1 | S3 + CloudWatch (ap-northeast-1) | なし |
| 認証情報 | ap-northeast-1 | Cognito (ap-northeast-1) | なし |

### 5.2 国外移転の禁止

- 要配慮個人情報（医療情報）の国外移転は行わない。
- AWS の機能を利用する際は、データが東京リージョン外に複製されないよう、リージョン制限 SCP（Service Control Policy）を適用する。
- クロスリージョンレプリケーションは医療データバケットに対して無効化する。

## 7. 契約上の要件

### 7.1 AWS との契約関係

| 契約種別 | 内容 | 充足状況 |
|---|---|---|
| **AWS サービス利用規約** | AWS Customer Agreement への同意（利用開始時） | 充足 |
| **AWS DPA（Data Processing Addendum）** | GDPR および各国データ保護法に準拠した個人データ処理契約 | 確認要（BAA相当） |
| **AWS Business Associate Addendum** | 医療情報（HIPAA 定義の PHI）に関する追加契約（日本法上の「委託契約」相当） | 確認要 |
| **SLA** | RDS: 99.95%、S3: 99.99%、Cognito: 99.9%（各サービスの SLA に基づく）| 充足 |

### 7.2 必須契約要件チェックリスト

委託先との契約において、以下の事項が明示されていること：

- [ ] 個人情報の利用目的の制限（委託業務の目的外利用の禁止）
- [ ] 再委託先（サブプロセッサ）の管理義務および通知義務
- [ ] セキュリティ措置の水準の明示（暗号化・アクセス制御・監査ログ等）
- [ ] インシデント発生時の通知義務（発覚後72時間以内の通知等）
- [ ] 監査権（委託元による委託先のセキュリティ確認権）
- [ ] 契約終了時のデータ削除義務（削除完了の証明書交付を含む）
- [ ] データ保存場所の制限（国内固定またはその変更時の事前通知）
- [ ] 法令遵守義務（個人情報保護法・MHLW ガイドライン準拠）

### 7.3 契約終了時のデータ削除

| 対象 | 削除対象 | 期限 | 証明 |
|---|---|---|---|
| RDS | 全データベースインスタンス・スナップショット | 契約終了から90日以内 | AWS 証明書または削除記録 |
| S3 | 全バケット・オブジェクト（Object Lock 期間経過後） | 保持期間終了後、契約終了から90日以内 | AWS 証明書 |
| Cognito | ユーザープールおよびユーザーデータ | 契約終了から30日以内 | AWS 証明書 |
| CloudTrail | ログファイル（法令上の保存義務終了後） | 法令保存期間終了後、契約終了から90日以内 | 削除記録 |

## 8. リスク評価サマリー

| AWS サービス | 固有リスク | 残存リスク | 総合評価 |
|---|---|---|---|
| RDS | 中（PHI の主保存先） | 低（pgaudit 設定要） | **低〜中** |
| Cognito | 中（認証基盤） | 低（MFA 設定確認要） | **低** |
| S3 | 中（文書・ファイル保存） | 低（Object Lock 設定確認要） | **低** |
| SES | 低〜中（PHI 含む可能性） | 中（DMARC・PHI メール方針要） | **中** |
| CloudTrail | 低（ログ保存のみ） | 低 | **低** |
| KMS | 低〜中（暗号鍵管理） | 低（CMK 設計書要） | **低** |

**総合評価**: AWS ISMAP 認定を前提とし、CareViaX での適切な設定・運用を前提とすれば、委託先リスクは許容範囲内と判断する。残存リスクについては Section 9 の改善計画に従い対応する。
