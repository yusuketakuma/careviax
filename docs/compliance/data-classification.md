# C-03 — データ分類基準

## 概要

PH-OS が保有・処理するデータの機密度分類、テーブル/カラム単位の機密度ラベル、データ保持ポリシー、削除ポリシー、暗号化要件を定義する。
個人情報保護法（APPI）の「要配慮個人情報」規定および 3省2ガイドライン（MHLW v6.0 §4.1）に準拠。

---

## 1. データ分類（3 段階）

### 1.1 機密（Confidential）— PHI / 要配慮個人情報

**定義**: 個人情報保護法第 2 条第 3 項に規定される「要配慮個人情報」を含むデータ、または漏洩した場合に患者・利用者に重大な不利益をもたらすデータ。

**特徴**:

- 法令上の特別な保護義務あり（取得に原則として本人同意が必要）
- 漏洩時に個人情報保護委員会への報告義務が生じる
- 最高レベルのアクセス制御・暗号化が必要

**該当データ種別**:

- 患者の氏名・生年月日・住所（識別情報）
- 診断名・病状・処方内容・服薬記録（医療情報）
- 訪問記録・ケアレポート（医療行為の記録）
- 検査値・バイタルサイン（生体情報）
- 保険証番号・被保険者番号（社会保障番号相当）

### 1.2 内部（Internal）— PII / 業務データ

**定義**: 個人情報（氏名・連絡先等の一般個人情報）または内部業務データ。漏洩した場合に業務上・競争上の不利益が生じるデータ。

**特徴**:

- アクセス制御が必要（ロールベース）
- 暗号化（at rest + in transit）が必要
- 外部公開不可

**該当データ種別**:

- 薬剤師・スタッフの個人情報（氏名・連絡先・資格情報）
- 組織（薬局）の業務データ・設定・シフト
- 請求候補・算定根拠データ
- 内部の監査ログ・操作ログ

### 1.3 一般（Public）

**定義**: 外部公開可能なデータ。漏洩しても個人・組織への実質的な不利益が生じないデータ。

**特徴**:

- 特別な保護は不要
- システムの正常動作に必要な公開情報

**該当データ種別**:

- 医薬品マスタ（DrugMaster / DrugPackage / DrugPackageInsert）: 公的データソースから取得した公開情報
- 画面文言辞書（LabelDictionary）
- システムステータス・ヘルスチェックレスポンス

## 3. データ保持ポリシー

### 3.1 法令基準

| データ種別                       | 法令上の保存義務                       | PH-OS の保持期間                       |
| -------------------------------- | -------------------------------------- | -------------------------------------- |
| 薬歴（処方・調剤・訪問記録）     | 薬剤師法施行規則第 15 条の 2: **3 年** | **5 年**（法令超過・ガイドライン推奨） |
| 療担規則上の帳票（管理計画書等） | **5 年**                               | **5 年**                               |
| 監査ログ・操作ログ               | MHLW ガイドライン: **最低 3 年**       | **5 年**                               |
| 個人情報保護法上の記録           | 第三者提供記録: **3 年**               | **5 年**                               |

### 3.2 システム別保持期間

| データ種別                                                     | 保持期間                                                         | 保持形式                                                                  |
| -------------------------------------------------------------- | ---------------------------------------------------------------- | ------------------------------------------------------------------------- |
| 薬歴・訪問記録・報告書                                         | **5 年**                                                         | RDS（論理保持） + S3 アーカイブ                                           |
| 監査ログ（AuditLog テーブル）                                  | **5 年**                                                         | RDS（論理保持） + S3 エクスポート                                         |
| 処方原本ファイル（S3）                                         | **5 年**                                                         | S3 Standard → S3 Glacier（2 年後）                                        |
| セッショントークン                                             | **24 時間**（Cognito アクセストークン有効期限）                  | Cognito 管理                                                              |
| 一時ファイル（薬歴一括出力 ZIP・PDF 生成・一時 presigned URL） | **アプリ内ダウンロード期限 72 時間、S3 ライフサイクル上限 7 日** | `bulk-export-artifact-cleanup` ジョブ + S3 ライフサイクルルールで自動削除 |
| バックアップ・スナップショット（RDS）                          | **35 日**                                                        | RDS 自動スナップショット                                                  |
| S3 バージョニング（旧版ファイル）                              | **90 日**                                                        | S3 バージョニング + ライフサイクル                                        |
| CloudTrail ログ                                                | **5 年**                                                         | S3（Object Lock: Compliance mode）                                        |
| CloudWatch ログ                                                | **1 年**                                                         | CloudWatch Logs（Glacier 移行後 1 年）                                    |

## 4. 削除ポリシー

### 4.1 削除の原則

PH-OS では、患者の医療記録（薬歴・訪問記録等）を保持期間内に物理削除してはならない。削除は以下の 3 段階で管理する:

```
論理削除（is_deleted フラグ）
    ↓ 保持期間経過
物理削除（月次バッチ）
    ↓
削除ログ（AuditLog に記録）
```

### 4.2 論理削除

| 対象               | 実装                                                                        | 備考                                                            |
| ------------------ | --------------------------------------------------------------------------- | --------------------------------------------------------------- |
| 患者レコード       | `deleted_at IS NOT NULL` でフラグ管理                                       | 業務上「終了」した患者                                          |
| ユーザーアカウント | Cognito `DisableUser` + `users.deleted_at`                                  | 退職・利用停止                                                  |
| 処方・調剤記録     | 論理削除なし（修正は追記式）                                                | 薬歴は改ざん防止のため不変                                      |
| 一時ファイル       | `bulk-export-artifact-cleanup` ジョブ + S3 ライフサイクルポリシーで自動削除 | 薬歴一括出力 ZIP は 72 時間でダウンロード不可、S3 は 7 日で削除 |

### 4.3 物理削除（月次バッチ）

実行タイミング: 毎月第 1 月曜日 03:00 JST

対象: 以下の条件を全て満たすレコード

1. `deleted_at IS NOT NULL` または保持期間を経過している
2. 保持期間（該当カテゴリの最長保持期間）を経過している
3. 監査ログへの影響確認が完了している（外部キー依存の確認）

物理削除の手順:

1. 削除対象レコードのリストを生成（バッチログに記録）
2. 監査目的のメタデータ（patient_id, created_at, deleted_at 等）を削除ログテーブルに移行
3. 対象レコードを物理削除
4. 削除件数・所要時間を AuditLog に記録
5. 削除エラーがある場合はアラートを発報

### 4.4 個人情報削除依頼への対応（APPI 第 34 条）

患者または代理人から個人情報の削除依頼があった場合:

1. 本人確認（身分証明書・代理権の確認）
2. 法令上の保存義務（薬剤師法等）との照合
3. 保存義務がある場合: 利用停止で対応（物理削除は不可）
4. 保存義務がない場合: 対応可能な範囲で論理削除・匿名化を実施
5. 対応結果を書面で本人に通知（30 日以内）
6. 対応記録を AuditLog に保存

## 5. 暗号化要件

### 5.1 保存時暗号化（at rest）

| データ保存場所                             | 暗号化方式                          | キー管理                            |
| ------------------------------------------ | ----------------------------------- | ----------------------------------- |
| RDS PostgreSQL                             | AES-256（AWS KMS）                  | KMS マネージドキー（`aws/rds`）     |
| S3 バケット（処方原本・報告書）            | AES-256-GCM（AWS SSE-S3 / SSE-KMS） | KMS マネージドキー（`aws/s3`）      |
| S3 バケット（監査ログアーカイブ）          | AES-256-GCM（SSE-KMS）              | KMS カスタマーマネージドキー（CMK） |
| IndexedDB（PWA オフラインキャッシュ）      | AES-256-GCM                         | 環境変数 `ENCRYPTION_KEY`（256bit） |
| Secrets Manager（DB パスワード・API キー） | AES-256（Secrets Manager 管理）     | KMS マネージドキー                  |

機密データ（PHI）は特に:

- RDS 保存時: PostgreSQL Row-Level Security + KMS 暗号化の二層保護
- S3 保存時: サーバーサイド暗号化（SSE-KMS）必須
- IndexedDB: PHI を含むオフラインキャッシュは `ENCRYPTION_KEY` で暗号化必須

### 5.2 転送時暗号化（in transit）

| 通信経路                                   | 暗号化要件                                             |
| ------------------------------------------ | ------------------------------------------------------ |
| クライアント（ブラウザ） → Amplify Hosting | **TLS 1.3**（HTTPS 強制、HTTP 自動リダイレクト）       |
| Amplify → RDS                              | **TLS 1.3**（`rds.force_ssl = 1` + `sslmode=require`） |
| Amplify → S3                               | **TLS 1.2 以上**（AWS SDK デフォルト）                 |
| Amplify → SES                              | **TLS 1.2 以上**（AWS SDK デフォルト）                 |
| Amplify → Cognito                          | **TLS 1.2 以上**（HTTPS エンドポイント）               |
| 外部共有リンク                             | **TLS 1.3** + 署名付き JWT トークン（期限付き）        |

### 5.3 暗号化アルゴリズム基準

| 用途                          | アルゴリズム                | 鍵長                                 |
| ----------------------------- | --------------------------- | ------------------------------------ |
| データ暗号化（対称）          | AES-GCM                     | 256 bit                              |
| TLS 証明書                    | RSA または ECDSA            | RSA 2048 bit 以上 / ECDSA P-256 以上 |
| パスワードハッシュ（Cognito） | SRP v6a（AWS Cognito 管理） | —                                    |
| JWT 署名（外部共有トークン）  | HS256（HMAC-SHA256）        | 256 bit（`JWT_SIGNING_SECRET`）      |
| ファイル完全性検証            | SHA-256                     | —                                    |

## 6. データ分類の適用・レビュー

### 6.1 新規テーブル追加時のルール

新しいテーブルまたはカラムを追加する際は以下を必ず実施する:

1. このドキュメントのセクション 2 に機密度ラベルを追加（PR レビューの必須チェック項目）
2. 機密データを含む場合は RLS ポリシーが適用されることを確認
3. 監査ログ（AuditLog）の記録対象かどうかを判断し、必要に応じて実装

### 6.2 年次レビュー

- 年 1 回、データ分類の全テーブル棚卸しを実施
- 法令改正・ガイドライン改訂に応じてアップデート
- レビュー結果をこのドキュメントの更新履歴に記録
