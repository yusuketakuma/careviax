# インシデント対応手順書

## 概要
PH-OS における情報セキュリティインシデント発生時の対応手順を定義する。
3省2ガイドライン（MHLW v6.0 + METI/MIC v1.1）に準拠。

## インシデント分類

| レベル | 分類 | 例 | 目標対応時間 |
|---|---|---|---|
| **Critical** | 個人情報漏洩・システム全停止 | PHI流出、DB破損、ランサムウェア | 1時間以内 |
| **High** | 一部機能停止・データ不整合 | API障害、認証障害、データ同期不良 | 4時間以内 |
| **Medium** | 性能劣化・軽微な不具合 | レスポンス遅延、UI不具合 | 24時間以内 |
| **Low** | 情報提供・改善要望 | ログ異常検知、設定ミス | 次営業日 |

## 対応フロー

### Phase 1: 検知（Detection）

1. **自動検知**
   - AWS CloudWatch アラート（CPU/メモリ/エラー率）
   - ヘルスチェック API（`/api/health`）の異常検知
   - CloudTrail による不正アクセス検知
   - アプリケーションログのエラーパターン監視

2. **手動報告**
   - ユーザーからの障害報告
   - 定期巡回での異常発見
   - 外部からの脆弱性報告

3. **検知時の即座アクション**
   - インシデントチケット起票（日時・検知方法・概要を記録）
   - インシデントレベルの暫定判定
   - 対応担当者のアサイン

### Phase 2: 報告（Notification）

| レベル | 報告先 | 報告手段 | タイミング |
|---|---|---|---|
| Critical | 経営層 + セキュリティ責任者 + 全関係者 | 電話 + メール | 即座 |
| High | セキュリティ責任者 + システム管理者 | メール + チャット | 30分以内 |
| Medium | システム管理者 | チャット | 4時間以内 |
| Low | システム管理者 | チケット | 次営業日 |

**個人情報漏洩の場合の追加報告先:**
- 個人情報保護委員会（速報: 発覚から3〜5日以内、確報: 30日以内）
- 本人への通知（速やかに）
- 関係医療機関への通知

### Phase 3: 隔離（Containment）

1. **即時隔離措置**
   - 不正アクセス: 該当ユーザー/IPのブロック（Cognito + WAF）
   - データ漏洩: アクセストークン無効化、該当APIエンドポイント停止
   - マルウェア: 該当リソースのネットワーク隔離

2. **証拠保全**
   - CloudTrail ログの保護（S3 Object Lock）
   - CloudWatch ログのエクスポート
   - RDS スナップショット取得（事後分析用）
   - 影響範囲の特定（アクセスログから影響ユーザー・データを洗い出し）

3. **影響範囲の確認**
   - 影響を受けた患者数・データ種別の特定
   - 影響を受けた組織・薬局の特定
   - 二次被害の可能性評価

### Phase 4: 復旧（Recovery）

1. **原因特定**
   - ログ分析による根本原因の特定
   - 脆弱性の有無確認
   - 攻撃経路の特定（不正アクセスの場合）

2. **復旧作業**
   - パッチ適用またはコード修正
   - RDS スナップショットからのリストア（必要時）
   - S3 バージョニングによるファイル復元（必要時）
   - 認証情報のローテーション（漏洩の場合）

3. **復旧確認**
   - ヘルスチェック API による正常性確認
   - 主要機能の動作確認
   - データ整合性チェック
   - モニタリング強化（再発の兆候監視）

### Phase 5: 再発防止（Post-Incident）

1. **事後レポート作成**（インシデント収束から5営業日以内）
   - タイムライン（検知〜復旧の時系列）
   - 根本原因分析（RCA）
   - 影響範囲の最終確定
   - 対応の評価（良かった点・改善点）

2. **再発防止策の実施**
   - セキュリティパッチの適用
   - 監視ルールの追加・調整
   - アクセス制御の見直し
   - 手順書の更新

3. **教育・訓練**
   - インシデント事例の共有
   - 対応手順の訓練実施（年1回以上）
