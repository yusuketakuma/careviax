# CareViaX / PH-OS Repository Audit - 2026-06-10

## Executive Summary

Overall health grade: **B-**. The repository is a production-leaning healthcare/pharmacy operations app with mature CI, real domain modeling, and many targeted tests, but several core workflows are still concentrated in very large files with weak branch coverage. The top remaining risks are: large high-change modules that are hard to review safely, legacy plaintext webhook signing secrets until backfill/contract migration is completed, and low/zero coverage around some jobs, PDFs, adapters, and operational routes. The highest-priority security issues found during this audit cycle were materially remediated: protected routes now fail closed when auth secret configuration is missing, PH-OS browser calls go through a same-origin server proxy instead of exposing bearer material, PH-OS logs hash sensitive identifiers, outbound webhooks now have a durable retry path, and new webhook registrations no longer store plaintext signing secrets. The top opportunities are: shrink the largest modules behind tested service/view-model boundaries, keep date/time semantics enforced through the explicit allowlist gate, and turn dependency/type hygiene into a recurring small upgrade process. No known vulnerabilities were reported by `pnpm audit --audit-level moderate` on this run. `pnpm test:coverage -- --reporter=dot` currently passes with 5,199 tests and 81.82% statement coverage, but branch coverage remains only 67.36%.

## Repo Map

**Purpose.** `README.md` defines PH-OS Pharmacy as a home-visit-friendly insurance-pharmacy workflow and collaboration platform for dispensing, audits, set audits, visit scheduling, and multi-professional collaboration (`README.md:1-4`).

**Stack.** The app is a private Node 24 / Next.js 16 / React 19 / TypeScript repository with Prisma 7, PostgreSQL adapter, NextAuth v4, AWS SDK, Sentry, Serwist, Vitest, Playwright, Tailwind, React Query, Zod, and healthcare-specific service code (`package.json:1-19`, `package.json:65-145`). `next.config.ts` composes Serwist and Sentry, enables Next auth interrupts, and keeps CSP in the request proxy because the CSP nonce is per-request (`next.config.ts:5-15`, `next.config.ts:17-37`).

**Architecture sketch.**

- `src/app/`: Next.js App Router pages and route handlers, including dashboard, PH-OS pages, and API routes (`README.md:18-24`).
- `src/server/`: server-only jobs, services, mappers, adapters, and orchestration (`README.md:20-22`).
- `src/phos/`: PH-OS contract, frontend client, Lambda-style backend handlers, API Gateway route catalog, and observability surfaces (`src/phos/infra/api-gateway-routes.ts:28-60`).
- `prisma/schema/`: domain model split across Prisma schema files; `IntegrationJob`, `WebhookRegistration`, and `WebhookDelivery` are in the admin schema (`prisma/schema/admin.prisma:182-204`, `prisma/schema/admin.prisma:320-360`).
- `tools/`: deployment, job schedules, browser harness, E2E tests, performance, pilot, DB, and ops scripts (`README.md:22-25`, `tools/infra/eventbridge-schedules.json:1-24`).
- `docs/`: compliance, operations, testing, UI/UX SSOT, and project documentation (`README.md:27-36`).

**Main control flow.** Browser pages use React Query and route handlers under `src/app/api`. API handlers enforce auth via `requireAuthContext` / `requireApiKeyOrAuthContext` and resolve org membership before executing route logic (`src/lib/auth/context.ts:94-219`, `src/lib/auth/context.ts:242-264`). Cross-cutting API CSRF, rate limiting, auth redirects, CSP, and security headers are centralized in `src/proxy.ts` (`src/proxy.ts:205-327`, `src/proxy.ts:334-374`). Job execution enters through `/api/jobs/[jobType]`, dispatches to `src/server/jobs`, and persists `IntegrationJob` rows (`src/app/api/jobs/[jobType]/route.ts:50-91`, `src/server/jobs/runner.ts:35-111`). PH-OS runtime calls are routed through a route catalog and a same-origin proxy to the configured PH-OS upstream (`src/phos/infra/api-gateway-routes.ts:28-60`, `src/app/api/phos/[...path]/route.ts:108-146`).

**Conventions observed.** Route handlers are mostly small wrappers around Prisma/services, Zod validation is common, tests live beside implementation as `*.test.ts` / `*.test.tsx`, and Japanese user-facing messages are embedded in route/service output while code identifiers remain English. UI/UX work has an explicit SSOT requiring meaningful information groups, common scaffolds, and heading hierarchy (`docs/ui-ux-design-guidelines.md:1-4`, `docs/ui-ux-design-guidelines.md:99-118`).

**Surprises.** The repo is much more production-shaped than the `0.1.0` version number suggests: CI includes lint, format, typecheck, coverage, build, migration gates, RLS checks, medical UI E2E, and manual production deployment approval (`.github/workflows/ci.yml:24-77`, `.github/workflows/ci.yml:98-187`, `.github/workflows/ci.yml:192-245`, `.github/workflows/ci.yml:295-346`). The biggest mismatch is not missing infrastructure; it is that some core business/UI/PDF modules are still monolithic.

### Code Quality

**High - Fixed in this cycle: stale job locks could permanently block recurring jobs.** `runJob` previously skipped any existing `running` row. It now treats only null or recent `locked_at` values as active, with a bounded stale-lock window (`src/server/jobs/runner.ts:5-28`). Consequence: an interrupted process is less likely to block recurring jobs forever while still preventing normal duplicate execution.

**Medium - Fixed in this continuation: direct local-calendar date slicing is removed and gated.** Shared date helpers exist (`src/lib/date-key.ts:1-20`) and stricter date-key validation exists (`src/lib/validations/date-key.ts:3-16`). Remaining direct `toISOString().slice(0, 10)` sites are classified in an allowlist with domain rationale and are limited to UTC audit/job/canonical-month/contract semantics (`tools/date-slice-allowlist.json:1-54`). CI runs the classification check (`.github/workflows/ci.yml:55-56`), and the check fails on unclassified or stale entries (`tools/scripts/check-date-slices.mjs:75-143`). Consequence: future JST/UTC calendar regressions now require an explicit domain decision instead of slipping in as ad hoc string slicing.

**Medium - Remaining: type checking still skips library checks.** `strict` is enabled, which is good, but `skipLibCheck` remains true (`tsconfig.json:6-7`). Consequence: broken or incompatible ambient dependency types can remain hidden, which matters with Next/React/Prisma major-version movement.

**Low - Good pattern to preserve: invalid calendar dates are no longer only regex-checked in shared date-key paths.** `isValidDateKey` verifies year, month, and day by round-tripping through UTC (`src/lib/validations/date-key.ts:5-11`). This should become the default for user-supplied `YYYY-MM-DD` input.

### Security

**High - Fixed in this cycle: protected app routes fail closed when auth secret is missing.** Production-like protected routes now return a 503 configuration error when no auth secret is available (`src/proxy.ts:142-158`, `src/proxy.ts:299-304`). Consequence: a misconfigured deployment no longer silently serves protected app routes without a verifiable session token.

**High - Fixed in this cycle: PH-OS browser code no longer needs bearer material in the client session.** The session callback exposes user/org/session metadata but not access, refresh, or ID tokens (`src/lib/auth/config.ts:175-187`), while server-side `getAuthAccessToken` reads the token for proxy use (`src/lib/auth/config.ts:203-210`). The PH-OS proxy then adds the bearer header server-side (`src/app/api/phos/[...path]/route.ts:85-94`, `src/app/api/phos/[...path]/route.ts:108-146`). Consequence: XSS/client-state exposure has a much smaller blast radius.

**High - Fixed in this cycle: outbound webhook delivery now has SSRF protection and durable retry.** Webhook URLs must be HTTPS and cannot resolve to loopback/private/link-local/unsafe IP ranges (`src/server/services/outbound-webhook.ts:247-265`). Delivery state is persisted with event, payload, URL, status, attempt count, and retry time (`prisma/schema/admin.prisma:336-360`, `prisma/migrations/20260610180500_add_webhook_delivery_outbox/migration.sql:1-42`). **Medium - Partially fixed: new webhook signing secrets are encrypted, and legacy backfill is scriptable, but DB apply/contract remain approval-bound.** `WebhookRegistration.secret` is now nullable and encrypted secret columns were added (`prisma/schema/admin.prisma:320-339`). ## Improvement Strategy

### Theme 1: shrink high-risk modules behind stable boundaries

Target state: no core UI/service/PDF module owns unrelated data fetching, domain decisions, formatting, rendering, and mutation orchestration in one file. Principle: extract around behavior seams with characterization tests first; do not rewrite UI and business rules simultaneously.

### Theme 2: make date/time semantics explicit

Target state: every `YYYY-MM-DD` path uses either `formatDateKey`, `formatUtcDateKey`, or a documented allowlist entry. Principle: calendar dates, timestamps, dedupe keys, and filenames are different types of data and should not share ad hoc ISO slicing.

### Theme 3: keep operational durability visible

Target state: recurring jobs, webhook delivery, bulk exports, and external integrations have durable state, bounded retries, stale-lock handling, and drift checks. Principle: user-facing healthcare workflows need recoverability more than optimistic in-process execution.

### Theme 4: move from aggregate coverage to risk-weighted coverage

Target state: aggregate coverage remains green, but weak critical files get local thresholds or named regression suites. Principle: one large high-risk file at 20% branch coverage is more important than an aggregate percentage.

### Theme 5: schedule dependency and type hygiene

Target state: dependency updates are a routine patch/minor batch with typecheck/build/E2E proof, and `skipLibCheck` is eventually disabled. Principle: avoid surprise major-version breakage by paying upgrade cost in small batches.

### Trade-offs

- Do not start with a visual redesign rewrite of `ScheduleDayView`; first extract view-models and add characterization tests so UI changes do not mutate scheduling semantics accidentally.
- Do not chase 100% coverage; focus on jobs, PDFs, patient detail, adapters, and dangerous route branches.
- Do not upgrade TypeScript 6 or major dependency lines casually; start with patch/minor packages that are within the existing major line, then handle major upgrades separately.
- Do not finish webhook secret encryption with a one-step destructive migration; the expand/contract first phase is in place, but legacy rows need an approved backfill/rotation step before removing plaintext compatibility.

### Done Signals

- Zero remaining Critical/High audit findings.
- `pnpm lint`, `pnpm format:check`, `pnpm typecheck`, `pnpm test:coverage -- --reporter=dot`, `pnpm build`, and focused changed-area tests all pass.
- Branch coverage threshold raised from 66% to at least 75%, then 80% after core workflow coverage improves.
- No direct `toISOString().slice(0, 10)` remains outside an explicit allowlist with UTC/canonical/timestamp rationale.
- `skipLibCheck` can be set to false without typecheck failures.
- No source module in the top-risk paths remains above roughly 1,500 lines without a documented extraction plan.
- New webhook signing secrets are not stored as plaintext in the registration row, and legacy plaintext rows have a completed backfill/contract migration.

### Top 3 Implementation Sketches

**1. Webhook secret encryption.** The first expand phase is implemented: encrypted fields were added, new registrations store ciphertext, and delivery dual-reads encrypted or legacy plaintext secrets. The remaining step is an approved backfill/rotation run for existing rows, followed by a contract migration that removes plaintext compatibility. Gotcha: HMAC signatures must remain byte-for-byte stable for existing endpoints during migration.

**2. Date-key cleanup gate.** Start with an allowlist file that classifies each remaining direct slice as local date, UTC date, timestamp-derived filename, or legacy exception. Replace high-risk local-calendar cases with `formatDateKey`, UTC cases with `formatUtcDateKey`, and add boundary tests using JST midnight-adjacent fixtures. Gotcha: billing months and dedupe keys may intentionally use UTC/canonical month semantics, so do not bulk replace blindly.

**3. Patient-detail extraction.** Freeze current behavior with DTO-focused tests, then extract read concerns one at a time: access/where builders, timeline slices, readiness, workflow preview, privacy masking, and presentation labels. Keep the public `getPatient...` functions stable until all callers are moved. Gotcha: this service crosses privacy, assignment scope, and external sharing; extraction must preserve authorization filters exactly.
