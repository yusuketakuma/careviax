# UI/UX Design System

本書は PH-OS（在宅訪問薬局プラットフォーム）の UI/UX 改修を**実装レベルで一貫適用するための原則集**である。
方針の SSOT は `docs/ui-ux-design-guidelines.md`（以下「ガイドライン」）と `docs/state-color-migration-map.md`（状態色の family×value×role 確定表）であり、本書はそれを**置き換えず参照する**。ガイドラインが「何を・なぜ」を定めるのに対し、本書は「具体的にどう書くか／何を禁じるか」の実装規範を定める。衝突時はガイドライン＞本書、個別要件＞ガイドラインの順で優先し、差分理由を明記する。

本書は 2026-06-26 の全画面 UI/UX 監査（auth / admin マスタ・臨床・運用分析 / 患者 / 処方せん / 調剤ワークベンチ / 訪問 / スケジュール / 報告・請求 / 連携・コアナビ）で横断的に検出された逸脱・医療安全リスク・共通部品ギャップを正本化したものである。

---

## デザイン方針

PH-OS の UI は「読み物」ではなく**毎日使う業務ワークベンチ**である。派手さではなく、信頼感・明快さ・速度（クリック削減）を最優先する。

- **足し算・引き算は視覚ノイズの操作であり機能削除ではない**。`dense ≠ cluttered`。密度は size / contrast / position / weight の情報階層で制御する。
  - 引き算する: 状態意味を持たない装飾カード・影・グラデーション、重複表示（タイトルとパンくず、上部 Strip と最下部 Rail の二重描画）、意味のない区切り線、生 Tailwind 状態色ベタ書き、低頻度機能の常時露出。
  - 足し算する: 能動的余白、見出し・位置・コントラストによる階層、状態の explicit 化、走査性（見出し→サマリー→詳細）、高頻度操作のショートカット、安全タグの常時可視化。
  - **引き算してはいけない**: 臨床判断に必要なデータ・状態・安全タグ（麻薬/アレルギー/ハイリスク）・監査痕跡。
- **共通部品を先に直す**。似た実装を画面ごとに増やさない。本監査で判明した未整備の共通部品（後述の各節）を新設・拡張してから個別画面へ展開する。
- **情報重力ゾーン（Pinned / Primary / Scroll）を全画面に適用**する。「これが見えなければ患者に害が及ぶ情報」だけを Pinned に置き、即時対応・最重要データを Primary（fold 内）に、履歴・補助を Scroll に置く。
- **trunk test を 3 秒で通す**: PH-OS であること・現在ページ・主要導線・次にできる操作・戻り先が即座に分かること。

## 医療システムとしてのトーン

- **基調は深ネイビー（レセコン `#1f4e79` 相当 = `--primary: oklch(0.38 0.09 252)`）**。信頼・清潔感を表す。白ベースで高コントラスト。彩度の高い面は重要操作と警告にのみ集中させる。
- **アラート色（赤/橙）は希少資源**として扱う。「赤を見たら必ず行動が必要」を守る。常態的に赤・橙が点灯する画面（例: API P95 タイルが達成時も常時橙、0 件でも至急通知が赤）は alert fatigue を生むため禁止。状態色は値×閾値で点灯/消灯を制御する。
- **偽の安全保証・偽シグナルを出さない**。実データ非連動のハードコード ✓（「社用車: 使用可」「薬剤師負荷: 許容範囲」を常に緑）、未接続時の全 0 表示を実測のように見せる、永続化されない「採用」操作で「直した」と誤認させる、件数を読み込み済み window のみで集計して過小表示する——これらはすべて医療安全リスクとして排除する。
- **PHI（要配慮個人情報）はマスクを既定**にする。電話/FAX/保険者番号/住所/患者氏名は、一覧・生成ビュー・外部共有・生 JSON ダンプで生値を露出させない。「変更あり」「記録あり」の事実とフィールド名のみを示し、生値は権限付き正本画面でのみ確認する。
- **破壊的・広域・準不可逆操作には確認を必須**にする（月次締め・マスタ全件取込・薬価更新・外部共有リンク発行・貸出取消・推奨車両一括反映）。確認には対象・件数・金額・取消不可の旨を提示する。
- **サンプル/未接続画面は明示**する。モック駆動のワークベンチや実データ未接続マスタには「サンプル表示（実データ未接続）」を明示し、本番同型 UI を機能するかのように見せない。

## 色の使い方

状態色は `globals.css` の中央トークン（`--state-*` / `--tag-*`）と共通部品 `StateBadge` / `StatusDot`（`role` を渡す）でのみ表現する。**6軸ロールが正本**であり、`docs/state-color-migration-map.md` の family×value×role 確定表に従う。

| role | 色 / token | 意味 |
| --- | --- | --- |
| `info` | 青 `--tag-info` | 通常の主操作 / 現在地(current) / 情報タグ(処方変更・返信待ちを一覧に出す) / 予定・待ち |
| `blocked` | 赤 `--state-blocked` | 止まっている理由 / ブロッカー / キャンセル / 通信なし / 送付失敗 |
| `done` | 緑 `--state-done` | 完了 / 承認済 / 確認済 |
| `confirm` | 橙 `--state-confirm` | 確認が必要 / 保留 / 差戻し / 延期 / 要対応 |
| `hazard` | 橙 `--tag-hazard` | 麻薬 / 冷所 / インスリン / 抗凝固 等の危険タグ（隠さない） |
| `waiting` | 紫 `--state-waiting` | 別の人(薬剤師/事務)の確認待ち |
| `readonly` | 灰 `--state-readonly` | 閲覧のみ / 権限なし / 終了・退院 / 中立 |
| `neutral` | 状態色なし | 既定 Badge / `text-muted-foreground`。稼働中(active)・下書き(draft) はこちら |

**色を増やさない・意味を固定する**:

- 生 Tailwind 状態色の直書き（`bg-emerald-100` / `text-blue-500` / `bg-rose-50` / `text-red-700` / `bg-slate-50` 等）は全面禁止。トークンへ置換する。
- raw shadcn `Badge variant`（`destructive` / `secondary` / `outline`）やローカル `statusVariant` を状態表現に使わない。`destructive`（赤）を「終了・期限切れ・休業日・回答待ち・低負荷状態」など**止まっていない状態**に流用しない（赤の濫用は本当の緊急の区別を失わせる）。
- 「回答待ち（疑義照会 pending）」は他者確認待ち＝`waiting`（紫）であり、`blocked`（赤）にしない。画面間で同一データを別色にしない。
- **塗り面積を最小化する**（2026-06-26 ratified）。状態は「面」でなく「点・線・ラベル」で伝える。
  - ステータスタイル/カードは背景 `bg-card`（中立）を保ち、状態は ①左ボーダー `border-l-4 border-l-state-*` ②ラベル文字色 `text-state-*` の2点で表す。`bg-state-*/10` をタイル全面に敷くのは不可。
  - 例外: 小さな件数ピル/章バッジ（`rounded-full`）は `bg-state-*/10 text-state-*` 可（面積が小さく点として機能するため）。
- **chart 系列色は状態色ではない**。`--chart-1..5` を使い、`--state-*` / `--tag-*` を系列へ流用しない。KPI 値の文字色に系列色を当てて装飾化しない（閾値超過時のみ confirm/blocked の点表示を足す）。
- **識別目的の固定色は別系統**として許可（SOAP の S/O/A/P、証跡カテゴリ、服薬カレンダー時間帯）。系列性は `--chart-*`、土日等の慣習色は専用トークン `--weekend` を新設し、生 Tailwind 直書きはしない。
- 変更タイプ色は 6軸へ写像（追加=info / 変更=info / 解除=readonly）。差分カードを `bg-state-*/10` 全面塗りにしない。

## タイポグラフィ

- **フォントスタック**: Meiryo → Noto Sans JP → system-ui（`--font-sans`）。Windows は Meiryo、他は next/font の Noto Sans JP、最終 system-ui。
- **下限を厳守**: 本文 14px 以上、ラベル 12px 以上、行間 1.6 以上。高密度画面でも下回らない。`text-[9px]` / `text-[10px]` / `text-[11px]`（処方一覧・inline-detail・ガント・カレンダーチップ・ExpiryCell 等で多発）は廃し、最小 `text-xs`(12px) へ。密度はブロック高・余白で調整する。
- **タイポグラフィスケールを role 単位で定義・再利用**（display / title / section / body / label / caption）。画面ごとの即席サイズ（`text-xl` と `text-2xl` の h1 混在など）を禁止。h1 は1つのスケールに統一する。
- **等幅数字 `tabular-nums` を必須**: 用量・薬価・点数・保険者番号・残数・余剰日数・金額・件数・残日数・部屋番号・時刻など、縦に並ぶ/比較する数値すべてに `font-variant-numeric: tabular-nums` を明示する。用量・徴収額の誤読防止に直結する。
- **見出し階層を飛ばさない**: ページ `h1` → 大グループ `h2` → セクション `h3`。`CardTitle` をセクション見出しに使う場合は `asChild` で `h2`/`h3` を付与する（装飾ラベル用途のみ text 可）。`h1`(sr-only)→`h3` のスキップや `CardTitle`(div) 混在を作らない。
- **コード記法をUI文言に漏らさない**: `match_keywords` / `in_app` / `sms` のようなバッククォート・生 enum キーを利用者向け文言にそのまま出さない。日本語ラベルへマップする。

## 余白

- **8pt グリッド**: margin / padding は 8 の倍数（8 / 16 / 24 / 32）。アイコン・小テキストは 4pt half-step。`mt-14` や `xl:pt-20` のようなマジック余白を作らない。
- ページ全体の主要グループ間は `space-y-6` 以上。グループ内主要セクション間は `space-y-4`〜`space-y-6`。見出しと本文の間は最低 8px。
- **Pinned クラスタは密に束ねる**: 識別ストリップ・セーフティボードなど Pinned zone の要素は内側 `space-y-3` で密集させ、以降の主要セクションと `space-y-6` で分離する。塊として一目で「常に一緒に見る情報」と分かるようにする。
- **固定 min-height を多用しない**: `min-h-[720px]` / `min-h-[760px]` / `min-h-[820px]` / `min-h-[190px]` のようなデータ非連動の巨大固定高は、内容が少ない時に巨大空白を生む。内容追従にし、列高さは grid で揃える。能動的余白とデッドスペースを区別する。
- 影（`shadow-sm`）は状態意味を持たないなら削り、`border` ＋余白で階層を表す。

## カード設計

- **1件ずつの小要素を全部カード化しない**（ガイドライン §104）。同じ役割の項目群を大グループで包み、その中で必要時のみ小カードを使う。連絡先・服薬・候補・ノートを1件ごとに `rounded-2xl` カード化して縦に積むのは禁止（罫線リスト / DataTable へ寄せる）。
- **角丸は控えめ**: `radius: 0.375rem`（`rounded-md`）を基準にする。`rounded-2xl` / `rounded-xl` の常用（認証クラスタ全面・proposals・[id] サマリ等）は `rounded-md` へ統一する。
- グループの入れ子は外側を強く（枠線+弱い背景+角丸）、内側を弱く（Separator / 罫線）見せる。装飾カードを増やさない。
- **KPI ストリップは例外**として横並び許容（1件ずつカード化禁止の例外）。ただし意味グループ外枠で内包し、影・過剰角丸は最小に。共通 `StatCard`(value / label / caption / icon / tone / isLoading) に集約し、画面ごとの `MetricCard` / `KpiCard` / `SummaryCard` / `SignalTile` 再発明をやめる。
- **ステータスカード（cat.3）は全面塗り禁止**。`bg-card` ＋ `border-l-4 border-l-state-*` ＋ `text-state-*` ラベルで表す。

## テーブル設計

- **比較・照合・監査・一覧はテーブル優先**（カード羅列でなく）。`DataTable` 共通部品を使い、loading / error / empty / 再試行を内包させる。手組み table + 手動 zebra（残薬テーブル等）は `DataTable` へ寄せる。
- **行に状態色をベタ塗りしない**。麻薬・冷所・ハイリスク等の行属性は専用「安全」列のトークンベースバッジ（`getHandlingTagBadgeClass` 等）で表す。行識別は zebra stripe ＋罫線 ＋ sticky header で行う。`bg-amber-50/60` 等の行全面塗りは禁止。
- **数値列に `tabular-nums` を必須**付与する。
- **操作列は末尾（右）**に置く。比較軸（種別/コード/算定値/状態）を連続させ、操作を中央に挟まない。
- **破壊的操作を密な行内に常時露出しない**。失効・削除・停止・退職・取消は行内の控えめな outline+アイコンか `…` メニューへ畳み、確定は ConfirmDialog で受ける（モバイルの誤タップ防止）。
- **件数ストリップはサーバ総数に基づく**。読み込み済みページ window のみの集計で過小表示しない。window 集計しかできない場合は「読込済み N 件中」と明示する。一覧の上限（例: audit-logs 100 件固定）は「表示は直近 N 件」と明示し、暗黙の切り捨て（監査証跡の欠落）を防ぐ。
- 件数が増える一覧はページング/仮想化を検討する。長い患者名・機関名は truncate と折返しを混在させず行高を揃える。`overflow-x-auto` + `role="region"` でラップする。
- **件数サマリー/KPI はテーブルの「上」**（サマリー→詳細の走査順）。テーブル/カレンダーの下に集計を置かない。

## フォーム設計

- **入力画面はゆとり重視**（1カラム中心、`max-width` 許可対象）。一覧画面はデータ密度重視。
- **shadcn の `Input` / `Select` / `Checkbox` を使う**。生 `<select>` / `<input>` / 自作 opacity-0 チェックボックス / `NativeSelect` を新規に作らない。`Select` には SSR enum 漏れ対策として**明示 children（`SelectValue` に children）**を必ず付け、初期表示で生 enum 値（all / warning 等）が漏れないようにする。
- **タッチターゲット 44px 以上を部品の既定**で保証する。ルートレベルの `[&_input]:!h-11` / `[&_button]:!min-h-[44px]` / `!h-11` 包括 override や `h-11 min-h-[44px] sm:h-11 sm:min-h-[44px]` の同義四重反復をやめ、`Input` / `Button` / `Select` のサイズトークンで解決する。設定行の `h-8`(32px) は 44px へ。
- **入力中の離脱防止・自動保存**: 長大フォーム（患者 intake 等）は dirty 時の `beforeunload` / ルート離脱ガードと、タブ単位の下書き自動保存を入れる。安全情報（腎機能/嚥下/麻薬/アレルギー）を入力途中で失わせない。
- **二重送信防止**: 送信中は送信ボタンを disabled（`LoadingButton`）。
- **エラーは到達可能に**: 送信失敗時はエラー要素へスクロール+フォーカスし `role="alert"` / `aria-live` で通知する。長大フォーム末尾の送信ボタンから押した利用者が画面外エラーを見落とさないようにする。`error summary` を冒頭に出す。
- **重複/破壊系の確認**: 「それでも登録する」（重複 PHI 生成）は単一クリック解除でなく、チェック+確定の二段にして登録ボタン活性条件と連動させる。
- **生 JSON を一次入力にしない**: テンプレート本文・条件・エリア定義・設定一括編集を生 JSON textarea で直編集させない。高頻度フィールドは構造化フォーム（チップ入力等）、JSON は上級者向け折りたたみへ退避。JSON 保存は差分プレビュー+確認ダイアログ+離脱防止を付ける。
- **OTP / パスワード強度 / 確認パスワードトグル**は共通部品化する（`OtpInput`: paste / backspace / フォーカス送り / `aria-label` / `tabular-nums` / 44px、`PasswordStrengthField`: 強度メーター + 要件チェックリスト + 表示トグル）。要件達成は緑色のみに依存せず Check アイコン＋達成テキストを併記する。新パスワード欄と確認欄で表示トグルの有無を不一致にしない。
- 各正本項目に項目メタ（最終更新日 / 更新者 / 確認元）を 12px muted で併記する。
- 必要書類等のチェックリストには受領件数サマリーと未受領の注意表示を足す。

## ボタン設計（Primary/Secondary/Tertiary/Destructive/System の使い分け）

操作の優先順位を**全画面で統一**する。1画面の主操作は1つに絞る。

| 階層 | 用途 | 見た目 |
| --- | --- | --- |
| **Primary** | その画面で次にやる主操作（調剤キューへ / 登録 / 送信 / 採用・反映） | `--primary`（深ネイビー）塗り。1画面1つ |
| **Secondary** | 主操作に次ぐ操作（戻る一歩手前 / 別ビューを開く / 比較対象追加） | outline |
| **Tertiary** | 補助・離脱（一覧へ戻る / キャンセル / 閉じる） | ghost / link |
| **Destructive** | 破壊的・取消不可（削除 / 失効 / 強制終了） | `--state-blocked` 系。Thumb zone に置かない。ConfirmDialog 必須 |
| **System** | システム連携・広域 mutate（全件取込 / 薬価更新 / 月次締め / 同期） | outline。確認ダイアログで対象・件数・最終実行を提示 |

- 下部アクションバーで「次工程へ」を含む3ボタンが全部 outline、のように**階層が消える配置を作らない**。次にやる操作を視覚的に最上位にする。
- 主操作色に `done`（緑）を使わない（完了の意味と衝突）。「対応済みにする」等の完了アクションは `info`（青=主操作）、done 緑はバッジ/文字のステータス表示に限定する。
- 同一の主操作導線を1画面に二重に置かない（カード下「このカードへ」と next-action リンクの競合、推奨詳細と3案カードの二重採用導線など）。
- モバイルは主操作を下部 Thumb zone（Y 70〜90%）に固定する。下部固定バーは最大2ボタン、3つ以上は Drawer へ。

## バッジ設計（StateBadge/StatusDot 6軸トークン準拠）

- 状態は `StateBadge`（`role` を渡す）/ `StatusDot` のみで描く。`CYCLE_STATUS_CONFIG` の `variant`/`className` 直渡し、`statusConfig`、ローカル `statusVariant`、生 `Badge variant` を経由しない。処方サイクル工程・薬局間協力ドメイン状態（draft / pending / accepted / confirmed / returned / revoked 等）は**6軸ロールへの写像表を拡張**して共通定数化する。
- **色だけに依存しない**: `StateBadge` の `showIcon` を活かしアイコン＋ラベルを併記する。経過バッジ（X 日経過）・期限・優先度は lucide アイコンを併記する。
- **期限/有効期限バッジは共通 `ExpiryBadge(date, thresholds)`** に集約し、残日数→ role の閾値を SSOT 化する。標準段階: 期限切れ/30日以内=`blocked`（赤・即対応）、90日以内=`confirm`（橙・予定）、それ以遠=`done`/中立。各画面（credentials の ExpiryBadge、facility-standards の ExpiryCell、車両車検/保険、institutions 最終処方日）で再実装しない。「2日以上先は非表示」のように近接でない期限を消さず、弱色で常時表示する。
- **安全タグの折り畳みルール**: 麻薬・ハイリスク薬・アレルギー・抗凝固・冷所・インスリン等の重大タグは `+N` の裏に隠さず常時表示し、軽微タグのみ省略する共通 `SafetyTagBadge`（オーバーフロー対応）を使う。
- enum / コード値（schedule_status / report status / scope key）は日本語ラベルへマップしてからバッジ化する。生英値を外部閲覧者・事務に見せない。

## アラート設計（即時中断/要確認/状態/期限リマインドの分離）

アラートは**4段階を厳格分離**し、同じ見た目にしない（「赤を見たら必ず行動」を守る）。共通 `AlertBanner(severity)` / `AlertTier` 部品を新設し、各画面が `bg-state-*/10` 全面ボックスを自前実装するのをやめる。

| 段階 | 意味 | 見た目 | role |
| --- | --- | --- | --- |
| **1. 緊急中断** | 割込み・要即時行動（重大かつ患者固有の閾値超過 / 算定不可 / 期限切れ / 送付失敗 / 持参薬未確定） | 赤の強い左ボーダー + `role="alert"`(assertive) | `blocked` |
| **2. 要確認** | 通常の確認待ち（90日以内期限 / CDS 注意 / 差戻し） | 橙の左ボーダー | `confirm` |
| **3. 記録上の状態** | 定常リマインダー・進捗状態（書類状態確認等） | 中立 + ラベル、`role="status"` | `readonly` / 中立 |
| **4. 期限リマインダー** | 期日の予告（次回算定可・改定未作成） | 情報色の左ボーダー + 期限テキスト | `info` |

- **重大度をフラット化しない**: 「算定不可（収益直結=赤・即対応）」と「期限 90 日以内（橙・予定）」、「期限切れ（赤）」と「90 日以内（橙）」を1つの橙バナーに同居させない。赤バナーと橙バナーを分離する。
- **塗り面積最小**: バナーも左ボーダー+文字色を基本にし、全面塗りを避ける。
- **偽アラートを出さない**: 0 件でも点灯する至急通知（赤）、達成時も常時橙の KPI、`role="alert"`(assertive) を常設の状態/リマインダーに使う——これらを禁止。状態色は値>0・閾値超過時のみ点灯し、平常時は中立。
- 情報バナーに警告アイコン（AlertTriangle）を使わない（即時性の語彙混線）。情報は Info アイコン。

## モーダル設計

Z軸（深さ）6段階を用途・リスクで使い分ける（ガイドライン §233-258）。

- **Modal（深さ4）は破壊的・取消不可の操作のみ**（確定送付・削除・強制終了・月次締め・外部共有リンク発行）。情報表示・参照・低リスク操作に Modal を使わない。
- **確認ダイアログには対象・件数・金額・取消不可を提示**する。月次締め・マスタ全件取込・薬価更新・貸出取消・推奨車両一括反映・疑義の「変更ありで確定」など副作用/不可逆操作に必須。承認/却下ダイアログは raw JSON.stringify でなく before→after を項目化する。
- **本文参照しながらの作業は Drawer（深さ3）**。Escape / 外クリックで閉じ、本文の現在地を保持する。競合解決・重量フォームを別ビュー全置換にして現在地（ヘッダ）を消さない。
- **二択の不可逆操作**（「最新を使う=自分の入力破棄」/「自分で上書き」）は差分提示つきの共通確認部品で扱い、両側を同等の警告強度にする。
- メニュー（報告書宛先選択等）は共通 `DropdownMenu`（キーボード操作・フォーカストラップ・初期フォーカス）を使う。手組み absolute popover で宛先誤選択を誘発しない。
- 自動印刷（`window.print()`）を無人で発火させない（PHI 意図せず出力）。カウントダウン/明示確認、または初回は手動ボタンを既定にする。

## ナビゲーション設計

- **ヘッダ部品を統一**: 通常ページ=`WorkflowPageIntro`（戻り導線つき）、ハブ系=`WorkflowPageHeader`、admin=`AdminPageHeader`。ハブ/設定/選択系で当てはまらない簡易ヘッダは共通部品を新設し、画面内 raw `h1`（`text-xl`/`text-2xl` 混在）を本文直置きしない。
- **ワークスペース型トップは `page.tsx` に `<h1 className="sr-only">`** を置き、本文先頭は `<h2>` から始める。可視 h1 の本文直置き・条件分岐での h1 可視/sr-only 切替をしない。
- **PageScaffold を必ず経由**する。直返し（facility-packet / residual-adjustment）で外枠・幅・余白を不統一にしない。
- **左ナビ・右補助3点セット（次にやること/止まっている理由/根拠・記録）はドロワー**として上部バーのボタンから開く。通常状態で本文幅を占有しない。実装では「右レール最下部に縦積み」でなく、即時判断情報（next-action / blocked 理由）を Primary zone（fold 内）に出す。3点セットの配置を画面ごとにバラバラ（最下部縦積み / 3カラム目 / 上部 Strip と最下部 Rail 二重）にしない。1配置に統一する。
- **対象日ナビ（前日/今日/翌日）を共通 `DayNavigator`** として提供し、`?date` URL 依存・mount 時固定をやめる（毎日使うスケジュール/conflicts/emergency-route/route-compare の致命的弱点）。日付+件数ストリップを sticky な Pinned にする。
- **DOM 順と視覚順を入れ替えない**（`order-1` / `order-2` 禁止）。重要要素（workbench）は DOM 上でも先頭に置く。モバイルは source order のまま縦積みする。
- **画面役割の重複を解消**: /dashboard と /my-day、カード内共有パネルと /share ページなど、入口を二系統に分散させない。
- リダイレクト専用ルートに `loading.tsx` を残さない（ちらつき要因）。

## ローディング設計

- **スケルトン必須**。スピナー単独・テキスト「読み込み中...」「読込中...」単独は不可。`<Loading/>` スピナーを Suspense fallback に多用しない。
- **実コンテンツの形状を保つスケルトン**にする。汎用カード/`SkeletonRows` を全高レセコン本体・ガントボード・カレンダー・テーブルの前に出してレイアウトシフト（CLS）を起こさない。`loading.tsx` と本体（`variant="bare"` 等）のシェル差・形状差をなくす。ワークベンチ4工程は共通の phase 引数つき `workbench スケルトン` に集約する。
- **false-zero / false-empty を防ぐ**: 初回ロード中に 0 を実値のように描画しない。`isLoading` を `isError` と別にガードし、ロード中はスケルトンを出す。
- **stale 表示**: 「X 分前のデータ + 更新」を明示する。`refetchInterval`（60s ポーリング等）はタブ非表示時に停止する。
- 全画面 `<Loading/>` 置換でヘッダ/現在地（trunk test）を消さない。ヘッダ・フィルタは残しリスト領域のみスケルトン化する。
- Suspense フォールバックの固定高（`h-[31rem]` 等）を実レイアウトと不一致にしない。スケルトンは実形状で。

## 空状態設計（取得失敗/未接続/権限不足/空 を別表現に）

`loading / error / empty / stale / partial-success` を**別物として表示**する。取得失敗を「データなし」に見せない。

| 状態 | 表現 |
| --- | --- |
| **取得失敗(error)** | 共通 `ErrorState`（再試行導線つき）。素の `<p className="text-destructive">` 1行で済ませない。`role="alert"` |
| **未接続** | 「サンプル表示（実データ未接続）」「準備中」を明示。STT 未接続等は専用文言で空と区別 |
| **権限不足** | `ErrorState`(forbidden)。`readonly` 系で「閲覧のみ/権限なし」を理由提示 |
| **空(empty)** | `EmptyState`。「まだありません」+ 次のアクション導線。空と error を取り違えない |
| **部分成功(partial)** | 成功/失敗件数を明示。toast だけに頼らず行内ステータスでも示す |

- 全クエリのエラーを共通 `ErrorState` に統一する（uat / metrics 系の素 `<p>` 退行を排除）。`isError` 分岐の欠落（false-empty: 取得失敗が空カレンダー/空一覧に潰れる）を作らない。
- ページ全体を1枚の QueryError でゲートせず、クエリ別 skeleton / ErrorState で partial 表示を可能にする。
- 失敗カテゴリは「取得失敗」マークを付け、0 件（成功して空）と区別する。

## エラー状態設計

- **エラーメッセージは安全に**: 認証失敗は「メールまたはパスワードが正しくありません」に統一し、どのフィールドが誤りかを漏らさない（列挙攻撃を助長しない）。`role="alert"` で支援技術へ通知。
- **行き止まりを作らない**: lockout の連絡先・解除時間はプレースホルダ固定値（`03-XXXX-XXXX` / `example-pharmacy.jp` / `30分`）を出さない。組織情報/環境設定から供給し、未設定時は「管理者にお問い合わせください」のフォールバックにする（虚偽連絡先で業務停止した利用者を到達不能にしない）。
- 動的エラーは `aria-live` / `role="alert"` で通知し、長大フォームではエラー要素へスクロール+フォーカスする。
- API 由来の長文エラーは Alert 内で折返す前提を満たす。
- 不可逆操作の失敗後は再試行/復帰導線（Undo Toast「元に戻す」等）を提示する。

## アニメーション設計

- **速く・控えめ・非妨害**（`less delight, more usability`）。装飾アニメーションは禁止。
- micro 100–200ms / transition 150–300ms、easing は `ease-out` 基本。
- ローディングはスケルトンの `pulse`。スピナー単独不可（再掲）。
- `prefers-reduced-motion` を尊重する。
- 状態変化（選択・展開）は色の点滅でなく形・位置・アイコンで伝える。

## レスポンシブ設計（mobile390/tablet768/desktop1440/wide1920）

- **高さは `100dvh` ベース**の `min-height`/`height`。固定 `px` / `100vh` / `min-h-screen` を使わない（モバイルブラウザ chrome 増減への追従）。複数の height ユーティリティ（`h-[calc(100dvh-9rem)]` と `h-[calc(100dvh-64px)]` の cn 衝突）を二重指定しない。
- **縦長フォームに内部スクロール到達性を保証**: 認証シェル等は外側 `overflow-y-auto` を持たせ、低ビューポート高（モバイル横向き）でフォーム上部が到達不能にならないようにする。`items-center` の縦中央寄せだけに頼らない。

| ブレークポイント | 方針 |
| --- | --- |
| **mobile 390** | 1カラム縦積み（source order 維持）。主操作は下部 Thumb zone 固定。多列テーブルは重要列に絞った縦カードへフォールバック。OTP は `grid-cols-6 w-full min-w-0`（固定 `w-12` の詰まり回避）。KPI は2列+折返し（3列固定で数値を潰さない） |
| **tablet 768** | 2カラム（主+根拠）を維持するか補助をドロワー化。3カラムを lg 専用にして tablet で補助カラムを本文下へ長く沈めない。中間幅グリッド（md/lg）を定義し、xl 専用多カラム（最大8列）で md〜xl を1カラム超長尺にしない |
| **desktop 1440** | 業務ワークスペースは全幅・全高。固定 rem 列合計で窮屈にしない |
| **wide 1920** | フルブリードを避ける（PageScaffold 経由で幅制御）。読み物/印刷/フォーム/設定・選択系のみ `max-width` |

- 多列 DataTable はモバイル横スクロール固定にせず縦カードフォールバックを用意する。月間グリッド（最大31列×人数）はモバイルで週送り/縦リストへフォールバックし、横スクロール時は日付ヘッダ/左列を sticky にする。
- セル内に重量フォーム（6入力）を埋め込まない（横スクロール内操作の回避）。ドロワーへ退避する。
- モバイルで本文・状態バッジ・リンクを同一行に押し込まない（`ml-auto` ピルの折返し崩れを避ける）。

## アクセシビリティ基準（WCAG2.2 AA, 44px, フォーカス可視, 入力保持）

- **WCAG 2.2 AA 必須**。コントラスト比 4.5:1 以上（屋外・直射日光想定で余裕を持たせる）。
- **タッチターゲット 44px 以上**を部品の既定で保証（ルート override ハックでなく）。
- **フォーカス可視**: キーボードフォーカスリングを残す。ドロワー/メニュー/モーダルはフォーカストラップ・Escape・初期フォーカスを備える。
- **入力保持**: エラー時に入力済みデータを消さない。離脱防止・自動保存（再掲）。
- **色だけに依存しない**: 状態・達成・曜日・チャートノードはラベル/アイコン/線/位置を併用する。要件達成チェックリストに Check アイコン＋達成テキスト、ルートチャートに凡例（赤=緊急/紫=確定固定/青=通常）を併記。
- **見出し要素で構造化**: グループ見出し・リスト項目（患者名・時刻）を `<p>` でなく見出し/リスト要素にし、支援技術で走査可能にする。`<main>`・正しい見出し順序を保つ。
- ドラッグ並べ替えにはキーボード/ボタンの代替順序変更を併設する。
- 棒/チャートに数値テーブル等の代替テキスト構造を併設する。

## 禁止事項

- 色を増やすこと。生 Tailwind 状態色（`bg-emerald-100` / `text-blue-500` / `bg-rose-50` 等）の直書き。
- 状態色の全面塗り（`bg-state-*/10` をタイル/カード/行/入力欄の面に敷く）。データテーブル行の状態色ベタ塗り。
- raw `Badge variant`（destructive/secondary/outline）・ローカル `statusVariant`・`CYCLE_STATUS_CONFIG` 直渡しでの状態表現。`destructive`（赤）を止まっていない状態（終了/期限切れ/休業/回答待ち/低負荷）へ流用すること。
- アラート4段階のフラット化（緊急中断と要確認/状態/期限リマインドを同じ見た目に）。偽アラート（0件で赤点灯・達成時も常時橙）。`role="alert"`(assertive) を常設状態に使うこと。
- 偽の安全保証・偽データ（ハードコード ✓、未接続の全 0 を実測表示、永続化されない「採用/解消」、window 集計の過小件数、暗黙の一覧切り捨て）。
- PHI の生値露出（一覧/生 JSON/外部共有/受付前の不要展開）。
- 破壊的・広域・準不可逆操作の確認なし即実行（月次締め/全件取込/薬価更新/貸出取消/外部共有発行/車両一括反映/疑義変更確定）。破壊的操作を Thumb zone・密な行内に置くこと。
- 安全情報（麻薬/ハイリスク/冷所/抗凝固/アレルギー/腎機能/年齢/患者識別）を Pinned/固定サマリーから欠落させる、`+N` の裏に隠す、モバイルで列ごと隠す（調剤方法=一包化/粉砕 等）こと。
- スピナー/テキスト単独ローディング、実形状と不一致のスケルトン、false-zero/false-empty。
- 12px 未満のラベル・本文（`text-[9px]`/`[10px]`/`[11px]`）。数値への `tabular-nums` 欠落。見出しレベルのスキップ。即席タイポサイズ。
- 角丸の過大（`rounded-2xl`/`rounded-xl` 常用）。状態意味のない影・装飾カード・重複表示。1件ずつの全カード化。
- 固定 `px`/`100vh`/`min-h-screen` 高さ、データ非連動の巨大 `min-h-[...]`、8pt グリッド外のマジック余白（`mt-14`/`xl:pt-20`）。
- 生 `<select>`/`<input>`/自作チェックボックス/`NativeSelect` の新規実装。`SelectValue` の明示 children 欠落（SSR enum 漏れ）。ルート override（`!h-11`/`[&_input]:!min-h-[44px]`）での 44px 担保。
- DOM 順と視覚順の入替（`order-*`）。可視 h1 の本文直置き。PageScaffold/共通ヘッダの未経由。対象日ナビなしの日付固定。
- 補助大機能が3つ以上ある画面を Tab/ドロワー/別ページに分割せず縦積みで肥大化させること。
- 生 JSON を一次入力面にすること。コード記法・生 enum 値を利用者向け文言に出すこと。
- 無人での自動 `window.print()`（PHI 意図せず出力）。
