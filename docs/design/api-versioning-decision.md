# API バージョニング戦略 決定文書

- 状態: 確定（現状維持 + deprecation ポリシーのみ導入）
- 対象: `src/app/api/**` の Next.js Route Handler 全体
- 関連: [`docs/api-versioning-policy.md`](../api-versioning-policy.md)（既存の簡易方針。本文書はその前提となる調査・比較・推奨理由を確定させ、ポリシーの正本として扱う）
- 補足: 本文書はコード変更を伴わない。実装への反映（レスポンスヘッダ等）は別タスクで行う。

## 1. 現状確認（2026-07-03 実測）

### 1.1 エンドポイント規模

- `find src/app/api -name "route.ts" | wc -l` → **363 件**（旧見積り「220+」から増加。実装は継続的に増え続けている）。
- 全エンドポイントが `/api/` 直下に配置され、URL 上のバージョン segment は存在しない（`/api/v1/...` は未実装）。
- `src/lib/api/route-catalog.ts` は **exhaustive ではない**運用カタログ（`routeCatalogMetadata.exhaustive: false` と明記）。全 route の唯一の正本は `src/app/api/**/route.ts` のディレクトリ構造そのもの。
- クライアントは実質 1 種類（自社 Next.js PWA フロントエンド）。サードパーティ SDK・公開 API コンシューマは現時点で存在しない。

### 1.2 外部連携面の棚卸し

社内 CRUD API とは別に、外部システム・外部関係者と直接やり取りする面が複数存在する。バージョニング判断はこれらの「契約の安定性要求」を軸に行う必要がある。

| 連携面                                    | 実装                                                                                                                                 | 契約の性質                                                              | 呼び出し元                                                                                                                       |
| ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| **アウトバウンド Webhook**                | `src/app/api/admin/webhooks/route.ts`、`src/server/services/outbound-webhook.ts`                                                     | イベント駆動（`prescription.created` 等5種、HMAC署名、fire-and-forget） | 外部システム（登録された任意のURL）。PH-OS が送信側 = ペイロード形状を自由に管理できるが、コンシューマ側の受信ロジックは静的想定 |
| **外部共有リンク（患者向け）**            | `src/app/api/external-access/**`                                                                                                     | トークン+OTP+有効期限つき一時公開URL（`permission: 'public'`）          | 患者本人・家族（ブラウザ直アクセス、SDK無し）                                                                                    |
| **外部専門職連携**                        | `src/app/api/external-professionals/**`                                                                                              | 多職種連携（医師・訪問看護等の外部関係者アクセス）                      | 薬局外の医療専門職                                                                                                               |
| **MCS（メディカルケアステーション）同期** | `src/app/api/patients/[id]/mcs-sync/route.ts`、`src/server/services/patient-mcs.ts`                                                  | 外部タイムライン同期（conflict検出あり）                                | MCS 側 URL を起点に PH-OS が pull する構成                                                                                       |
| **レセコン請求データ連携**                | `src/server/adapters/claims-export/`（現在は CSV/CLAIMS-XML 変換のスタブ実装）、`billing-candidates/close` から fire-and-forget 送信 | 月次締め後の CLAIMS-XML 送信。レセコン側フォーマットに厳密準拠が必要    | レセプトコンピュータ（未接続、将来フェーズでアダプタ差し替え予定）                                                               |
| **ジョブ/スケジューラ実行**               | `src/app/api/jobs/[jobType]/route.ts`（`permission: 'canAdmin\|apiKey'`）                                                            | API key 認証によるバッチトリガー                                        | 内部スケジューラ（EventBridge 相当）のみ。外部第三者ではない                                                                     |

**確認できた事実**: 現時点で「PH-OS の `/api/*` を第三者が SDK/固定契約として直接消費している」ケースは無い。外部連携はいずれも (a) PH-OS が送信元となる webhook、(b) 短命トークンでスコープされた読み取り専用の公開リンク、(c) 未接続のスタブアダプタ、のいずれかであり、「広範な外部クライアントが `/api/patients` のような内部エンドポイント群に直接依存している」状態ではない。

## 2. 比較: URL prefix vs ヘッダー方式 vs 現状維持+deprecationポリシーのみ

| 観点                                | ① URL prefix（`/api/v1/...`）                                                                                                                                                                                                                             | ② ヘッダー方式（`Accept: application/vnd.ph-os.v1+json`）                                                                                                               | ③ 現状維持 + deprecation ポリシーのみ                                                                                                                                                  |
| ----------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 実装コスト（363 route 全改修）      | 高。全 route を `v1` ディレクトリへ move、内部 import・テスト・route-catalog・rate-limit 設定を全て追随させる必要がある                                                                                                                                   | 中〜高。ミドルウェアでヘッダー解析→ルーティング分岐を全 route に挟む必要があり、Next.js App Router のファイルベースルーティングと相性が悪い                             | ゼロ。既存構造のまま                                                                                                                                                                   |
| クライアント実装コスト              | 低（URLだけ見れば良い）                                                                                                                                                                                                                                   | 高（クライアントが Accept ヘッダーを正しく送る必要。ブラウザの直リンク・キャッシュ・CDN との相性が悪い）                                                                | 変化なし                                                                                                                                                                               |
| CDN/ロードバランサ/キャッシュ親和性 | 高（パスベースでルーティング・キャッシュ分離できる）                                                                                                                                                                                                      | 低（同一URLで内容が変わるため `Vary` ヘッダー運用が必須になり、Amplify Hosting 側のキャッシュ設定が複雑化）                                                             | 現状のまま（バージョン概念が無いので考慮不要）                                                                                                                                         |
| 医療システムの外部連携安定性        | 将来レセコン・MCS等の外部コンシューマが増えた場合、破壊的変更を **新パスへの追加** として明示でき、旧パスを凍結できる。監査証跡上も「どのバージョンのAPIが叩かれたか」がログのURLだけで分かり、3省2ガイドライン下の変更管理・トレーサビリティと相性が良い | ヘッダー欠落時のデフォルト動作の曖昧さが医療データの取り違えリスクになりうる（特に外部共有リンクや webhook のような「クライアント実装を完全に統制できない」経路で顕著） | 現在の外部連携（webhook/external-access/レセコン）はいずれも PH-OS 側が仕様を握っているか、コンシューマ数が限定的（0〜数件）。破壊的変更の実害を「変更告知＋移行猶予」で吸収できる規模 |
| 今すぐ導入する場合の投資対効果      | 低。363 route を移行するコストに対し、現時点で恩恵を受ける外部クライアントが存在しない（YAGNI）                                                                                                                                                           | 低。同上、かつ実装複雑性が最も高い                                                                                                                                      | 高。ゼロコストで「破壊的変更を無秩序に出さない」という規律だけ導入できる                                                                                                               |
| 将来の移行しやすさ                  | -                                                                                                                                                                                                                                                         | -                                                                                                                                                                       | 現状維持のまま `/api` = v1 相当という前提を明文化しておけば、必要になった時点で ① へ機械的に移行できる（今回の決定が将来の選択肢を潰さない）                                           |

## 3. 推奨案

**③ 現状維持（`/api` 直下 = v1 相当とみなす）+ deprecation ポリシーのみを正式導入する。** URL prefix / ヘッダー方式は今は採用しない。

### 理由

1. **外部コンシューマの実態が薄い**: 棚卸しの結果、`/api/*` を直接消費する第三者クライアントは存在しない。webhook はペイロード形状を PH-OS が握っており、external-access は短命トークンの読み取り専用リンク、レセコン連携は未接続のスタブ。363 route 全体を「複数バージョン運用」する必要性が今は無い。
2. **363 route の移行コストが投資対効果に見合わない**: URL prefix 化は全 route のディレクトリ移動・テスト更新・route-catalog / rate-limit 設定の追随を要し、医療システムとして最優先すべき RLS/CDS/監査ログ関連の改修より優先度が低い（YAGNI）。
3. **ヘッダー方式は医療システムの安定性要求と相性が悪い**: Accept ヘッダー欠落時のフォールバック挙動があいまいになりやすく、患者向け外部共有リンクのように「クライアント実装を統制できない」経路では事故リスクが URL prefix より高い。
4. **将来の選択肢を閉じない**: 「現状の `/api` = v1 相当」という前提を明文化しておくことで、外部コンシューマが実際に増えた時点（レセコン本接続、MCS双方向連携拡大等）で URL prefix 方式へ機械的に移行できる。今回は「意思決定を先送りする」のではなく「今は不要と判断し、トリガー条件を明記する」形で確定させる。

### 将来 URL prefix 方式へ移行するトリガー条件（次回この判断を見直すべき条件）

- レセコン連携が本接続され、レセコン事業者が固定 API 契約を要求する場合
- MCS 以外の外部医療情報システムとの双方向 API 連携が新設される場合
- PH-OS API を第三者（他社製アドオン等）に公開する計画が承認された場合
- 上記いずれかが発生した場合、`/api/v2/` を新設するのではなく、既存 `/api/*` を `/api/v1/*` へ retro-fit し、以後の破壊的変更のみ `/api/v2/` に切り出す（全面 v1 リネームの一括作業は 1 回のみ許容し、以後は本ポリシーの deprecation ライフサイクルに従う）。

## 4. 破壊的変更の deprecation ポリシー草案

### 4.1 バージョンライフサイクル（`docs/api-versioning-policy.md` と共通）

| フェーズ       | 説明                                         |
| -------------- | -------------------------------------------- |
| **Current**    | 最新版。積極的に開発・維持する               |
| **Deprecated** | 後継が存在する。移行猶予期間を設ける（§4.3） |
| **Sunset**     | 廃止済み。`410 Gone` を返す                  |

### 4.2 破壊的変更の定義

以下を「破壊的変更」とみなす:

- レスポンスフィールドの削除・型変更・意味変更
- 必須リクエストフィールドの追加
- エラーコード体系・HTTP ステータスコードの変更
- 認証・認可方式の変更（permission 要件の強化を含む）
- エンドポイントの削除・パス変更
- webhook イベントペイロードのフィールド削除・型変更（`WEBHOOK_EVENT_TYPES` に対応するペイロード契約を含む）
- レセコン CLAIMS-XML / claims-export のスキーマ変更

非破壊的変更（オプションフィールド追加、新エンドポイント追加、バグ修正、パフォーマンス改善）は本ポリシーの対象外とし、通常のリリースサイクルで随時反映する。

### 4.3 移行猶予期間（連携面ごとの差分）

内部クライアント（自社 PWA）のみが消費する内部 CRUD API と、外部関係者が消費する連携面とでは、統制可能性が異なるため猶予期間を分ける。

| 連携面                                                     | 最低移行猶予期間                                                         | 理由                                                                                                                |
| ---------------------------------------------------------- | ------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------- |
| 内部 CRUD API（`/api/patients` 等、社内 PWA 専用）         | デプロイ即時可（フロントと同時デプロイのため）                           | クライアントとサーバーが単一 monorepo で同時にデプロイされ、事実上のバージョン不整合が起きない                      |
| アウトバウンド Webhook（`WEBHOOK_EVENT_TYPES` ペイロード） | **最低6ヶ月**、事前通知必須                                              | コンシューマの実装統制ができない外部システムのため                                                                  |
| 外部共有リンク（external-access）                          | 新規発行分から適用可（既存発行済みトークンは有効期限内は現行契約を維持） | トークンは元々最長 720 時間（30日）で失効するため、長期の後方互換維持は不要。ただし発行中トークンの契約は破壊しない |
| MCS 同期                                                   | **最低6ヶ月**、事前通知必須                                              | 外部医療情報システムとの双方向連携のため                                                                            |
| レセコン claims-export                                     | **最低6ヶ月**、事前通知必須（本接続後）                                  | 医療請求データの誤送信は財務・コンプライアンス影響が大きいため                                                      |

### 4.4 通知方法

- **社内向け**: `CHANGELOG` 相当の変更ログ + `.agent-loop` / gbrain への `ImplementationDecision` 記録。
- **Webhook コンシューマ向け**: Deprecated 移行時は該当イベントタイプに `deprecated_at` メタを含む告知イベント（例: `webhook.deprecation_notice`）を検討、または管理画面（`admin/webhooks`）上に警告表示。
- **レスポンスヘッダー（将来実装、コード変更を伴うため本文書では確定のみ）**:
  ```
  X-API-Version: 1
  X-API-Deprecated: true          # Deprecated フェーズに入った時点で付与
  X-API-Sunset-Date: 2027-04-01   # Sunset 予定日を事前告知
  ```
  実装対象は Deprecated 対象エンドポイントのみに限定し、全 363 route への一律追加は行わない（コスト対効果の観点で対象を絞る）。

### 4.5 例外: セキュリティ修正

医療情報システムの性質上、脆弱性修正・PHI 漏洩防止に関わる変更は本ポリシーの猶予期間の対象外とし、**即時適用**を優先する（`CLAUDE.md` の Compliance by Design 原則に準拠）。この場合も変更内容と影響範囲は事後速やかに記録する。
