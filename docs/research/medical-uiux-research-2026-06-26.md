## 領域 1: 臨床 EHR/EMR の UI ベストプラクティス

### 1.1 認知負荷と情報密度

**原則**: 臨床画面は「判断に必要な情報だけを即時に見せる」こと。

| 課題         | エビデンス                                                                                                    | 採用ルール                                                                                             |
| ------------ | ------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| アラート疲労 | ICU では 66 床で月間 200 万件超のアラート（患者 1 人あたり 1 日 187 件）。VA プライマリケア医は 1 日 100 件超 | 中断型アラートは「重大かつ患者固有の閾値超過」のみ。リマインダーとアラートを同じ見た目で出さない       |
| アラート無視 | 「重大」とされる警告でも上書き率が最大 96%                                                                    | 重大度を 4 段階（緊急中断 / 要確認 / 記録上の状態 / 期限リマインダー）に厳格分離し、視覚的に別物にする |
| 情報過多     | EHR 情報過多が治療遅延・医療ミスに直結                                                                        | 役割別ダッシュボード：薬剤師は処方・薬歴優先、事務は保険・請求優先。ランダムに全データを並べない       |
| 診察時間     | 臨床ポイント・オブ・ケアで確保できる集中時間は 90〜120 秒                                                     | 「最重要判断」を fold より上に置く。詳細は progressive disclosure で 2 階層まで                        |

**採用ルール（数値付き）**:

- 1 画面の主作業は 1 つ。補助情報は `PageSection` または折りたたみで分離
- クリック数削減: 高頻度タスク（薬歴記録・訪問記録・処方確認）は 3 クリック以内
- テンプレート・ショートカット・オートフィルで繰返し入力を排除
- Progressive disclosure は最大 2 階層。3 階層目は別ページへ

**出典**:

- [AHRQ PSNet — Alert Fatigue](https://psnet.ahrq.gov/primer/alert-fatigue)
- [AHRQ — Clinical Decision Support Chapter 1](https://digital.ahrq.gov/ahrq-funded-projects/current-priorities/clinical-decision-support-cds/chapter-1-approaching-clinical-decision)
- [AAFP — EHR Usability Burden Reduction](https://www.aafp.org/practice-operations/administrative-simplification/ehr-usability/techniques-ehr-usability-burden)
- [Phenomenon Studio — EHR System Design](https://phenomenonstudio.com/ehr-system-design/)
- [Thinkitive — EHR Click Reduction](https://www.thinkitive.com/blog/ehr-usability-optimization-for-reducing-clicks-and-burnout/)

### 1.2 エラー防止

**原則**: 破壊的・取消不可の操作は確認ダイアログ必須。入力途中の離脱防止。

- 処方確認・監査確定・報告書送付は「対象データ + 根拠 + 影響 + 確認ボタン」を同一画面で提示
- エラーメッセージは「何が問題か・どう直すか」を具体的に。汎用「エラーが発生しました」禁止
- フォームエラー後の入力保持（WCAG 2.2 3.3.8 に沿う）
- 誤クリック防止: 破壊的ボタンは間隔 8px 以上・外枠で囲む・赤系のみ

## 領域 2: 日本の薬局システム・在宅訪問 UX パターン

### 2.1 業務フローの構造

在宅訪問薬剤師の 1 日の主な業務フローは以下のとおりである。

```
[処方受付] → [調剤準備] → [監査/鑑査] → [セット/配薬準備]
    → [訪問スケジュール確認] → [訪問/服薬指導] → [記録入力]
    → [報告書作成] → [多職種への情報提供] → [レセプト/請求]
```

これが PH-OS の画面順序設計の基本軸となる。

### 2.2 日本の参照製品分析

#### シジダス（PHBDesign株式会社、2024年2月サービス開始）

**定義**: 「新しい時代の薬局 DX 支援システム」。複雑化する施設調剤業務を薬剤師 1 人に依存せず、非薬剤師スタッフとシステムで分担する設計。

| 機能                 | 設計特徴                                | PH-OS 採用可否                       |
| -------------------- | --------------------------------------- | ------------------------------------ |
| 訪問診療時の変更共有 | リアルタイム変更伝達（薬局側 ↔ 施設側） | ✅ 処方変更通知・ケアノートとの連動  |
| 他院・持参薬管理     | 複数医療機関の処方を統合管理            | ✅ 患者詳細の持参薬セクション        |
| 検品・鑑査支援       | 非薬剤師でも容易に実施できる UI         | ✅ 調剤ワークベンチの鑑査サポート    |
| 配薬漏れ防止         | チェックリスト + 薬剤師による遠隔確認   | ✅ セット/配薬確認フロー             |

**UI 設計の示唆**: 薬剤師と事務・外部スタッフの権限境界を画面上で明示し、「誰がどのステップを担当するか」が常に見える設計。

**定義**: 訪問看護ステーション向けクラウド型業務管理。記録・スケジュール・請求をワンプラットフォームで提供。

| 機能                 | 評価スコア（5点満点） | 設計特徴                             |

| 保険請求             | 4.0                   | 実績連動で自動生成                   |

#### Musubi（カケハシ）

**定義**: 「患者に見せながら服薬指導し、その行為が同時に薬歴として記録される」UX コンセプトを核とした電子薬歴。

**設計上の最も重要な学習点**:

1. **記録と指導の二度手間解消**: タブレット画面で指導中にタッチした項目が即座に SOAP 下書きに変換される。これが核心的 UX 革新。
2. **処方差分の自動表示**: 現在処方と過去 2 回分が自動並列表示。同一成分薬が隣に並ぶ。
3. **ハイリスク薬の可視化**: 麻薬・インスリン・抗凝固薬などに色付きアイコンを常時表示（隠さない）
4. **在宅訪問モード**: 複数患者情報のワンタッチ切り替え。訪問先で前後患者をシームレスに移動
5. **ケアメモ → 薬歴 → 報告書の一気通貫**: 訪問中に入力したメモが薬歴と報告書にそのまま流用
6. **スマホ操作感**: PC 操作不慣れな薬剤師でも操作できるタッチ UI。タイピング最小化

| Musubi の設計原則             | PH-OS 採用方針                                                    |
| ----------------------------- | ----------------------------------------------------------------- |
| 指導中タッチ → 薬歴自動下書き | ✅ 訪問記録入力でのタッチ駆動記録（優先実装）                     |
| 処方 3 世代自動並列表示       | ✅ 患者詳細 → 薬歴タブの差分表示                                  |
| ハイリスク薬アイコン常時表示  | ✅ `--tag-hazard` による麻薬/冷所/インスリン/抗凝固タグ（既実装） |
| ワンタッチ患者切り替え        | ✅ 在宅訪問ビューの患者一覧 ↔ 詳細スワイプ                        |
| ケアメモ → 報告書             | ✅ 訪問記録から報告書へのデータ引き継ぎ                           |

**出典**:

- [Musubi 電子薬歴](https://musubi.kakehashi.life/musubi-yakureki)
- [Musubi 瞬間薬歴](https://musubi.kakehashi.life/product/product-shunkanyakureki)
- [カケハシ 在宅訪問サポート追加（2025年4月）](https://xexeq.jp/blogs/media/topics44721)

#### MEDIXS（メドレー）

**定義**: 電子薬歴（2015〜）→ かかりつけ支援 → 在庫管理 → レセコン（2026年4月〜）を統合したクラウド型薬局プラットフォーム。累計 17,000 店舗超・継続率 99%。

| AI服薬指導アシスト    | 指導評価・分析支援                                         |

**設計上の学習点**:

| カテゴリ     | 主画面                             | 最重要 UX 要件                                                           | 典型的な問題                              |

## 領域 3: ミニマリズム — 足し算と引き算の設計哲学

### 3.1 原則

「引き算」（削ること）と「足し算」（追加すること）を並列で設計する。機能削除でなく視覚ノイズ削除。

**削るもの（Subtract）**:

- 状態意味を持たない装飾カード・影・グラデーション
- 同じ情報の重複表示（ページタイトルとパンくずの両立など）
- 全ユーザーに見せる必要のない low-frequency 機能（→ progressive disclosure の 2 階層目へ）
- 空の状態と取得失敗を区別しない汎用表示
- 高頻度タスク以外のアラート・通知
- `bg-blue-100` などの生 Tailwind 状態色のベタ書き（→ トークン化）

**加えるもの（Add）**:

- 余白による視覚的グループ化（`space-y-6` のグループ間、`space-y-4` のセクション間）
- 状態の explicit な明示（loading / error / empty / stale / partial-success を別物として表示）
- 「次に何をするか」が画面最上部 fold 内で完結するアクション配置
- スキャン可能なレイアウト: 見出し → サマリー → 詳細の F 字パターン
- 高頻度操作にキーボードショートカット・タッチショートカットを提供

### 3.2 Progressive Disclosure の適用

Nielsen Norman Group の定義に基づく 2 段階構造:

| 階層                  | 表示するもの                     | 例                                       |
| --------------------- | -------------------------------- | ---------------------------------------- |
| 第 1 層（常時表示）   | 最も使用頻度が高い操作・判断材料 | 処方確認・薬歴記録ボタン・患者アレルギー |
| 第 2 層（展開時表示） | 詳細・補助情報・高度設定         | 過去処方履歴全件・管理設定・ログ詳細     |
| ❌ 第 3 層            | 実装しない（=別ページに分離）    | —                                        |

**出典**: [Nielsen Norman Group — Progressive Disclosure](https://www.nngroup.com/articles/progressive-disclosure/)

## 領域 4: コンシューマ品質の医療ツールへの転用

### 4.1 タイポグラフィ

| 要素                 | 規則                                  | 根拠                                                           |
| -------------------- | ------------------------------------- | -------------------------------------------------------------- |
| 本文フォントスタック | Meiryo → Noto Sans JP → system-ui     | 日本語 CJK での視認性最優先                                    |
| 本文サイズ           | 14px 以上（高密度画面でも）           | 医療現場での読み取りエラー防止                                 |
| ラベルサイズ         | 12px 以上                             | WCAG ラージテキストの閾値（18pt/24px）未満でも十分な可視性確保 |
| 行間                 | line-height 1.6 以上                  | 高密度テーブルでも走査性維持。ヘルスケア.gov ベースライン 1.5  |
| 数値フォント         | tabular-nums 必須                     | 服薬量・点数の桁ズレ防止。用量誤読リスク低減                   |
| 見出しスケール       | h1 → h2 → h3 を論理順守。スキップ禁止 | 支援技術のナビゲーション保証                                   |

**フォント選定根拠**:

- **Inter / IBM Plex Sans（英数）**: スクリーン描画専用設計。tabular figures 対応
- **Noto Sans JP（日本語）**: 150 以上の書記体系対応。1,000 言語。Google 公開ライセンス
- 医療データテーブルでは tabular figures を `font-variant-numeric: tabular-nums` で明示的に有効化

### 4.2 スペーシング・リズム

```
グループ間:        space-y-6 以上（24px）
セクション間:      space-y-4〜6（16〜24px）
見出し〜本文:      最低 8px
テーブル行高:      高密度でも 40px 以上（44px 推奨）
```

### 4.3 モーション・マイクロインタラクション

**原則（Better.care Design System より）**: 「less delight, more usability」。装飾的アニメーション禁止。

| 用途                       | 使用可                        | 使用禁止                       |
| -------------------------- | ----------------------------- | ------------------------------ |
| データ読み込み（2 秒以上） | スケルトンローダー            | スピナーのみ                   |
| フォーム送信確認           | 成功時の緑フラッシュ（200ms） | 派手なお祝いアニメーション     |
| ページ遷移                 | フェードイン（150ms）         | スライドアニメーション複数重ね |
| アラート表示               | スケール 0.95→1.0（100ms）    | バウンス・シェイク             |
| 操作フィードバック         | ボタン押下時の opacity 変化   | 長い入場アニメーション         |

- easing: `ease-out` を基本（ユーザー操作への即応感）
- 持続時間: micro は 100〜200ms、transition は 150〜300ms

### 4.4 空/読み込み/エラー状態の設計

| 状態                      | 表示内容                              | 禁止パターン                 |
| ------------------------- | ------------------------------------- | ---------------------------- |
| loading                   | スケルトン UI（レイアウト形状を保つ） | 空白画面、汎用スピナーのみ   |
| error                     | 何がエラーか + 再試行ボタン + 連絡先  | 「エラーが発生しました」のみ |
| empty（データなし）       | 理由 + 次のアクション                 | loading と同じ見た目         |
| stale（最終取得時刻表示） | 「X分前のデータ」+ 更新ボタン         | 古いデータを最新として表示   |
| partial success           | 成功/失敗の件数をそれぞれ表示         | 全件成功として扱う           |

### 4.5 モバイル・フィールド使用最適化

在宅訪問薬剤師の利用環境:

- 屋外（直射日光）・暗い居室・手袋装着・片手持ち・移動中
- 端末: iPad・Android タブレット・スマートフォン

| 要件                 | 規則                                                           |
| -------------------- | -------------------------------------------------------------- |
| タッチターゲット     | 44px 以上（Apple HIG 準拠。WCAG 2.2 AA の 24px を大幅超過）    |
| 主要アクションの配置 | 画面下部サム・ゾーン（Y 位置 60〜100% の範囲）に集中           |
| 屋外コントラスト     | 通常の 4.5:1 をさらに上回る 7:1 を推奨（高輝度環境補正）       |
| 光/暗の両対応        | ライト/ダークモード両実装。医療者は 1 シフト内に複数環境を移動 |
| 認証頻度             | 生体認証（FaceID/TouchID）で再認証負担を最小化                 |
| オフライン対応       | ネットワーク断でも現在の訪問記録入力が継続できること           |

## 領域 5: アクセシビリティ — WCAG 2.2 AA の具体的規則

### 5.1 コントラスト比

| テキスト種別         | 最低コントラスト比 | 大テキスト定義                              |
| -------------------- | ------------------ | ------------------------------------------- |
| 通常テキスト（AA）   | **4.5:1**          | —                                           |
| 大テキスト（AA）     | **3:1**            | 18pt (24px) 以上 / 14pt Bold (18.66px) 以上 |
| UIコンポーネント境界 | **3:1**            | アイコン・フォームボーダー                  |

PH-OS のプライマリ色（`oklch(0.38 0.09 252)` ≒ `#1f4e79` 相当）は白背景上で **19:1** を達成しており WCAG AAA 水準を超過している。

### 5.2 WCAG 2.2 新追加要件（2023年10月）

| 成功基準                             | 内容                                                      | PH-OS 適用                                                          |
| ------------------------------------ | --------------------------------------------------------- | ------------------------------------------------------------------- |
| 2.4.11 Focus Not Obscured (AA)       | フォーカスがスティッキーヘッダー等に隠れてはならない      | 上部固定ヘッダーはフォーカス行にかぶせない                          |
| 2.5.8 Target Size Minimum (AA)       | タッチターゲット最低 **24×24 CSS px**                     | PH-OS は 44px を維持。OK                                            |
| 3.3.8 Accessible Authentication (AA) | パスワード入力にパスワードマネージャー・コピペを許可      | Cognito の入力フォームで paste を塞がない                           |
| 2.4.13 Focus Appearance (AAA)        | フォーカスインジケーターが **2px 以上・3:1 コントラスト** | `focus-visible:ring-2` + 十分なコントラストを全コンポーネントで確認 |

### 5.3 色に頼らない状態表現

```
NG: 赤 = エラー（色のみ）
OK: 赤 + ❌アイコン + 「入力エラー」テキスト

NG: 緑 = 完了（色のみ）
OK: 緑 + ✓アイコン + 「完了」ラベル

NG: 橙 = 要確認（色のみ）
OK: 橙 + ⚠アイコン + 「確認が必要です」テキスト
```

- `StateBadge` / `StatusDot` はアイコン + テキスト + 色 の三重表現が原則
- 8% の男性が赤緑色盲。赤 + 緑の組み合わせを status の対比に使わない

### 5.4 スクリーンリーダー対応

| 要件             | 実装                                                  |
| ---------------- | ----------------------------------------------------- |
| 動的アラート     | `role="alert"` または `aria-live="polite"`            |
| ローディング状態 | `aria-busy="true"` + `aria-label="読み込み中"`        |
| アイコンボタン   | `aria-label` 必須。`title` 属性だけでは不十分         |
| テーブル         | `<th scope>` + `<caption>` で表の構造を明示           |
| モーダル         | `aria-modal` + フォーカストラップ + Escape でクローズ |

## 領域 6: 医療ステータスのカラーシステム

### 6.1 アラート疲労を生まないカラー設計の原則

航空業界モデル（AHRQ 推奨）: コックピットの警告は「乗客・乗員の安全に直結するもの」に限定し、それ以外は警報として出さない。これを EHR に転用すると:

> **赤を見たら必ず行動が必要な状態にだけ赤を使う。**
> 赤の濫用は「また赤か」という無視反応を引き起こし、本物の緊急事態の見落としに直結する。

### 6.2 PH-OS 6軸セマンティックカラーの検証

既存の 6 軸設計（`docs/ui-ux-design-guidelines.md` の `--state-*` / `--tag-*`）は、本研究の医療 UX エビデンスと整合していることを確認。

| 軸             | トークン           | 医療 UX 根拠                                                        |
| -------------- | ------------------ | ------------------------------------------------------------------- |
| info（青）     | `--tag-info`       | 信頼・安定感を表す青。通常ワークフローの「現在地」と「待ち」に最適  |
| blocked（赤）  | `--state-blocked`  | 赤は「止まっている/失敗」に厳限定。緊急アラートと視覚的に重複しない |
| done（緑）     | `--state-done`     | 緑は「完了/正常」。Better.care 設計システムの推奨と一致             |
| confirm（橙）  | `--state-confirm`  | 橙は「要注意/中間状態」。赤緑色盲でも識別可能                       |
| hazard（橙）   | `--tag-hazard`     | 麻薬・冷所・インスリン等の危険タグ。Musubi の実装と一致（隠さない） |
| waiting（紫）  | `--state-waiting`  | 紫は中断理由（他者待ち）に独自割当。状態語彙が明確                  |
| readonly（灰） | `--state-readonly` | 灰＝無効/終了。色覚障害でも明確に区別可能                           |

**追加ガイドライン**:

- **赤+緑の組み合わせ禁止**: 8% の男性が赤緑色盲
- **単色による状態表現禁止**: 色 + アイコン + テキストの三重表現
- **グラフ系列色と状態色の分離**: `--chart-1..5` と `--state-*` は別系統
- **高彩度色の使用面積を最小化**: 重要操作 / 警告 / 現在位置に集中させる

### 6.3 臨床スコアリングの 4 色一貫ルール

血圧・血糖値・バイタルサインなどの数値的リスク層を表示する場合は以下の 4 段階を使用:

| リスク      | 色                       | 使用例                       |
| ----------- | ------------------------ | ---------------------------- |
| 正常        | 緑（`--state-done`）     | 体重 55kg、服薬遵守率 95%    |
| 要注意      | 橙（`--state-confirm`）  | 血圧 140/90、体重増加 3kg    |
| 危険        | 赤（`--state-blocked`）  | 血圧 180/110、アレルギー反応 |
| 不明/未計測 | 灰（`--state-readonly`） | 未記録、欠損値               |

## 高頻度画面モデル: 在宅訪問薬局の優先スクリーン

### 優先順位ランキング

1 日に薬剤師・事務が最も多く触れる画面を頻度・リスク・業務インパクトで評価。

| 順位 | 画面                           | 触れる頻度     | #1 UX 勝利条件                                                                         |
| ---- | ------------------------------ | -------------- | -------------------------------------------------------------------------------------- |
| 1    | **今日の訪問リスト（My Day）** | 毎日/複数回    | 全患者の当日ステータスが fold 内に収まり「未対応 / 対応中 / 完了」が 3 秒でわかる      |
| 2    | **患者詳細**                   | 訪問前後に毎回 | アレルギー・ハイリスクタグ・最新処方・次回訪問日が最上部固定。スクロールせず判断できる |
| 3    | **訪問記録入力（キャプチャ）** | 訪問 1 件ごと  | タッチ主体で薬歴 SOAP 下書きを自動生成。タイピング最小化。写真・音声入力対応           |
| 4    | **調剤ワークベンチ**           | 調剤1件ごと    | 4工程（処方 → 調剤 → 鑑査 → セット）が 1 画面内で完結。作業者とチェッカーが明確        |
| 5    | **訪問スケジュール**           | 毎日           | 日/週ビュー切替。職員別・患者別フィルタ。記録済み/未記録が色と記号で即識別             |
| 6    | **処方受付**                   | 処方到着ごと   | 電子処方箋 QR → 患者紐付け → 調剤キュー投入が 3 アクション以内                         |
| 7    | **報告書作成**                 | 訪問後に毎回   | 訪問記録からの自動引き継ぎ。PDF 出力→送付が同一画面で完結                              |
| 8    | **在庫・配薬確認**             | 調剤後・配送前 | QR/バーコードで配薬対象薬剤を確認。配薬漏れを視覚的に検出                              |

#### 1. 今日の訪問リスト（My Day）

```
[今日 2026-06-26 木] [← 前日] [翌日 →]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 採用推奨ルール: エグゼクティブサマリー（10 項目）

1. **アラート 4 段階厳格分離**: 中断型は「重大緊急」のみ。リマインダー・情報通知は非中断型で別 UI。アラート疲労（重大警告の無視率 96%）を防ぐ。

2. **高頻度タスク 3 クリック以内**: 今日の訪問リスト → 訪問記録入力、処方確認 → 薬歴保存など主要フローは 3 アクション以内で完結させる。

3. **タッチ駆動記録（Musubi パターン）**: 服薬指導中のタッチ → SOAP 下書き自動生成。訪問現場でのタイピングをゼロに近づける。

4. **ハイリスク情報の常時表示**: アレルギー・麻薬・冷所・インスリン・抗凝固薬フラグは患者詳細の最上部固定。折りたたみ禁止。

5. **色 + アイコン + テキストの三重表現**: 色だけで状態を伝えない。8% 男性が赤緑色盲。`StateBadge` / `StatusDot` を全状態表示に使う。

6. **WCAG 2.2 AA の数値遵守**: 通常テキスト 4.5:1、タッチターゲット 44px（WCAG 最低 24px を超えた Apple HIG 値）、フォーカスインジケーター 2px・3:1 コントラスト。

7. **empty / loading / error / stale の 4 状態分離**: 取得失敗を空状態と誤認させない。各状態に固有の UI と文言を持つ。

8. **訪問スケジュール画面が競合他社の弱点**: カイポケのスケジュール評価は 3.0/5.0（最低スコア）。ここを超えることが即座の差別化要因になる。

9. **報告書の自動引き継ぎ**: 訪問記録からの報告書生成は「ゼロ転記」を目標に。入力データが報告書に流れ込み、医師・ケアマネへの送付まで同一フローで完結。

10. **オフライン継続性**: ネットワーク障害時も訪問記録入力・処方確認が継続できること（MEDIXS のオフライン機能が競合水準）。PWA + Dexie の現状構成は正しい投資。
