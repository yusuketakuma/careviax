# Compliance Applicability — 適用ガイドラインと Traceability Matrix（Phase 1 統合）

作成日: 2026-07-11 / 入力: `docs/ui-ux-refresh/phase1/g1〜g6` 調査ノート + `docs/ui-ux-refresh/01-scope-and-users.md`

本書は Phase 1 調査（g1〜g6）を UI/UX リフレッシュの観点で統合し、適用ガイドラインの台帳と要求単位の追跡表（Compliance Traceability Matrix）を提供する。**本書は準拠の断定ではない**（§4 の注記を必ず参照）。

適用性分類の凡例: Mandatory / Applicable / Conditionally applicable / Not applicable、および横断ラベル Requires legal review / Requires clinical safety review / Requires security review（複合可）。

## 2. Compliance Traceability Matrix

- UI/UX に影響する要求を中心に、1ガイドライン（群）あたり主要要求 5〜15 行を抽出。網羅ではない（全要求の棚卸しは各 phase1 ノートおよび `docs/compliance/three-ministry-guideline-mapping.md` 側の責務）。
- **SSOT rule** 列は既存 `docs/ui-ux-design-guidelines.md` の§参照。未定義の場合は「SSOT未定義→Phase6候補」。
- **Implementation file / Test case** は現段階で repo 内根拠が既知のもののみ記載。未調査は「Phase 5 で対応付け」。Test evidence は Phase 5 監査で採取するため現段階は原則「未採取」。
- **Owner** はレビュー主担当の役割（法務／薬事／臨床安全／セキュリティ／デザイン／FE／BE）。**Review status** は本書時点の状態（未レビュー＝専門家未確認）。

## 3. 専門家レビュー待ち項目一覧

以下はいずれも**専門家（法務・薬事・臨床安全・セキュリティ）のレビュー待ち**であり、本書執筆時点で AI 調査ベースの整理に留まる。確定判断・適合宣言には使用しないこと。

### 3.1 Requires legal review（法務レビュー待ち）

1. 令和7年法律第37号（処方箋・調剤録の保存3年→5年）の**施行期日政令と経過措置**の確認、保持ポリシー設定値への反映（G5 §3/§4/§11）
2. 公費制度別の薬歴5年保存要求の**一次資料**確認（各公費実施要領）（G5 §5）
3. APPI 令和8年改正法の公布日・法律番号・政省令（課徴金・子ども保護・統計/AI 特例の適用条件）（G6 §2）
4. 医療・介護ガイダンス 令和8年4月一部改正の新旧対照表の逐条確認と、多職種共有（outbound/inbound/onward sharing）の同意根拠整理（G6 §3、01-scope §12-5）
5. 2省GL 第2.0版**確定版本文**の照合（案 redline ベースの現整理との差分）、約款契約のリスク表示条項（民法548条の2）、利用規約・重要事項説明（G2 §1.1/§4）
6. 保守委託機関編を前提とした契約書・SLA への事業者責任明記（G1 §3）
7. 電子署名・タイムスタンプ（HPKI 等）を要する文書の特定（G1 適用性表）
8. 障害者差別解消法の B2B 医療 SaaS への具体的要求水準（G4 §1.5）
9. 死者に係る情報の取扱い（個人情報と同等の安全管理措置）の契約上の位置づけ（G2 §2）
10. 居宅同意取得型オンライン資格確認の同意取得記録要件（G5 §2）
11. org-wide アクセスモデルのリスク受容の明文化（アクセス最小化との整合説明）（01-scope R-4、G1/G2 の権限要求と関連 — 法務＋セキュリティ共同）

### 3.2 Requires clinical safety review（臨床安全レビュー待ち）

1. CDS アラート全般の表示強度・文言・オーバーライド設計（false-negative の害が最大。アレルギー・腎機能を優先）（G3 §4、G6 §5、01-scope §10-3）
2. 患者取り違え防止策の UI＋運用二重防御の妥当性（PatientHeader・交付時確認・QR 突合）（G3、G1 混同防止）
3. 調剤ワークベンチのフォーカス外観・キーボード操作の誤確定防止（WCAG 2.4.13 相当の採否含む）（G4 §2.3）
4. 安全性アラートの role=alert 割り込み設計（読み上げ中断と誤操作誘発のトレードオフ）（G4 §2.4）
5. disabled 表示でも参照が必要な臨床情報の識別（コントラスト除外に頼らない判断）（G4 §2.1）
6. 薬剤検索の文字数・正規化挙動、単位確認、変更・中止可視化、交付前一画面確認の現行実装監査（G3 §4 → Phase 5 use-error risk register）
7. Use-error risk register の整備と、アラートで潰すリスク/運用で受容するリスクの切り分け記録（JAHIS リスクマネジメント枠組み）（G3 §2.2）
8. 管理サービス側チェックと自前 CDS の出所区別表示（G5 §1）

### 3.3 Requires security review（セキュリティレビュー待ち）

1. 二要素認証の実装状態（クライアント/サーバ両面、表14-1 の許容組合せ）と認証 UI 変更（hard-stop 規律対象）（G1 §6、G4 SC 3.3.8）
2. PH-OS 運営主体としての P-mark/ISMS 取得計画（AWS の ISMAP では代替されない）（G2 §2）
3. 監査ログの記録範囲（参照系含む）・改ざん防止・アラート・管理者向け閲覧/出力 UI（G1 §7、チェックリスト 2-⑬）
4. アクセス権限モデル（org-wide）の必要最小限原則との整合、管理者権限最小化、利用者台帳（チェックリスト 2-④、G1 §6.2）
5. 退職者・未使用アカウント無効化の運用と UI（チェックリスト 2-⑤）
6. 接続元制限（テナント別 IP 制限等）を提供する場合の設計（チェックリスト 2-⑭）
7. 外部委託（受託者としての PH-OS）の安全管理措置の 3省2GL 側要求との突合（G6 §3、`docs/compliance/three-ministry-guideline-mapping.md`）
8. 破壊的な認証方式変更・監査ログ設計変更は human 承認必須（careviax 運用ルールの hard stop 領域）

## 4. 注記 — 本文書の性質（準拠の断定ではない）

1. 本書における「準拠」「Mandatory」等の表記は、**要求と設計・実装・テストの間の追跡可能性（traceability）を確立するための分類**であり、PH-OS が当該ガイドライン・法令に**準拠していることの断定・宣言・保証ではない**。
2. 各行の Applicability / rationale は AI による一次資料調査（phase1 g1〜g6、確認日 2026-07-11）に基づく整理であり、法令解釈・薬事該当性・臨床安全性の確定判断は §3 の各専門家レビューを経るまで**未確定**として扱うこと。適合宣言（例: 3省2GL 真正性・保存性への適合）は、確定後訂正ワークフロー・起算点別 retention 等の実装完了と専門家レビューの両方を待つ（01-scope §7 Review required と同旨）。
3. 「確認できた版」は 2026-07-11 時点の公式サイト確認結果であり、それより新しい版が存在しないことの保証ではない。特に施行日未確定の改正（薬剤師法保存年限、APPI 令和8年改正）と未照合文書（2省GL 第2.0版確定版、保守委託機関編）は再確認が必要。
4. Test evidence 列が「未採取」の行は検証未実施を意味する。Phase 5 監査で証跡（テスト結果・スクリーンショット・監査記録）を採取するまで、当該要求の充足を主張しないこと。
