## 2. 観察パターン

### 2.1 段階的「監査ゲート」で受付から会計まで進む直列フロー

- **Product/source**: Recepty NEXT / 公式操作マニュアル（2次元シンボル読み取りオプション、平成31年3月版）
- **Screen or workflow**: 処方箋取込〜会計の全体フロー
- **Observed pattern**: 患者検索画面でスキャン →（新患のみ）患者登録 → 「患者・保険監査」画面 → 「医療機関・科目・医師監査」画面 → 「処方監査」画面 → 処方画面（指導料・コメント入力）→ 会計、という直列のチェックポイント列。各段で内容を確認し、ファンクションキー（F8確定/F10確認）で明示的に承認しないと次へ進めない。
- **Problem solved**: 外部由来データ（処方箋2次元シンボル）を無検証のままレセプト・薬歴に流し込む事故の防止。確認責任の所在を画面単位で明確化。
- **Strength: 患者→保険→処方元→処方内容の順に「誰の・どの保険で・誰が出した・何の処方か」を分解して検証させるため、確認漏れの構造的防止になる。
- **Limitation**: 全件で多段確認が挟まるため、正常系でも操作段数が多い。忙しい時間帯には形骸化（無思考のF8連打）リスク。
- **Possible misinterpretation**: 「監査」はここでは調剤鑑査（薬の照合）ではなく、取込データの事務的検証を指す。調剤鑑査UIと混同しない。
- **Patient safety implication**: 患者取り違え・保険誤り・処方内容誤取込という上流事故を会計前に多段で遮断する。
- **Applicable principle for this system**: 外部データ取込（電子処方箋・QR・FAX OCR）は「取込→ステージング表示→明示的確定」の段階ゲートにし、各ゲートの確認対象を1種類に絞る。PH-OS の受付〜調剤ワークフロー状態遷移と親和的。
- **Adopt/Reject**: **Adopt（構造のみ）**。ただし正常系は差分ゼロなら一括確認を許すなど、確認負荷の逓減策を併設して形骸化を防ぐ。
- **Evidence**: https://emsystems.co.jp/em-denshisyohosen-stkit/file/NEXT2次元シンボル読み取りオプションマニュアル190320.pdf（p.2）
- **Access date**: 2026-07-11

### 2.2 患者照合の「一致条件の明示」による同姓同名対策

- **Product/source**: Recepty NEXT / 公式操作マニュアル（同上）
- **Screen or workflow**: 患者検索・患者選択画面
- **Observed pattern**: 2次元シンボルから読み込んだ「生年月日・性別・漢字氏名」の3項目のいずれかが一致する患者を抽出。複数一致した場合は患者選択画面を表示し、候補ごとに**どの条件で一致したか**（例: 生年月日・性別が一致）を画面下部に明示。候補が正しくなければ F1患者追加（新規登録）や F7全患者検索へ逃げられる。
- **Problem solved**: 同姓同名・氏名表記ゆれによる患者取り違えと、既存患者の重複登録の両方。
- **Strength**: 「なぜこの候補が出たのか」の根拠を見せることで、操作者が氏名だけを見て安易に選択する行動を抑止する。
- **Limitation**: 一致根拠は文字列表示で、生年月日など不一致項目側の強調表示までは確認できない（公開資料の範囲では）。
- **Possible misinterpretation**: 「3項目いずれか一致で抽出」は検索網を広げるための OR 条件であり、本人確認の十分条件ではない。抽出ロジックをそのまま同定ロジックと誤読しない。
- **Patient safety implication**: 患者誤選択は調剤過誤の最上流。曖昧一致を隠さず提示し人間に裁定させる設計は false-negative（別人を同一人物と誤認）を減らす。
- **Applicable principle for this system**: 患者検索結果に「一致した属性/一致しなかった属性」をバッジ表示し、部分一致選択時は生年月日・性別の確認ステップを挟む。PH-OS の患者検索 API に一致根拠メタデータを持たせる。
- **Adopt/Reject**: **Adopt**。同姓同名識別の公開実例として最も具体的。PH-OS では不一致属性の赤系強調まで一歩進める。
- **Evidence**: 同上マニュアル（p.3）
- **Access date**: 2026-07-11

### 2.3 「取込データ vs 登録データ」の並置照合と一致/要確認の色分け

- **Product/source**: Recepty NEXT / 公式操作マニュアル（同上）
- **Screen or workflow**: 患者・保険監査画面、医療機関・科目・医師監査画面
- **Observed pattern**: 画面上段に「取り込みデータ」、下段に「検索結果（システム登録済みデータ）」を並置。両者が同一なら「一致」を青字、異なれば「確認」を赤字で表示し、赤字時は保険追加（F2）/保険切替（F4）などの解決操作を同一画面から実行させる。医療機関・医師はコード一致で自動選択、不一致時はマスタ一覧から手動選択。
- **Problem solved**: 外部データと院内マスタの突合を目視だけに頼らず、システム判定（一致/要確認)＋人間の裁定に分業。
- **Strength**: 判定結果（青/赤）と解決手段（追加/切替）が同じ画面にあり、エラーからの復帰動線が短い。
- **Limitation**: 青字「一致」への過信リスク（2次元シンボル側の出力品質は医療機関により異なる、とマニュアル自身が注記）。色のみでなく文字ラベル併記だが、赤/青の2値で中間状態（部分一致）の表現は確認できない。
- **Possible misinterpretation**: 「一致」表示は形式一致であって資格の有効性確認ではない（福祉制度等は手動修正が必要とマニュアルに明記）。
- **Patient safety implication**: 保険・処方元の誤登録は直接の健康被害より請求・記録の信頼性problem だが、誤患者文脈での薬歴汚染を防ぐ意味で安全に寄与。
- **Applicable principle for this system**: 外部取込の照合画面は「source vs system」の2カラム並置＋一致状態のラベル（色+テキスト）＋その場で解決できるアクションを原則とする。PH-OS の警告3段階（重大=赤/注意=橙/情報=青）にマップ可能。
- **Adopt/Reject**: **Adopt**。ただし PH-OS では WCAG AA に従い色+アイコン+テキストの三重符号化にする。
- **Evidence**: 同上マニュアル（p.4-5）
- **Access date**: 2026-07-11

### 2.4 処方監査画面: 原本データと「仮入力」の左右並置、確定までは下書き扱い

- **Product/source**: Recepty NEXT / 公式操作マニュアル（同上）
- **Screen or workflow**: 処方監査画面 → 処方入力画面
- **Observed pattern**: 左に「読み取りデータ表示」（処方箋原本由来）、右に「ReceptyNEXT での仮入力表示」（システムが解釈した結果）を並置。操作者は仮入力欄を確認して問題なければ F8確定。確定するまでは仮入力＝ドラフト状態であり、レセプト・薬歴側の正式データにならない。行単位の状態は背景色でエンコード: **赤帯=薬品/服用法がマスタ未関連付け（そのままでは処方入力へ取込不可）**、**黄色帯=補足レコードあり（コメントとして取込選択可）**、**グレー=処方対象外として明示的に除外した行**。赤帯はその行から F1 で変換マスタ編集に入り解決するまで先へ進めない（fail-closed）。
- **Problem solved**: OCR/コード変換の解釈誤りを「原本と解釈の突合」で人間が検証できるようにし、未解決データの黙殺（silent drop）を防ぐ。
- **Strength**: (1) 原本と解釈の並置は照合作業の認知負荷を最小化する。(2) 「未解決=赤=進行ブロック」「除外=グレー=明示的意思決定の痕跡」という状態の可視化が一貫している。(3) 除外時も確認ダイアログ（マスタ登録するか否か）を挟む。
- **Limitation**: 色帯の意味は学習が必要。行状態が3色+通常の4状態あり、新人には研修前提。
- **Possible misinterpretation**: 赤帯は「危険な薬」ではなく「マスタ未整備」の意味。臨床アラートの赤と混同すると、臨床上重要な赤の注意力を消耗させる恐れがある（同一画面系で意味の異なる赤が併存し得る点は PH-OS では避けるべき教訓）。
- **Patient safety implication**: 解釈不能データを警告なしに欠落させると処方の一部が薬歴から消える。fail-closed ＋明示的除外の記録は、この false-negative を構造的に防ぐ。
- **Applicable principle for this system**: (1) 取込系は必ず「原本ビュー vs 解釈ビュー」並置。(2) 未解決項目はブロッキング状態とし、除外はユーザーの明示操作＋監査ログとして残す。(3) 「事務的未解決の赤」と「臨床アラートの赤」は視覚語彙を分離する（PH-OS の6軸状態色トークンに整合）。
- **Adopt/Reject**: **Adopt（1,2）/ Adjust（3: 赤の意味過積載は不採用、状態色トークンで分離）**。
- **Evidence**: 同上マニュアル（p.6-7, 13-14）
- **Access date**: 2026-07-11

### 2.5 ハイブリッド同期による BCP: 10分毎同期・サブマシン代運用・遮断時PDF閲覧

- **Product/source**: Recepty NEXT 公式カタログ / MAPs for PHARMACY 製品ページ / 公式プレスリリース
- **Screen or workflow**: システム構成・非常時運用
- **Observed pattern**: Recepty NEXT はローカル機＋データセンターの「リアルタイム同期機能」を搭載し、差分データを常時データセンターに蓄積（10分毎の同期処理と記載）。外付けHDDに差分バックアップと更新ログを日次保存し、トラブル時は**サブマシンによる代運用**で長時間の業務停止を防止。ネットワークは厚労省安全管理ガイドライン準拠の閉域網系サービス経由。クラウド型の MAPs はネットワーク遮断時「薬歴データのPDF化機能により閲覧可能」とし、さらに「サーバへのアクセス不調時に**現場の判断ですぐ切り替え、窓口業務を継続できる**」新BCP機能を開発中（2026年2月予定と記載）。
- **Problem solved**: 災害・停電・回線断・端末故障の各障害モードで「調剤業務を止めない」こと。クラウド型の単一障害点（回線）への手当て。
- **Strength**: 障害モード別に対策が分かれている（端末故障→サブマシン、回線断→ローカル継続 or PDF閲覧、拠点喪失→データセンター復旧）。切替の判断主体を「現場」に置く方針を明言。
- **Limitation**: クラウド型（MAPs）の遮断時対応は現時点の公開情報では「閲覧（PDF）」のみで、オフラインでの新規入力・同期競合解決の仕様は公開情報なし。
- **Possible misinterpretation**: 「リアルタイム同期」と「10分毎同期」が併記されており、RPO=0 ではない可能性がある。公開資料からは正確な RPO/RTO は断定できない。
- **Patient safety implication**: 非常時に薬歴・処方歴を参照できないことは重複投薬・アレルギー見落としに直結。閲覧手段の確保は安全機能そのもの。
- **Applicable principle for this system**: PH-OS の PWA/Dexie オフライン設計に対し、(1) 「閲覧の継続」を最低保証ラインとして明文化（重要患者の薬歴スナップショットのローカル暗号化キャッシュ）、(2) オフライン切替は自動判定＋現場の手動切替の両方を用意、(3) RPO/RTO を仕様として明示する（競合は曖昧なまま）。
- **Adopt/Reject**: **Adopt（発展形）**。PH-OS はオフライン「入力」まで既に射程に入れており、競合の公開仕様（閲覧のみ）より一歩先を仕様化・訴求できる。
- **Evidence**: https://service.emsystems.co.jp/product/pdf/receptynext.pdf / https://service.emsystems.co.jp/maps_series/for_pharmacy/ / https://prtimes.jp/main/html/rd/p/000000027.000081158.html
- **Access date**: 2026-07-11

### 2.6 ファンクションキー中心のキーボード操作体系と Do 入力

- **Product/source**: Recepty NEXT 公式カタログ（確認できた版: 2017年頃）
- **Screen or workflow**: 処方入力画面
- **Observed pattern**: 画面最下段に Esc〜F12 の常設ファンクションキーバー（中止/設定/参照情報/薬歴一覧/DC/処方切替/処方追加/会計/入力済 等の機能割当）。カタログは「多彩な機能が集約されたファンクションキー。Do入力や一般名入力、代替入力など様々な処方入力も最短距離で」と記載。画面遷移後のファンクションキー割当のカスタマイズも可能と記載。処方入力画面は「わかりやすく色分けされた画面表示」「運用に合わせて選べる画面レイアウト」。
- **Problem solved**: 1日数十〜数百枚の処方箋入力での操作距離最小化。マウス往復の排除。前回同一処方（Do処方）の高速入力。
- **Strength**: 物理キーへの固定マッピングは筋記憶化でき、熟練者のスループットが最大化する。画面下部の常設キーバーが「今この画面で押せる操作」のアフォーダンスを兼ねる。
- **Limitation**: F-key 体系は学習コストが高く、ラップトップ/タブレットでは F-key が遠い・存在しない。画面ごとに割当が変わる場合、慣れるまで誤操作リスク。
- **Possible misinterpretation**: 「キーボード最適化＝古い UI」ではない。レセコン市場では熟練事務の入力速度が製品価値の中核であり、意図された設計。
- **Patient safety implication**: 高速入力と安全確認はトレードオフ。確定系キー（F8等）が常に同位置だと、確認画面での無思考押下（confirmation fatigue）を助長し得る。
- **Applicable principle for this system**: PH-OS の調剤ワークベンチ（レセコン風4工程、F12→Enter の autoFocus 設計済み）と方向一致。(1) 主要工程アクションへの固定キー割当と画面下部の常設キーガイド表示、(2) Do/差分入力の最短化、(3) ただし破壊的・確定的操作のキーは通常操作キーと離す or 二重確認を維持。
- **Adopt/Reject**: **Adopt（キーガイド常設・Do入力最短化）/ Adjust（確定キーの位置分離で警告疲労対策を上乗せ）**。

### 2.7 薬歴管理画面: 「参照は左・記録は右」の1画面完結と変更カテゴリ付き処方比較

- **Product/source**: Recepty NEXT 公式カタログ（同上）
- **Screen or workflow**: 薬歴管理画面（服薬指導・薬歴記載）
- **Observed pattern**: カタログ記載「過去処方などの参照情報は左に、記録エリアは右に表示。たくさんの情報もタブで切り替えられるから、1画面で完結」「患者メモや処方チェック、医薬品情報など、業務に便利な情報画面もボタン一つで簡単に表示」。掲載スクリーンショットからは、(a) 過去処方を複数回次（過去①/過去②…）並べて比較する列構成、(b) 「変更前/変更後」の対で薬品追加・用法変更・用量変更等の**変更カテゴリ**を一覧するエリア、(c) アレルギー情報の常設表示枠、(d) 行の色分け（中止・変更等のマーカー）が観察できる。※スクリーンショット観察に基づく記述であり、正確な機能名・語彙は公開文書では確認できない。
- **Problem solved**: 服薬指導中の「前回から何が変わったか」の把握を、2画面往復なしで完了させる。指導しながら記録する二重作業の同時進行。
- **Strength**: 変更点を「差分カテゴリ（追加/用法変更/用量変更…）＋変更前後の値」に構造化して見せる方向性は、生データ2列を目視比較させるより認知負荷が低い。参照/記録の左右分離は視線移動が一定になる。
- **Limitation**: 情報密度が非常に高く（1画面に患者情報・過去処方2回次・今回処方・差分・記録欄）、小型画面や新人には過負荷。増量/減量の区別など差分語彙の細かさは公開資料では確認できない（公開情報なし）。
- **Possible misinterpretation**: スクリーンショットの読解であるため、差分が自動生成か手動記録かは断定できない。「差分自動計算機能がある」と断定して設計根拠にしない。
- **Patient safety implication**: 前回処方との差分の見落としは過量・重複・急な中止の見逃しに直結。差分の構造化表示は服薬指導の質を底上げする。
- **Applicable principle for this system**: PH-OS の処方差分表示は「新規/増量/減量/中止/用法変更」のカテゴリ＋変更前後値のペアを一次情報として構造化保持し（Structured Data First）、指導画面では参照（左）/記録（右）の固定レイアウト＋タブで1画面完結を狙う。アレルギーは全タブ共通の常設枠（PatientHeader 相当）。
- **Adopt/Reject**: **Adopt（差分のカテゴリ構造化・左右分離・アレルギー常設）**。密度は PH-OS ガイドライン（一覧=高密度/入力=ゆとり）に従い、記録側はゆとり寄りに調整。
- **Evidence**: https://service.emsystems.co.jp/product/pdf/receptynext.pdf（薬歴管理画面ポイント1-2 および掲載画面）
- **Access date**: 2026-07-11

### 2.8 タブレット/在宅: 「端末に情報を残さない」ナビゲート型入力

- **Product/source**: Recepty NEXT（ランシステムNEXT iPad アプリ）公式カタログ / MAPs for PHARMACY 製品・機能ページ / スマレキ with Recepty 製品ページ
- **Screen or workflow**: 在宅訪問業務・タブレット運用
- **Observed pattern**: (1) ランシステムNEXT（iPad）: 「どこからでも簡単に接続、**端末に情報が残らず安心**」「項目選択や変更作業だけでより速く正確に入力」「在宅初心者でもナビゲートに従い入力するだけ。報告書の作成や薬歴反映を効率化」「記録も投薬日も訪問日もカレンダーでしっかり管理」。(2) MAPs: Windows PC/タブレット両対応で「薬局内で使用しているタブレットをそのまま訪問先へ持ち出して利用可能」、在宅は「訪問カレンダー」「訪問患者リスト」で管理し「選択式で簡単に入力」。DX は PC版/iPad版選択制、「薬剤師ごとの端末による店舗間応援・在宅訪問活用」。(3) スマレキ: 「iPad一台で、どこでも薬局に」「指一本で簡単操作、音声入力対応」。

- **Limitation**: 「端末に情報が残らない」方式は接続前提であり、通信不能な訪問先（山間部・建物内）での動作は公開情報なし。オフラインファーストではない。
- **Possible misinterpretation**: 「音声入力対応」は入力手段の話であり、音声からの構造化（後述の生成AI薬歴）とは別レイヤー。

- **Adopt/Reject**: **Adopt（選択式入力・カレンダー主導線）/ Adjust（端末非保持は不採用、暗号化オフライン保持で在宅圏外対応を優位性にする）**。

### 2.9 AI 出力は常に「下書き」: 確認・修正してから薬歴に引用

- **Product/source**: ReceptyNEXT 生成AIオプション 公式LP / MAPs for PHARMACY 機能ページ / MAPs DX ページ・薬事日報記事
- **Screen or workflow**: 薬歴作成（SOAP）、患者フォローアップ取込
- **Observed pattern**: (1) 生成AIオプション: 服薬指導の会話を録音→話者分離（説明する側=薬剤師/聞く側=患者と判定）→文字起こし→SOAP形式の指導文を自動作成→薬剤師が「最終確認、修正し完成」「確認・編集した上で、薬歴に引用することができる」。(2) MAPs: AI が処方箋情報・診療科・性別年齢・季節から病名推論を表示、指導文章をサジェストし「サジェスト文章を A/P へ反映」、患者ヒアリング項目は「S/O へ自動反映」で「二重入力が不要」。(3) DX: LINE で患者に確認事項 URL を送り「返信内容をシステムへ取り込み」、薬事日報記事では「電子薬歴の**下書き**として反映」と記載。
- **Problem solved**: 薬歴記載の時間コスト（スマレキ LP では約5分→1分と訴求）と、指導中の記録による対話中断。
- **Strength**: AI・患者回答・ヒアリングのいずれの自動生成物も「薬剤師の確認を通ってから正式な薬歴になる」責任構造が一貫。生成物の投入先が SOAP の該当区画（S/O、A/P）に対応付けられており、構造化を壊さない。
- **Limitation**: サジェストの根拠提示（なぜこの病名推論か）の UI は公開情報なし。推論の誤りが下書き経由で薬歴に固定化するリスクは運用（確認の実質性）に依存。
- **Possible misinterpretation**: 「AI が薬歴を書く」ではなく「AI が下書きを作り薬剤師が確定する」。自動確定と誤読して設計しない。
- **Patient safety implication**: 病名推論・指導文の誤りは指導内容の誤りに直結し得るため、下書き→薬剤師確定のゲートは安全上必須の構造。
- **Applicable principle for this system**: PH-OS で AI/自動取込を導入する場合、(1) 生成物は必ず draft 状態（視覚的にも「下書き」と識別可能）で保存し、確定操作と操作者を監査証跡に残す。(2) 生成物は SOAP 等の構造フィールド単位で挿入し、自由文の塊にしない。(3) 患者自己申告（フォローアップ回答）も同じ draft パイプラインに載せる。
- **Adopt/Reject**: **Adopt**。PH-OS の Audit by Default と組み合わせ、「draft 由来か手入力か」の来歴も記録する点で競合公開仕様より厚くできる。
- **Evidence**: https://mktg.emsystems.co.jp/receptynext_ai / https://service.emsystems.co.jp/maps_series/for_pharmacy/kinou/ / https://www.yakuji.co.jp/entry109117.html

### 2.12 コード横断の薬品同定と「一度教えれば次回から自動」の変換マスタ学習

- **Product/source**: Recepty NEXT / 公式操作マニュアル（平成31年3月版）
- **Screen or workflow**: 処方監査画面・薬品/服用法/コメント変換マスタ
- **Observed pattern**: 薬品マスタは「電算コード・YJコード・JANコード・HOTコードのいずれか」で抽出。一般名処方は医療機関単位または患者単位で調剤薬（後発品等）を関連付けでき、「次回以降、同一医療機関または同患者の変換マスタ登録済みの一般名データを読み込むと、自動的に関連付けた調剤薬が表示」。複数候補紐付け時はドロップダウンで今回の調剤薬を選択。関連付け済み薬は『(代替)+薬品名』形式で明示され、カーソルを合わせると元の一般名情報を表示。服用法・コメントも同様に医療機関単位の変換マスタで学習し、メインメニューから一覧・修正・削除可能。

- **Strength**: (1) 複数コード体系のフォールバック同定は PH-OS の医薬品マスタ設計（SSK/HOT/YJ 横断）と同型の実務解。(2) 変換結果に『(代替)』の来歴マーカーを付け、元の一般名へ遡れる=変換の透明性。(3) 学習範囲を医療機関単位/患者単位で制御。
- **Limitation**: 変換マスタは施設ローカルの手作り資産であり、誤登録が自動再適用される（学習の負の面）。修正 UI はあるが、誤変換検知の仕組みは公開情報なし。

- **Patient safety implication**: 誤った紐付けの再適用は同じ調剤過誤の反復になるため、変換由来であることの明示（来歴マーカー）と定期レビュー導線が安全上重要。
- **Applicable principle for this system**: (1) 一般名→調剤薬の org 単位マッピングを学習・再適用し、適用時は「変換由来」バッジ＋元コードへの遡及表示を必須にする。(2) マッピングの作成者・作成日・使用回数を監査可能にし、棚卸し画面を用意する（競合公開情報に見えない部分を強化）。
- **Adopt/Reject**: **Adopt（学習型変換＋来歴明示）/ 強化（マッピングの監査・棚卸し）**。
- **Evidence**: 同マニュアル（p.7-12）
- **Access date**: 2026-07-11

## 4. PH-OS への示唆（要約）

1. **段階ゲート＋fail-closed 取込**（§2.1, 2.4）: 外部データは「原本 vs 解釈」並置のステージングで確定させ、未解決は進行ブロック、除外は明示操作＋監査記録。
2. **患者照合の根拠明示**（§2.2）: 一致/不一致属性のバッジ化で同姓同名対策。競合の公開実例より一歩進める。
3. **差分のカテゴリ構造化**（§2.7）: 新規/増量/減量/中止/用法変更を構造データとして保持し指導画面の一次情報にする。
4. **AI・患者回答は draft パイプライン**（§2.9): draft 状態の視覚識別＋確定の監査記録＋来歴（AI/患者/手入力）保持。
5. **非常時**（§2.5): 「閲覧の継続」を最低保証、暗号化オフライン入力まで対応する PH-OS の設計を明確な優位性として仕様化。
6. **競合の公開情報が薄い領域**（アラート階層・オーバーライド監査・疑義照会 UI・操作ログ閲覧・オフライン入力）は、PH-OS が仕様の透明性ごと差別化できる余地。
