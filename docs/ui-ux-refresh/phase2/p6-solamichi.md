# P6: ソラミチシステム『CARADA 電子薬歴 Solamichi』UI/UX 操作原理調査

調査日: 2026-07-11 / 出典種別: 公開一次資料

- 主対象: 『CARADA 電子薬歴 Solamichi』（株式会社ソラミチシステム、クラウド型電子薬歴）
- 比較対象（自選定）: 『Musubi』（株式会社カケハシ、クラウド型電子薬歴・在宅対応）
- 調査方法: 公式製品サイト・公式ニュース/プレスリリース・グッドデザイン賞（公益財団法人日本デザイン振興会）公開ページのみを使用。ログインが必要な資料・非公開マニュアルは未使用。競合製品の画面文言・画像は複製せず、抽象原理レベルで記録。

## 重要な前提訂正（製品分類について）

依頼文では「クラウド型レセコン一体薬歴」とされていたが、公式 FAQ では **Solamichi は現行では電子薬歴システムであり、レセコン一体型ではない**（NSIPS® 対応レセコンとの連携方式）と明記されている（https://site.solamichi.com/faq 、確認日 2026-07-11）。レセコン「調剤くん」（ネグジット総研）との**一体型システムは 2025-09-08 発表のプレスリリース時点で「2025年11月より開発開始予定」**であり、狙いは「服薬指導・薬歴記録から会計・請求業務までの一元管理」「二重入力の解消」「患者情報の双方向連携・電子処方箋データ連携」とされる（https://prtimes.jp/main/html/rd/p/000000107.000077012.html 、確認日 2026-07-11）。**一体型の提供済み UI は公開情報なし**のため、本書は現行のレセコン連携型 Solamichi の公開情報を対象とする。

## 観察パターン一覧

### P6-1: 3回分の処方・薬歴を1画面に並置（画面切替なしの経時比較）

- **Product/source**: Solamichi / 公式グッドデザイン賞受賞インタビュー
- **Screen or workflow**: メイン薬歴画面（薬歴記載・処方確認）
- **Observed pattern**: 「今回・前回・前々回の3回分の処方と薬歴を1画面で見られる」構成。画面を切り替える手間なく処方の連続性を確認できるよう設計。膨大な情報を1画面に収めつつ、紙薬歴のオペレーションと同じ体験を再現することを追求。
- **Problem solved**: 薬剤師は処方の経時変化（継続/変更）を見て判断するため、画面遷移のたびに文脈が失われる問題を解消。紙めくりの認知モデルを維持。
- **Strength**: 電子化しても紙運用の認知負荷を増やさない。ITリテラシーが低いユーザーでも移行しやすい。
- **Limitation**: 3回分固定の並置は長期経過（数ヶ月〜年単位）の俯瞰には不向き。1画面高密度は小画面（タブレット縦）での再現性に課題が残りうる。
- **Possible misinterpretation**: 「3回並置=差分が自動強調される」わけではない。新規/増量/減量/中止のマーキング表示の有無は公開情報なし（後述 P6-欄外）。並置は比較の「材料提示」であり差分計算ではない可能性がある。
- **Patient safety implication**: 前回処方との比較を薬剤師の記憶や画面往復に頼らせないことは、変更見落とし（用量変更・中止薬の再開など）の防止に直結する。
- **Applicable principle for this system**: 調剤/監査/薬歴画面で「今回+直近2回」を既定並置し、スクロール・タブ切替なしで経時比較できるカラムレイアウトを標準にする。PH-OS の chouzai workbench・薬歴画面に適用可能。
- **Adopt/Reject**: **Adopt**。PH-OS の「情報密度: 一覧はExcel的」方針と整合。ただし並置に加えて差分の明示マーキング（新規/増量/減量/中止）を上乗せし、並置だけに頼らない（Solamichi 公開情報では差分マーキング未確認のため、そこは他社比較や自社設計で補う）。
- **Evidence**: https://site.solamichi.com/news/2021/10/20_02 （2021年度グッドデザイン賞受賞特別インタビュー）、https://www.g-mark.org/award/describe/52866 （受賞掲載ページ。詳細本文はページ構造上取得不可だった点を明記）
- **Access date**: 2026-07-11

### P6-2: 指導ナビ — チェック式の指導文提案と「Do処方でも毎回異なる」生成（特許）

- **Product/source**: Solamichi / 公式機能ページ・ソリューションページ
- **Screen or workflow**: 薬歴記載（服薬指導内容の作成）
- **Observed pattern**: 処方データ・過去薬歴・患者情報から望ましい服薬指導内容例を自動表示し、チェックを入れて「薬歴に反映」する2〜3ステップ操作（画面を開く→チェック→反映）。前回指導内容と重複しない独自ロジックで、Do処方でも毎回異なる指導文を生成（2020年特許取得と記載）。「誰でも60点の薬歴が書ける」ことを設計目標として公言。
- **Problem solved**: 薬歴の自由記述は品質が個人差に依存し、Do処方では指導・記録が形骸化（コピペ化）しやすい。提案+選択方式で最低品質を底上げ。
- **Strength**: 手入力を最小化しつつ、監査（個別指導）で問題となる「毎回同一文面」を構造的に避ける。経験の浅い薬剤師でも一定品質。
- **Limitation**: 提案文の選択に慣れると「読まずにチェック」する自動化バイアスのリスク。個別性の高い患者では提案が的外れになりうる（カスタマイズ可とは記載あり）。
- **Possible misinterpretation**: 「AIが薬歴を書く」機能ではなく、ロジックベースの指導文候補提示+人の選択。最終責任は薬剤師の選択・追記にある設計。
- **Patient safety implication**: 指導内容の記録漏れ・画一化を防ぎ、ハイリスク薬などの確認事項を提案に織り込むことで確認漏れを減らす（P6-3参照）。一方、無思考チェックが常態化すると記録と実指導の乖離が生じうる。
- **Applicable principle for this system**: 「自由記述の前に、文脈依存の候補を提示して選択させる」入力原理。PH-OS の服薬指導・訪問報告書で、処方変化・前回指導・患者状態から候補文/確認項目を生成し、選択+追記で完成させる。同一文面の機械的反復を検出して変化を促す仕組みも有効。
- **Adopt/Reject**: **Adopt（選択式候補提示の原理）**。ただし候補の採用操作は「内容が表示された状態でのチェック」に限定し、全選択一括反映のような無確認経路は監査記録（誰がどの候補を採用したか）とセットで設計する。特許取得済みロジック（重複回避生成）の実装模倣は**Reject**（抽象原理のみ学習）。
- **Evidence**: https://site.solamichi.com/solution/sol1 、https://site.solamichi.com/fn/drughistory 、https://site.solamichi.com/news/2021/10/20_02
- **Access date**: 2026-07-11

### P6-3: 加算マークによるアラート — 算定要件と安全確認の同時ナビゲーション

- **Product/source**: Solamichi / 公式薬歴記載ページ・検索結果掲載の公式ソリューションページ
- **Screen or workflow**: 薬歴記載・加算算定（ハイリスク薬加算、乳幼児加算等）
- **Observed pattern**: 処方内容に基づき、ハイリスク薬加算等の算定要件に沿った確認項目・指導文を提案し、「加算マーク」でアラート表示。要件を完全に暗記していなくても確認が完了する構成。指導文は薬効群・剤形・患者層ごとに出し分けカスタマイズ可能。
- **Problem solved**: 算定漏れ（経営課題）と、ハイリスク薬の確認事項漏れ（安全課題）を同一の UI 導線で解決。
- **Strength**: 「儲かるからやる」動機と「安全のためにやる」動機を同じチェックリストに束ね、実施率を上げる行動設計。
- **Limitation**: 算定ドリブンの提示は、算定対象外だが臨床的に重要な確認事項の優先度を相対的に下げるリスク。
- **Possible misinterpretation**: 加算マーク=安全性アラートではない。算定ナビと安全アラート（禁忌・相互作用）は別レイヤーであり、公開情報上も処方監査アラートとは別機能。
- **Patient safety implication**: ハイリスク薬の定型確認（副作用初期症状・検査値等）の実施率向上に寄与。逆に算定と無関係な安全確認の埋没に注意。
- **Applicable principle for this system**: PH-OS でも算定支援（薬服管指導料・在宅関連加算）と安全確認チェックを同一画面のタスクリストに統合しつつ、**安全項目は算定の有無に関係なく独立表示**する2層構造にする。
- **Adopt/Reject**: **Adopt（動機の束ね方）/ 部分Reject（算定を安全確認の主導線にすること）**。安全確認は算定ロジックから独立させ、算定はその副産物として算出する順序を守る。
- **Evidence**: https://site.solamichi.com/fn/drughistory 、https://site.solamichi.com/solution/sol3
- **Access date**: 2026-07-11

### P6-4: 処方監査 — 成分・系統レベルのチェックと「監査ボタンにアラートが上がる」通知方式

- **Product/source**: Solamichi / 公式処方監査ページ・機能一覧ページ
- **Screen or workflow**: 処方監査（薬歴画面内）
- **Observed pattern**: YJコード単位だけでなく「成分」「系統（同種同効薬）」での監査。処方薬×併用薬・疾患・OTC・飲食物・アレルギーによる患者禁忌判定、適応病名禁忌判定、年齢・体重による用量監査。表示は「患者の薬歴画面の処方監査ボタンにアラートが上がる」方式。電子処方箋連携（レセコン経由）による重複投薬チェックにも対応と記載。
- **Problem solved**: コード完全一致依存のチェックでは同成分別剤形・同効薬の見落としが起きる問題を、成分・系統レベルのマッチングで低減。
- **Strength**: 誤陰性（見落とし）を減らす方向のマッチング設計。OTC・飲食物・病名まで判定対象を広げている。
- **Limitation**: アラートの重症度階層（禁忌/併用注意等の段階表示）、色分け、ブロック型か通知型か、警告疲労対策（オーバーライド記録・抑制設定）は**公開情報なし**。系統レベルまで広げると誤陽性が増え、警告疲労を招く可能性があるが、その対策は公開されていない。
- **Possible misinterpretation**: 「監査ボタンにアラート」という受動通知方式を、割り込み型の強制確認と誤解しない。公開情報からはユーザーがボタンを見に行く設計に読める（確定はできない）。
- **Patient safety implication**: 成分・系統マッチングはアレルギー・重複の誤陰性を減らす（PH-OS の CDS が YJ 先頭7桁 prefix 一致を採る判断と同方向）。一方、非割り込み通知は見落としリスクと警告疲労回避のトレードオフ。
- **Applicable principle for this system**: (1) アレルギー・重複は成分/系統レベルの粗いマッチングで誤陰性を優先的に潰す。(2) アラートは重症度で提示様式を変える（禁忌=割り込み・確認必須+理由記録、注意=非割り込みバッジ、情報=一覧内表示）。Solamichi の公開情報にない階層設計は PH-OS 側で明示的に定義する。
- **Adopt/Reject**: **Adopt（成分・系統マッチング、判定対象の広さ）**。通知方式（ボタン上のアラートのみ）は**Reject**候補 — 禁忌級は受動通知では不十分で、重症度別の提示階層を自前で設計する。
- **Evidence**: https://site.solamichi.com/fn/audit 、https://site.solamichi.com/fn
- **Access date**: 2026-07-11

### P6-5: AI音声入力 — 会話をAIが要約し服薬指導形式へ変換

- **Product/source**: Solamichi / 公式トップ・機能ページ
- **Screen or workflow**: 服薬指導〜薬歴作成
- **Observed pattern**: 患者との会話内容を音声から AI が要約し、服薬指導形式（薬歴記載形式）に自動変換するオプション機能。処方薬・過去薬情報を AI が考慮して要約精度を高めると記載。公式には薬歴作成時間を大幅削減とうたう（検索結果上「最大70%削減」表記）。
- **Problem solved**: 指導中のメモ取り・指導後の想起入力という二重負荷を除去し、対話への集中を可能にする。
- **Strength**: 入力起点を「話す」に変えることで、タイピング能力への依存を断つ。在宅・立ち仕事文脈と相性が良い。
- **Limitation**: 要約の誤り（言っていないことの混入・重要発言の脱落）の検証責任はユーザーに残る。確定前レビュー UI の詳細は公開情報なし。
- **Possible misinterpretation**: 音声がそのまま法定記録になるのではなく、AI 出力はあくまで下書きで、薬剤師の確認・確定を経る前提と読むべき（確定フローの公開情報はない）。
- **Patient safety implication**: 記録の網羅性向上の反面、AI 幻覚が薬歴（医療記録）に混入するリスク。下書き/確定の区別と差分レビューが安全要件になる。
- **Applicable principle for this system**: 音声/AI 生成物は必ず「下書き」状態で保持し、確定操作・確定者・AI 由来である旨を監査証跡に残す。生成部分と人手追記部分を視覚的に区別する。
- **Adopt/Reject**: **Adopt（音声→下書き→人手確定の原理）**。下書き/確定区別のない直接記録化は**Reject**。
- **Evidence**: https://site.solamichi.com/ 、https://site.solamichi.com/fn
- **Access date**: 2026-07-11

### P6-6: 在宅ワークフロー — 訪問前計画・訪問後の薬歴→報告書自動転記・施設単位一覧

- **Product/source**: Solamichi / 公式在宅訪問ページ・在宅ソリューションページ
- **Screen or workflow**: 在宅訪問（訪問前→訪問中→訪問後）、報告書作成
- **Observed pattern**: 訪問前=計画書作成、訪問時=状況確認・服薬指導、訪問後=薬歴記載→薬歴内容を医師・ケアマネージャーへの報告書へ自動反映、という3段階を明示。報告書はテンプレート+処方内容自動反映。施設ごとの患者一覧化、患者マーク（薬局ごと最大100個）での分類・報告書一括印刷。介護契約書・重要事項説明書をボタン一つで作成・印刷（契約期間・負担割合で文言可変）。移動中の隙間時間にクラウドで薬歴確認・作成。
- **Problem solved**: 在宅は同一内容を薬歴・報告書・帳票へ多重転記する事務負荷が大きい。「一度書けば下流帳票へ流れる」構造で転記ミス・記載漏れを排除（導入薬局の声として明記）。
- **Strength**: 施設単位のバッチ処理（絞り込み→一括印刷）が在宅の実運用（施設一括訪問・月次報告）に合致。契約書類の自動生成は在宅開始時の事務障壁を下げる。
- **Limitation**: 報告書の宛先は医師・ケアマネ中心で、訪問看護等を含む双方向の多職種連携機能は公開情報なし（紙/データ送付との記載まで）。オフラインでの訪問先動作は公開情報なし。
- **Possible misinterpretation**: 「多職種連携」は共有ワークスペース型ではなく、帳票出力（印刷・送付）型と読むのが公開情報に忠実。
- **Patient safety implication**: 転記レスは報告書と薬歴の不整合（別文書間の矛盾）を防ぎ、医師の処方判断材料の正確性を担保する。
- **Applicable principle for this system**: PH-OS の訪問報告書は「薬歴/指導記録を単一ソースとし、報告書は投影+編集」で生成する。施設×期間×患者タグでの一括出力、契約書類のパラメトリック生成を備える。PH-OS は双方向の多職種連携（コメント・ワークフロー）を持つ点で公開情報上の Solamichi より進んだ形を狙える。
- **Adopt/Reject**: **Adopt**（単一ソース→帳票投影、施設単位バッチ、契約書類生成）。
- **Evidence**: https://site.solamichi.com/fn/athome 、https://site.solamichi.com/solution/sol2
- **Access date**: 2026-07-11

### P6-7: タブレット写真記録 — 残薬・セット薬の視覚証跡（撮影者・日時付き）

- **Product/source**: Solamichi / 公式在宅訪問ページ
- **Screen or workflow**: 在宅訪問先での記録
- **Observed pattern**: タブレットで患者の様子・薬（セット薬・残薬）・メモ書きを写真撮影して患者記録に保存。撮影者・撮影日時が記録され、クラウド経由で薬局 PC 側からいつでも参照可能。
- **Problem solved**: 残薬状況・服薬環境など文章化しにくい情報を、転記なしで正確に共有。訪問者と店舗内スタッフの情報格差を縮小。
- **Strength**: 撮影者・日時のメタデータは記録の証跡性を担保。文章より速く、誤読が少ない。
- **Limitation**: 患家での撮影は同意・プライバシー配慮が前提（運用ルールは公開情報なし）。写真の保存期間・アクセス制御の詳細も公開情報なし。
- **Possible misinterpretation**: 写真は薬歴本文の代替ではなく補助証跡。写真だけで残薬数等の構造化データが得られるわけではない。
- **Patient safety implication**: 残薬・一包化セット状態の視覚記録は、服薬アドヒアランス評価と調剤過誤の遡及調査の両方に効く。
- **Applicable principle for this system**: 訪問記録に写真添付（撮影者・日時・患者紐付けの自動メタデータ、S3 Object Lock で改竄耐性）を用意し、残薬数などは写真+構造化入力の併用にする。
- **Adopt/Reject**: **Adopt**。PH-OS は Structured Data First なので、写真は構造化データの補助証跡と位置付ける。
- **Evidence**: https://site.solamichi.com/fn/athome
- **Access date**: 2026-07-11

### P6-8: 色彩・トーン設計 — ブルー基調・「疲れない」「迷子にならない」配色

- **Product/source**: Solamichi / 公式グッドデザイン賞受賞インタビュー
- **Screen or workflow**: 全画面共通のビジュアル設計
- **Observed pattern**: メインカラーは空と医療への信頼感を表すブルー。グレー基調の従来医療システムと差別化し、「見ていて疲れない」ことと「迷子にならない色彩設計」を両立。頻繁なクラウドアップデートでも当初のデザインコンセプトを維持する運用。
- **Problem solved**: 長時間注視する業務システムでの視覚疲労と、機能追加によるデザイン一貫性の崩壊。
- **Strength**: 「アップデートしてもコンセプトを守る」というガバナンスを公言している点。色を情動（安心感）とナビゲーション（迷子防止）の2役で使う整理。
- **Limitation**: 具体のカラートークン・警告色階層は公開情報なし。
- **Possible misinterpretation**: 「ブルー基調=警告も青系」ではない。警告色設計は別論点であり公開されていない。

## PH-OS への示唆（要約）

1. **経時比較は並置が業界標準形**（Solamichi 3回分/Musubi 2回分・処方元グルーピング）。PH-OS は並置+明示的差分ラベル（新規/増量/減量/中止）で一歩先を行く。
2. **「提案+選択」入力と「行為=記録」入力**が薬歴作成の二大効率化原理。いずれも下書き/確定の明確な2状態と監査証跡を前提に採用する。
3. **成分・系統レベルの粗いマッチング**で誤陰性を潰す方向は Solamichi も採用しており、PH-OS の CDS 方針（YJ prefix 一致）と整合。重症度階層・オーバーライド記録は両製品とも非公開のため PH-OS が明示設計で差別化できる。
4. **在宅は「単一ソース→帳票投影」+「訪問モード専用の削減画面」+「施設単位バッチ」**が共通解。高速患者切替には取り違え防止の患者常時表示を必ず併設。
5. **オフライン縮退・非常時運用・監査証跡の可視化は両製品とも公開情報なし** — PH-OS の PWA/オフライン対応・Audit by Default は公開情報ベースで明確な差別化ポイント。
