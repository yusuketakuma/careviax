## 3. 採用する抽象原理（8軸）

各原理は「複数製品で観測された抽象構造」であり、根拠ノートを付す。採用理由は PH-OS 設計原則との整合であって観測数ではない。

### 3.1 情報優先順位

- **「対面/集中モーメントの必要情報セット」を先に定義してから画面を組む**。指導・監査中はスクロール/タブ切替ゼロで完結する情報（患者コンテキスト+今回処方+要注意情報+直近履歴）を単一視野に置き、それ以外は意図的にスクロール外へ（p2 パターン1、p4 パターン4、p5 パターン2、p6 P6-1）。根拠: 指導中の画面操作は対話を中断し、見えていない情報は存在しないのと同じ。PH-OS の「一覧=高密度/入力=ゆとり」方針の具体化。
- **安全属性（アレルギー・ハイリスク・在宅区分）は基本画面ヘッダへ昇格し常時掲出**（p5 パターン9、p3 §2.7 のアレルギー常設枠）。PatientHeader SSOT の tier 設計に統合する。
- **強調は「要チェック項目のみ」に絞る**。全情報の同列羅列+装飾過多を避け、色強調の対象を厳選して意味論を守る（p2 パターン1、p4 パターン6）。

### 3.2 認知負荷

- **原本と同型の入力レイアウト**: 入力画面は原本（処方箋・報告書式）の記載順と同型にし、転記時の視線移動・記憶負荷を消す（p4 パターン2）。
- **取込系は「原本ビュー vs 解釈ビュー」並置**: OCR/QR/電子処方箋の取込は source vs system の2カラム照合+一致状態のラベル（色+アイコン+テキストの三重符号化）+その場で解決できるアクション（p3 §2.3–2.4、p4 パターン12）。
- **選択式・ガイド付き入力をモバイルの既定に**: 訪問先はテンプレート/チェック先行・フリーテキスト補助（p3 §2.8、p5 パターン1）。「最小限の情報で次のアクションが明確」という PH-OS モバイル方針と一致。
- **入力モダリティは複線化し強制しない**（タッチ/キーボード/音声。p5 パターン12）。

### 3.3 差分認識

- **経時比較は並置+明示的差分ラベルの二段構え**: 同一処方元でグルーピングした直近数回の並置（p1 P-02、p6 P6-1/P6-10）に加え、変化種別（新規/増量/減量/中止/用法変更）を**構造化データとして保持**しアイコン+テキスト併記で表示する（p3 §2.7）。並置＝材料提示、ラベル＝判断支援であり両方必要。差分語彙は全製品で公開情報がなく、**PH-OS が独自定義する**。
- **差分は入力/監査の「最中」に見せる**（事後アラートでなくリアルタイム。p4 パターン3）。差分ゼロ時も「前回と同一」を積極表示する。
- **同一成分の行アラインメント**（YJ 上位桁）で剤形変更・同効薬切替が「中止+新規」に見える文脈喪失を防ぐ（p1 P-02）。
- **差分は薬剤師向け監査ビューと患者向け説明ビューの二形態**を持つ（p2 パターン5: 差分を患者コミュニケーション資材として設計する視点）。

### 3.4 患者コンテキスト維持

- **PatientHeader（患者識別 SSOT）を全画面・全モードで固定表示**し、特に高速患者切替（施設ラウンド）の直後は識別表示を強調する。クイック切替は取り違えも速くする諸刃であるため、切替高速化には確認レイヤを必ず併設（p1 P-07、p5 パターン8、p6 P6-11）。
- **患者照合は「一致/不一致属性の根拠明示」**: 検索・外部取込時に、なぜこの候補かを属性バッジで示し、部分一致選択時は生年月日・性別確認を挟む（p3 §2.2、p5 パターン9 の取込時相違の赤字確認）。公開実例が最も具体的な p3 を基礎に、不一致属性の強調まで PH-OS が一歩進める。
- **下流ロールへの読み取り専用の先行アクセス**（会計前の薬剤師先確認。p3 §2.10）: ワークフロー状態遷移と独立に、同一患者コンテキストへの並行参照レーンを許す。

### 3.5 状態遷移伝達

- **下書き/確定の2状態をデータモデルと UI の両方で明示**し、確定は人間の明示操作のみ。AI・OCR・患者自己申告などの自動生成物はすべて draft パイプラインに載せ、由来（AI/患者/手入力/変換マスタ）をデータとして保持・表示する（p1 P-01/P-09、p2 パターン6/8、p3 §2.9/2.12、p5 パターン5/6、p6 P6-5）。
- **確定を単一品質ゲートとし、下流帳票（報告書・計画書・情報提供書）は確定情報の投影として派生生成**する（p5 パターン5、p6 P6-6）。確定後修正は訂正履歴必須の別フロー。
- **未完了（下書き・未記載・差戻し・未署名）はホーム/ダッシュボードに件数+経過時間で常設可視化**し、クリックで再開できる（p4 パターン7、p5 パターン10、p2 パターン10 の例外駆動コンプライアンスキュー）。締め処理前の必須ゲートにする。
- **未解決の取込データは fail-closed**（進行ブロック）とし、除外は明示操作+監査記録として残す。黙殺（silent drop）を構造的に排除（p3 §2.4）。

### 3.6 高頻度業務短縮

- **「行為=記録」の一体化**: 指導・確認のタップ/チェックが構造化データとして記録に直結し、記録が業務の副産物になる（p1 P-01、p4 パターン5、p5 パターン1、p6 P6-2/P6-9）。指導フローの順序=入力 UI の順序。形骸化対策として確定前の必須確認と個別性フィールドを併設。
- **キーボード完結パスを一級市民として設計し、キーストローク互換性をリリースポリシーに含める**（p4 パターン1）。既存の調剤ワークベンチ F12→Enter 文化と一貫。
- **選択コンテキスト依存のキー操作ガイド**: フォーカス行で「今できる操作」を常設表示し、熟練者の速度と初心者の発見可能性を両立（p2 パターン2、p3 §2.6）。
- **「書く単位（患者）」と「送る単位（施設・宛先）」の分離**: 施設単位の一括作成・一括出力を提供しつつ、確定は 1 件ずつの確認を強制する（p1 P-08、p6 P6-6）。
- **一度の人間の判断を再利用する学習型変換**（一般名→調剤薬の org 単位マッピング）+適用時の来歴バッジ+棚卸し導線（p3 §2.12）。

### 3.7 誤操作防止

- **最終照合は「UI レス」のスキャン照合+NG 時のみブロッキング介入**: 監査工程は GS1/JAN スキャンを追加画面操作なしで組み込み、照合イベントを監査証跡へ自動記録。NG 継続には理由必須（p4 パターン8）。
- **役割分担の承認イベント化**: 非薬剤師作業の薬剤師承認を明示イベントとしてモデル化し、承認画面に機械監査結果を必ず並置する（p4 パターン9）。maker/checker のドメイン内実装。
- **アラートは「一次表示=存在+種別、二次表示=根拠詳細」の二層**で画面密度を守りつつ（p1 P-03）、重症度で提示様式を変える（禁忌=割り込み+理由記録、注意=非割り込み、情報=一覧内）。CDS 照合キーは成分/薬効レベル（YJ prefix）で誤陰性を優先的に潰す（p2 パターン3、p6 P6-4 — PH-OS の既存 CDS 方針と同方向）。
- **確定キーは通常操作キーと位置分離**し、無思考連打（confirmation fatigue）を抑制する（p3 §2.6 の教訓）。
- **「事務的未解決の赤」と「臨床アラートの赤」の視覚語彙を分離**する（p3 §2.4 の教訓。PH-OS 6軸状態色トークンで実装）。

### 3.8 例外回復

- **劣化モードの明示設計**: 非常時（回線断・障害）に使える機能範囲を明文化し、切替/復帰を明示的 UI と手順にし、復帰後の同期検証を要件化する（p2 パターン9、p3 §2.5）。「閲覧の継続」を最低保証ラインとし、PH-OS はさらに暗号化オフライン入力+復帰時同期（PWA/Dexie）まで仕様化して差別化する — 全製品の公開情報上の空白領域。
- **訪問データの選択的プリフェッチ+PHI の時限自動削除（TTL パージ）**: 訪問予定に紐づく事前取得と、期限付きローカル保持で可用性と機密性を両立（p5 パターン7）。予定外訪問向けのオンデマンド経路も残す。
- **エラー地点での解決手段の同居**: 照合不一致・要確認状態の表示と、その解決アクション（追加/切替/マスタ編集）を同一画面に置き、復帰動線を最短化（p3 §2.3）。
- **次回確認事項（OP 相当）のクローズループ**: 申し送りを「次回必ず表示される場所」に構造化し、確認済み操作まで含めて閉じる（p1 P-06。表示だけでは読まれる保証がないため、公開仕様より一歩保守化）。
- **申し送り・フォローアップの自動サーフェスとエスカレーション**: 対象提案→予定→実施→記録のワークフロー化、異常回答の優先度付き昇格（p4 パターン13）。

## 4. 採用しない原理とその理由

| 不採用の原理 | 観測元 | 理由 |
|---|---|---|
| 端末非保存（シンクライアント型）のモバイル | p3 §2.8、p4 パターン10 | オンライン前提であり、在宅特化の PH-OS では圏外業務継続（暗号化オフライン PWA）が中核価値。非保存の「説明可能なセキュリティ物語」だけは採り、リモートワイプ/キー失効/TTL の可視化で漏えい対策の説明責任を果たす |
| フルHD 前提の固定高密度レイアウト | p2 パターン1、p4 パターン4、p5 パターン2 | Mobile First と矛盾。情報構造・用語・状態色の同一性を保ち**密度のみ可変**にする（p2 パターン7 の抽象原理側を採用） |
| 個人別の自由レイアウト設定 | p4 パターン4 | 応援勤務・サポート・監査可能性・マルチテナント SaaS の一貫性と相性が悪い。役割別の固定プリセットに留める |

| 禁忌級アラートの受動通知（ボタン上のバッジのみ） | p6 P6-4 | アレルギー・禁忌は false-negative の害が桁違いであり、重大アラートは割り込み+確認必須+理由の監査記録とする |

| Do 処方複写の無条件高速化 | p3 §2.6、p4 パターン3 の教訓 | 「前回と同じ=安全」ではなく誤りの固定化リスクがある。Do 入力の短縮は差分表示との併置を条件とする |
| 紙処方箋の記載順最適化への過剰投資 | p4 パターン2 の Limitation | 電子処方箋の主流化で相対価値が低下する。原理（原本と同型）は採るが、紙固有レイアウトへの固定はしない |
| 手動 useMemo 等の実装レベル模倣一般 | — | 本ベンチマークは操作原理の学習であり、実装手段は PH-OS の技術方針（React Compiler、状態色トークン等）に従う |

## 5. コピー回避の判断（固有表現を複製しないための抽象化方針)

1. **抽象化の階層を固定する**: 参照してよいのは「問題→抽象原理→トレードオフ」の層のみ。画面文言・ラベル語彙・アイコン意匠・配色値・レイアウト座標・キー割当表・キャッチコピーは各ノートでも複製しておらず、本書からの設計にも持ち込まない。差分ラベル（新規/増量/減量/中止/用法変更）等の語彙は**全製品で公開情報がない領域であり、PH-OS が独自定義する**（そもそも複製元が存在しない）。
2. **特許・意匠が明示された手法は具体実装を模倣しない**: AnyCOMPASS の赤マーク表示（特許出願済）、Solamichi の重複回避指導文生成（特許取得）、Musubi SOAP 自動生成関連は、抽象原理（必須項目の厳選強調、文脈依存候補提示）のみ参照し、実現ロジック・表示手法は独自設計する（p4 パターン6、p6 P6-2 に明記）。
3. **機能名・商標は事実参照に限定**: 「瞬間薬歴」「1分薬歴」等の名称は出典明示の事実記述としてのみ使用し、PH-OS の機能名・UI 文言には流用しない。
4. **推測による補完の禁止を複製回避にも適用する**: 公開情報がない画面仕様を「おそらくこうなっている」と再構成する行為は、誤った設計根拠になると同時に、実物と偶然一致した場合の意図せぬ複製リスクになる。各ノートの Possible misinterpretation / 公開情報なし節に従い、空白は空白のまま PH-OS 独自設計の領域とする。
5. **採用時は必ず PH-OS 既存規範へ翻訳する**: 採用原理は状態色6軸トークン、PatientHeader SSOT、WCAG AA（色+アイコン+テキスト）、警告3段階、draft/confirmed 状態機械、Audit by Default という PH-OS 固有の語彙・部品系に写像してから実装する。この翻訳工程自体が表現の独立性を担保する。
6. **出典の追跡可能性**: すべての観測は製品ノートの Evidence(URL)+Access date に遡れる。設計レビュー時に「どの観測に基づく原理か/どこからが独自判断か」を分離検証できる状態を維持する。
