#### 第1.1版 → 第2.0版の主な改定内容（案ベース + 総務省報道資料）

1. **対象事業者の明確化**: 直接契約のない資源・役務提供事業者、患者指示による情報受領事業者まで対象範囲を明文化。
2. **合意内容の明確化**: 医療機関等へ情報提供すべき項目（表4-1）の精緻化。第三者認証の位置づけ見直し — プライバシーマーク認定または ISMS 認証の取得は引き続き必須で、「特にプライバシーマーク認定が望ましい」「ISMS 認証は管理の適正の認証であり安全対策の有効性まで認証するものではない」旨を追記。代替可能なクラウドセキュリティ認証等から JASA CS ゴールドマーク・ISACA 認定を削除し、ISMAP（**ISMAP-LIU は含まない**）、FedRAMP（**LI-SaaS は含まない**）、SOC2/SOC3、および「民間事業者による医療情報に係るクラウドサービスの評価」（保健医療福祉情報安全管理適合性評価協会）へ整理。ただし「これら認証の取得をもって本GLが求める安全管理水準を満たすわけではない」と明記。
3. **リスクコミュニケーションの実効化**（5.1.6）: 約款契約（パブリッククラウド型 = PH-OS の提供形態）では個別契約調整が難しいため、事業者側に「より丁寧でわかりやすい情報提供」を要求。具体策として (a) リスク判断に必要な基礎資料の提供（MDS/SDS 等、主要セキュリティ事項のチェック結果）、(b) 医療機関等から説明を求められた場合の対応の表示、(c) **約款契約におけるリスクの表示（民法548条の2〔定型約款〕関係）**、(d) 医療機関等がリスク・役割分担等を確認した上でサービス利用する旨の明示と合意、を列挙。
4. **インシデント事例のコラム追加**: 徳島県つるぎ町立半田病院ランサムウェア事案を引き、責任分界・初動対応・脆弱性管理・設計上の課題（リスクコミュニケーション不足）を教訓化。
5. **継続的リスクマネジメント**: リスクは選定時から提供中に変化しうるため、契約締結後も継続的に見直し・再合意（JIS Q 27001:2023 / ISO/IEC 27001:2022 参照へ更新）。
6. 死者に係る情報も個人情報と同等の安全管理措置（医療・介護個人情報ガイダンス 令和6年3月27日最終改正 参照）。

#### 医療機関等へ情報提供すべき項目（表4-1、案ベース）

- 外部保存受託事業者の選定基準として最低限確認される項目: 情報機器等への国内法の適用、安全管理基本方針・取扱規程の整備状況、サイバー攻撃被害防止のためのバックアップ取得・管理状況、実施体制、実績に基づく信用度、経営の健全性、プライバシーマーク認定または ISMS 認証、（未取得の場合の代替認証等）、情報機器の設置場所（地域・国）、受託事業者への国外法の適用可能性。
- 共通理解形成のための項目: 役割分担の明確化事項、安全管理評価の結果、**医療情報システム等の全体構成図**、**リスク対応一覧**、運用管理規程に含める事項（下記(ア)〜(サ)）、制度上の要求事項への対応。

#### 事業者が定める運用管理規程の必須項目（ア)〜(サ)（5.1.6(2)、案ベース）

(ア) 安全管理基本方針（本GL・厚労省GL・個情法遵守、死者情報の同等管理、情報セキュリティポリシー等） / (イ) 提供体制（最終管理責任者・システム管理者・個人情報保護責任者、再委託先体制含む） / (ウ) 契約書・マニュアル等の文書管理（不正閲覧・操作のアクセス制限と検出・記録） / (エ) 機器等の管理責任の所在・台帳管理 / (オ) リスク対応策の運用方法 / (カ) **事故発生時の対応方法および医療機関等への報告方法**（被害拡大防止、緊急時の代替手段、原因調査のためのログ等の保全、報告タイミング・報告フロー） / (キ) 個人情報を格納する記憶媒体の管理 / (ク) 医療機関等の危機管理対応時の受託事業者側体制・対応内容 / (ケ) **医療情報の外部保存に係る患者等への説明方法**（医療機関等への資料提供、または事業者が直接説明する場合の方法） / (コ) 監査の実施方針（他社システム利用時の監査方針含む） / (サ) **医療機関等の管理者からの一元的な問い合わせ窓口**（連絡先・連絡方法・受付時間帯）。

#### チェックリスト構成（令和8年度版、マニュアル目次より）

1. **体制構築**〔医療機関・薬局確認用／事業者確認用〕: ① 医療情報システム安全管理責任者の設置（事業者側も提供に係る管理責任者の設置が必要）。
2. **医療情報システムの管理・運用**〔両用〕:
   - ① サーバ・端末・ネットワーク機器の台帳管理
   - ② リモートメンテナンス（保守）機器の有無の相互確認（**事業者は外部接続点の存在を定期的に医療機関等へ通知**）
   - ③ **医療情報セキュリティ開示書（MDS/SDS）の提出**（JIRA/JAHIS 様式、厚労省標準規格）
   - ④ 職種・担当業務別の情報区分毎のアクセス利用権限設定（管理者権限は最小限のユーザに限定、利用者ID台帳整備が望ましい — マニュアルの台帳例には「薬剤師/非常勤薬剤師/事務/非常勤事務」の利用者属性例あり）
   - ⑤ 退職者・未使用アカウントの削除または無効化
   - ⑥ セキュリティパッチ適用
   - ⑦ パスワードは英数字混在の8文字以上
   - ⑧ パスワード使い回しの禁止
   - ⑨ USB ストレージ等の外部記録媒体・情報機器の接続制限
   - ⑩ バックグラウンドで動作する不要なソフトウェア・サービスの停止
   - ⑪ **アプリケーションログイン時の二要素認証の実装**
   - ⑫ OS ログイン時の二要素認証
   - ⑬ アクセスログの管理（サーバ）
   - ⑭ 接続元制限（ネットワーク機器）
3. **インシデント発生に備えた対応**〔両用〕: ① 組織内・外部関係機関（事業者、厚労省、警察等）の連絡体制図 ② 診療（調剤）継続に必要な情報の検討とバックアップ・復旧手順の確認 ③ サイバー攻撃の想定を含む BCP 策定。
4. **規程類の整備**〔医療機関・薬局確認用〕: 1〜3 の実施方法を運用管理規程等に明文化。

#### 事業者確認用の回答規約（責任分界の運用ルール）

- 「はい」= 保守契約範囲に含まれ**事業者側の責任**で対応済み。「いいえ」= 契約範囲内だが未対応（令和8年度中の対応目標日を記入）。「**対象外**」= 保守契約範囲外で**当該項目の責任は医療機関等が負う** — 事業者は医療機関等との間で**責任分界の認識齟齬がないか必ず確認**すること。
- 事業者と契約のない医療情報システム（製品売買のみで運用・保守契約なし）は事業者確認用の作成不要だが、その場合の保守管理責任は医療機関等側。
- 少なくとも年1回の点検、立入検査前の再確認。医療機関等は各事業者からチェックリストを回収・保管。

## 2. SaaS 事業者（PH-OS）としての責任分界・リスクコミュニケーション整理

| 契約形態 | 約款契約（定型約款）による画一的提供では、より丁寧・平易な情報開示が必要。民法548条の2 関係のリスク表示 | **Mandatory / Requires legal review** | 利用規約・重要事項説明にリスク表示条項。表示 UI は法務確認後に設計 |
| 開示文書 | MDS/SDS（JIRA/JAHIS 様式・厚労省標準規格）、サービス仕様適合開示書、SLA、システム全体構成図、リスク対応一覧 | **Mandatory** | MDS/SDS を作成し顧客薬局へ提出可能にする。別紙（02besshi_20250328.pdf）の参考例に準拠 |

| 第三者認証 | プライバシーマーク認定または ISMS 認証は必須（P-mark 推奨）。ISMAP 等クラウド認証は代替可だが「取得 = GL水準充足」ではない | **Mandatory / Requires security review** | **AWS が ISMAP 取得済みであることは PH-OS 自身の適格性を示さない**（PH-OS 運営主体としての P-mark/ISMS 取得計画が必要）。ISMS はスコープを医療情報の入口から出口まで包括的に設定し、適用宣言書は医療機関等への開示前提で記載 |

## 3. UI/運用への含意（チェックリスト項目 → PH-OS 機能マッピング）

チェックリスト「事業者確認用」で PH-OS が「はい」と答えられる状態を作ることが製品要求になる。UI に直結するもの:

1. **二要素認証（2-⑪ アプリケーションログイン）** — Mandatory。Cognito MFA（TOTP/FIDO2）採用方針と整合。テナント（薬局）管理者が MFA 強制状態を確認できる管理 UI があると、薬局側チェックリスト記入を直接支援できる。
2. **パスワードポリシー（2-⑦⑧: 英数字混在8文字以上・使い回し禁止）** — Mandatory。Cognito パスワードポリシーで担保し、パスワード設定画面の要件表示文言をチェックリスト文言と揃える（「英数字混在8文字以上」を下回る要件にしない）。
3. **職種・担当業務別アクセス権限 + 管理者権限最小化 + 利用者ID台帳（2-④）** — Mandatory / **Requires security review**。マニュアルの台帳例は「薬剤師/非常勤薬剤師/事務」の属性別権限を例示。PH-OS の org-wide アクセスモデル（薬剤師フル/事務 read-all）は意図的仕様だが、(a) 管理者権限（Admin）付与の最小化、(b) 所属・氏名・ユーザID・権限・状態を一覧できる**利用者台帳エクスポート/画面**は立入検査対応として直接的な支援機能になる。
4. **退職者・未使用アカウントの削除/無効化（2-⑤）** — Mandatory。ユーザ管理画面に「最終ログイン日時」表示と無効化操作、長期未使用アカウントの検出リストが有効。
5. **アクセスログ管理（2-⑬）** — Mandatory。Audit by Default 方針と整合。薬局管理者が自局の操作ログを確認・出力できる画面は、チェックリスト「はい」の根拠提示に使える。
6. **接続元制限（2-⑭）** — Conditionally applicable / Requires security review。SaaS では薬局内ネットワーク機器の話が主だが、テナント別 IP 制限等を提供する場合はその設定 UI。
7. **バックアップ・復旧手順の確認（3-②）と BCP（3-③）** — Applicable。PH-OS のオフライン PWA（Dexie/IndexedDB）は「システム停止時の調剤継続」という BCP 文脈で顧客価値になるが、**非常時モードであることの明示（オフラインバナー等）と復帰手順の提示**が UI 要求になる。事業者側としてはバックアップ取得・管理状況の開示（表4-1）が必要。
8. **連絡体制図（3-①）・インシデント連絡** — Mandatory（運用）。アプリ内に障害・セキュリティ告知の一貫した表示位置（バナー/お知らせ）と、問い合わせ窓口・受付時間帯の常設表示。
9. **機器台帳（2-①）** — Not applicable（薬局側義務）だが、薬局が PH-OS 利用端末を台帳管理する際に参照する「ログイン中の端末/セッション一覧」があれば支援になる（優先度低）。
10. **立入検査対応** — 薬局は薬機法に基づく立入検査でチェックリスト・台帳・連絡体制図・BCP・規程類の現物を確認される。PH-OS 側で「チェックリスト記入支援パック」（事業者確認用回答済み様式 + MDS/SDS + 利用者台帳出力 + 監査ログ出力）を提供できれば、在宅特化 SaaS としての運用差別化になる。
