## 2. 第6.0版 → 第7.0版の改定確認（調査テーマの必須確認事項）

- 第6.0版（2023年5月）は**改定済み**。実行時点（2026-07-11）の最新版は**第7.0版（令和8年6月）**。
- 第7.0版の主な変更点（概説編 1.「はじめに」より）:
  - 根拠法に**サイバーセキュリティ基本法**（平成26年法律第104号）を追加し、「重要インフラのサイバーセキュリティに係る安全基準等策定指針」との整合性を確保。
  - すべてのサーバのセキュリティアップデートを事業者に委託している医療機関等の負担軽減のため、**保守委託機関編を創設**（4編→5編構成に）。
  - 二次情報（策定通知の報道）では、二要素認証のクライアント端末・サーバ双方での対応明確化、サイバー対処能力強化法成立等が背景として挙げられている（一次資料のシステム運用編14章で裏付け確認済み。§6 参照）。
- **本プロジェクト（CLAUDE.md 等）が参照している「MHLW ガイドライン v6.0」は旧版となったため、参照の更新が必要。**

## 3. 構成（第7.0版・5編）

概説編 3.（図3-1）より:

| 編 | 想定読者 | 内容 |
| --- | --- | --- |
| 概説編 | 全読者 | 各編に共通する前提（目的・対象・構成・安全管理3要素 C/I/A・関連法令・認証認可・外部保存の考え方） |
| 経営管理編 | 経営層 | 医療情報システムの安全管理に関する統制。経営層として遵守・判断すべき事項 |
| 企画管理編 | システムの安全管理者（企画管理者） | 組織全体の安全対策の管理、組織的な対応（規程整備・責任分界・リスク管理・非常時方針・認証権限規程等） |
| システム運用編 | システム運用担当者 | 技術的な対応（アーキテクチャ、ネットワーク、認証認可、ログ、バックアップ、電子署名等） |
| 保守委託機関編（新設） | 医療機関管理者・担当者 | クラウドサービス利用やオンプレサーバの保守委託等により、セキュリティアップデートを含む保守業務を委託できている医療機関等向けの統制・管理・対策 |

- 重要: 「セキュリティアップデートを含むサーバの保守を事業者に全て委託できている医療機関等」は、**概説編＋保守委託機関編のみに対応することでガイドライン遵守とみなされる**（概説編 2.1／3.1.5）。想定は専任システム担当者がいない小規模医療機関等で、クラウド（特に SaaS 型）の積極採用により特別な契約なく保守委託を実現することを想定と明記。
- → **PH-OS（SaaS）の顧客薬局はこの軽量遵守ルートの主要な対象**。PH-OS 側が保守委託機関編の前提（事業者がセキュリティアップデート責任を負うことの契約書・SLA 等への明記）を満たすことが、顧客の遵守コストを大きく下げる商品価値になる。

## 4. 薬局への適用性

- 概説編 2.1 が対象の「医療機関等」に**薬局を明記**。
- 企画管理編 1.1.2 は、医薬品医療機器等法施行規則第11条第2項第1号を引用: 薬局の管理者は「その薬局の業務に係るサイバーセキュリティ…の確保のために必要な措置を講じ」る義務を負う（法令上の直接義務）。
- 薬局業務文書の e-文書法対象（電子保存に3条件が必要な文書。企画管理編 表1-3）:
  - 薬剤師法第28条の**調剤録**／保険薬局及び保険薬剤師療養担当規則第6条の調剤録
  - 薬剤師法第26条・第27条の**処方せん**、保険薬局及び保険薬剤師療養担当規則第6条の処方せん、麻薬及び向精神薬取締法第27条第6項の処方せん 等
- 外部保存を認める記録（企画管理編 表1-4）: 薬剤師法第27条の調剤済みの処方せん、同法第28条の調剤録、保険薬局及び保険薬剤師療養担当規則第6条の調剤済みの処方せん及び調剤録 等 → **AWS 東京リージョンでの外部保存（クラウド保存）は外部保存通知の基準充足を前提に可能**（概説編 4.7）。
- 訪問看護ステーション・介護事業者も対象であり、訪問看護計画書・報告書等も e-文書法対象文書として列挙（企画管理編 表1-3 続き）→ PH-OS の多職種連携ドメインにも同一規律が及ぶ。
- サイバーセキュリティ対策チェックリストは第7.0版から**医療機関・薬局統合版**として提供。薬局用 BCP 確認表・手引きも公式提供あり。

## 5. 真正性・見読性・保存性（電子保存の3条件）

企画管理編 1.1.2（施行通知第二の2（3）の引用）より:

1. **見読性の確保**: 必要に応じ電磁的記録に記録された事項を出力することにより、直ちに明瞭かつ整然とした形式で使用に係る電子計算機その他の機器に表示し、及び書面を作成できるようにすること。
   - (ア) 情報の内容を必要に応じて肉眼で見読可能な状態に容易にできること
   - (イ) 情報の内容を必要に応じて直ちに書面に表示できること
2. **真正性の確保**: 保存すべき期間中における改変又は消去の事実の有無及びその内容を確認することができる措置を講じ、かつ、当該電磁的記録の作成に係る責任の所在を明らかにしていること。
   - (ア) 故意又は過失による虚偽入力、書換え、消去及び混同を防止すること
   - (イ) 作成の責任の所在を明確にすること
3. **保存性の確保**: 保存すべき期間中において復元可能な状態で保存することができる措置を講じていること。

補足（システム運用編 14.3「電子カルテデータの確定」）:
- 真正性とは「正当な権限で作成された記録に対し、虚偽入力、書換え、消去及び混同が防止されており、かつ、第三者から見て作成の責任の所在が明確であること」。**「混同」とは患者を取り違えた記録や、記録された情報間の関連性の誤り**を指す（→ 患者コンテキストの明確なUI表示が真正性要件に直結）。
- 訴訟等において証拠能力を有するレベルの正確さの担保が要求される。
- ネットワーク経由の外部保存では転送途中の改ざん・混同防止措置が必要。
- 入力者及び確定者の識別・認証、記録の確定、識別情報の付与、更新履歴の保存措置。代行入力時は入力者と確定者の責任関係が明確になる識別・認証方法。

## 6. 認証・アクセス管理

### 6.1 利用者の登録・認証（企画管理編 13章、システム運用編 14章）

- リスク評価に基づく認証・アクセス権限規程の整備（企画管理編 13 遵守事項①）。
- 利用者登録時は**高い強度の身元確認**（対面又はこれに準じた形）。NIST SP800-63-4 の IAL を参照し、医療機関等では **IAL2 以上が望ましい**（企画管理編 13.1.2 及び脚注5）。人事名簿等と整合した ID 付与、退職時の利用者登録削除。
- 当人認証は**二要素認証又はこれに準じた方法**を要する（企画管理編 13.1.2）。当人認証保証レベル Level 1〜3 の考え方を提示（表13-1。Level 1 は多要素認証必須＋フィッシング耐性をもつ認証手法を全利用者に必須とする最も厳格な水準）。
- **二要素認証の期限（システム運用編 14 遵守事項⑤・表14-1）**: 令和9年度（令和9年4月1日時点）で稼働が想定される医療情報システムを新規導入又は更改する際は二要素認証を採用（又は相当する対応）。技術的理由で困難な場合は令和9年度以降直近の更新までが期限。
  - **クライアント端末とサーバのいずれも**二要素認証の実装を要する。クライアント端末はアプリケーションのログイン時、サーバは OS での二要素認証（表14-1: 「OS で一要素、アプリで一要素」の組合せは許容されない）。
  - サーバ OS の代替: 別ドメインの踏み台端末からの RDP/SSH 接続限定＋踏み台の OS ログインに二要素認証、等の同等措置を許容。
- **パスワードポリシー（システム運用編 14.1.1）**: 二要素認証採用が前提なら英数字混在8桁以上（PIN は4桁以上）。二要素認証を採用しない場合は**13桁以上**。**パスワードの定期的な変更は不要**とする。類推されやすいパスワードの制限、システム間の使い回し禁止、ハッシュ保存（平文復元不可）、**一定回数の認証失敗で以降一定時間ログイン試行不能とする仕組み**（遵守事項⑥）。
- ID の台帳管理・定期棚卸し・不要 ID の削除（遵守事項⑧、企画管理編 13 遵守事項⑦）。
- 外部アプリケーション連携（REST API）: API 連携により利用するユーザ・アプリケーション・デバイスの範囲の限定、責任分界とアクセスポリシーやログ管理を明確にした認証・認可の実装（システム運用編 14.1.2）。

### 6.2 アクセス権限の管理（システム運用編 14.2、企画管理編 13.1.3）

- 情報の種別・重要性・利用形態に応じた区分管理を行い、情報区分ごと・利用者/利用者グループ（業務単位等）ごとに利用権限を規定。**付与する権限は必要最小限**。
- 「参照のみ」「作成権限あり」「更新権限あり」等、**参照・更新・実行・追加のきめ細かな権限設定**が可能なシステムの採用を推奨。医療従事者の資格・院内権限規程に応じた設定。
- 人事異動等による担当業務の変更等に応じたアクセス権限の見直し。
- クラウドサービスのデフォルト設定への自動変更（ポリシー改変）による意図せぬ権限変更・情報送信リスクへの検知措置。
- システム上の権限と院内ルール上の利用範囲が異なる場合は規程等で文書化し権限の範囲を明確化（企画管理編 13.1.3）→ UI 上の権限表示と運用ルールの整合が必要。

## 7. 監査証跡（監査ログ）

システム運用編 17.「証跡のレビュー・システム監査」遵守事項:

1. 利用者のアクセスについてアクセスログ等を記録し、**定期的にログを確認**。アクセスログは少なくとも「**利用者のログイン時刻、アクセス時間、及びログイン中に操作した医療情報**」が特定できるように記録。記録機能がない場合は業務日誌等で代替。
2. ログへのアクセス制限を行い、**ログの不当な削除／改ざん／追加等を防止**する対策。
3. ログ記録に用いる時刻は**信頼できる時刻源**を利用し、内部で同期、標準時刻と定期的に一致させ、診療事実の記録として問題のない精度を保つ。
4. 監査実施計画の作成・証跡の整理を行い企画管理者へ報告。

解説（17.1）:
- 個人情報を含む資源は**全てのアクセスログを収集**し、定期的に不正利用がないことを確認（全ログの目視ではなく、監視・閾値スクリーニング後の確認を想定）。
- アクセスログ自体に個人情報が含まれ得るため保護必須。
- アクセスログを分析し**緊急時にアラートを発する仕組み**も求められる。
- 管理委託時は事業者とログの管理方法・提供等を明確に取り決め（→ PH-OS の SLA/契約に監査ログ提供条項が必要）。
- 企画管理編 5.（安全管理におけるエビデンス）は、運用実態を証明する証跡の整備・レビュー・管理を組織要件として規定。

## 8. 非常時（BCP／サイバー攻撃時）運用

企画管理編 11.「非常時（災害、サイバー攻撃、システム障害）対応と BCP 策定」・12.「サイバーセキュリティ」:

- 非常時対応方針と対応手順を整理し **BCP を策定**（通常時への復旧計画を含め経営層承認）。非常時の定義の明確化（11 遵守事項①）。
- 非常時の3類型と必要対応（表11-1〜11-3）:
  - **災害**: フェールセーフ、冗長化した情報システム等への切り替え、安全な停止、リスクを踏まえた臨時措置（**認証方法の変更など**）、復旧・復帰手順、バックアップ整備（大規模災害を想定した遠隔保管を含む）。
  - **サイバー攻撃**: 被害状況・業務影響の把握、検知・遮断・隔離、**代替運用や手段への切り替え**、**証拠・証跡の保全**、原因究明、CSIRT 整備、**追記不能型のバックアップ**・ネットワーク構成分割、復旧計画。
  - **システム障害**: 障害状況の把握、代替運用への切り替え、冗長化（ホットスタンバイ／コールドスタンバイ、ネットワーク二重化）、復旧・改修計画。
- サイバーセキュリティ対応計画の策定・訓練・**少なくとも年1回以上の見直し**（12.2）。
- サイバー攻撃を受けた（疑い含む）場合で個人情報漏えいや医療提供体制に支障が生じるおそれがある場合、「医療機関等におけるサイバーセキュリティ対策の強化について」（平成30年医政局通知）に基づく**所管官庁への連絡**等（厚労省・都道府県警察等への報告を含む連絡体制の平時からの整備。12 遵守事項⑦、12.3）。
- 薬局向けには公式の **BCP 策定手引き・確認表（薬局用）** が提供されている（§1 の URL 表参照）。
- → オフライン対応 PWA（Dexie/IndexedDB）は「代替運用や手段への切り替え」「フェールセーフ」の実装として本章の要求に直接対応する設計要素。

## 9. UI に影響する要求の整理（PH-OS への含意）

企画管理編 13 遵守事項⑧は、**電子カルテにおける記録の確定に関して「入力者及び確定者の識別・認証」「記録の確定手順、識別情報の記録の保存」「更新履歴の保存」「代行入力を実施する場合、代行入力を認める業務、代行が許可される依頼者と実施者」を規程等に含める**ことを求め、システム運用編 14 遵守事項⑨がそのシステム機能要件を定める。調剤録・薬歴・訪問報告書等の「記録の確定」に読み替えて適用する。

| # | ガイドライン要求（出典） | UI への含意（PH-OS 設計指針） |
| --- | --- | --- |
| 1 | 確定された情報の登録時に、入力者及び確定者の氏名等の識別情報＋信頼できる時刻源を用いた作成日時を含む（システム運用編 14⑨(1)(a)） | 記録詳細・帳票に入力者/確定者/確定日時を常時表示。時刻はサーバ側の信頼時刻源で付与（クライアント時計に依存しない） |
| 2 | 「記録の確定」に当たり内容を十分に確認できる。確定は権限を持つ利用者に限定（14⑨(1)(b)(c)） | 確定前の内容確認ステップ（プレビュー/確認ダイアログ）。確定ボタンは権限保持者のみ活性。二重確認は既存の破壊的操作ダイアログ方針と整合 |
| 3 | 一定時間経過後の自動確定運用では入力者・確定者を特定するルールを規程に定める（14⑨(1)(e)）。確定者が確定操作できない場合の責任所在の明確化（(f)） | 自動確定の有無・タイミングを UI に明示（「未確定・○時間後に自動確定」等の状態表示）。確定代行時の責任者表示 |
| 4 | 確定後の更新は更新履歴を保存し、更新前後の内容を照合可能に。複数回更新でも順序性が識別できる（14⑨(3)(4)） | 版履歴 UI（誰が・いつ・何を変更したかの差分表示）。物理削除ではなく訂正履歴として表現 |
| 5 | **代行入力**は誰の代行が誰によって行われたかを都度記録。できるだけ速やかに確定者による「確定操作（承認）」を行い、内容確認なしの確定禁止（14⑨(5)(6)） | 代行入力モード（依頼者・実施者の明示）、未承認レコードの明確なバッジ表示、確定者への承認待ちキュー/通知。ワンクリック一括承認は「内容の確認を行わずに確定」に該当し得るため不可 |
| 6 | 虚偽入力・書換え・消去及び「混同」（患者取り違え・情報間の関連性の誤り）の防止（14.3、施行通知3条件） | 患者コンテキストの常時明示（PatientHeader の全画面共通表示は本要求に整合）。患者切替時の明確な視覚遷移、複数患者情報の同時表示時の混同防止 |
| 7 | 権限は必要最小限。参照/更新/実行/追加のきめ細かな権限、資格・規程に応じた設定（14.2、企画管理編13①④） | ロール/資格（薬剤師・事務等）に応じた画面/操作の出し分け。閲覧のみユーザーには編集 UI を非表示または無効化し、権限起因であることを認知できる表示 |
| 8 | 二要素認証（クライアント: アプリログイン時。令和9年度以降稼働の新規導入システムは原則必須）（システム運用編14⑤・表14-1） | ログイン UI に TOTP/FIDO2 等の第二要素ステップ（Cognito MFA と整合）。新規開発の PH-OS は経過措置に頼らず初期から必須化が妥当 |
| 9 | パスワード: 2FA 前提で8桁以上英数字混在、定期変更不要、失敗ロックアウト（14.1.1、14⑥） | パスワード設定 UI のポリシー表示・強度チェック。定期変更の強制はしない。連続失敗時のロックアウトと回復動線 |
| 10 | アクセスログ: ログイン時刻・アクセス時間・操作した医療情報の特定。ログの改ざん防止・分析・緊急アラート（システム運用編17） | 全 PHI アクセスの監査ログ記録（参照含む）。管理者向け監査ログ閲覧・検索画面、異常検知アラート表示。ログ閲覧自体も権限制御＋ログ対象 |
| 11 | 見読性: 直ちに明瞭かつ整然とした形式で表示・書面作成（施行通知3条件①） | 記録の即時表示性能と印刷/PDF 出力機能（帳票整形）。高コントラスト・可読性の既存 UI 方針と整合 |
| 12 | 非常時の代替運用・フェールセーフ・臨時措置（認証方法の変更等）（企画管理編11、表11-2/11-3） | オフライン PWA モードの明示的な状態表示（オンライン/オフライン/同期状態）、非常時閲覧モード、復帰時の同期・照合 UI |
| 13 | 患者等からの照会に対応する窓口整備（企画管理編1遵守事項⑦） | （運用寄り）問い合わせ導線の整備。外部共有機能の照会対応を想定 |

| 保守委託機関編への対応（事業者側の責任明記） | Applicable（PH-OS の商品設計・契約に直結） | 概説編2.1/3.1.5。SaaS 事業者がセキュリティアップデート責任を負うことを契約書・約款・SLA 等に記載した場合のみ顧客が軽量遵守ルートを選択可能 |
| 破壊的な認証方式変更・監査ログ設計変更 | Requires security review | careviax 運用ルール（auth/security は hard stop 領域）に加え、本ガイドラインの認証・証跡要求に直結するため |
