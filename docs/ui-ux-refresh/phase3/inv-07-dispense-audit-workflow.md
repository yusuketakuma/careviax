## このバケットで気づいた De facto パターン・逸脱・重複実装（Phase 5 監査の種）

### De facto パターン（事実上の標準）

- **page.tsx 標準形**: 全11ルートが `PageScaffold` + `metadata.title`（「— PH-OS」suffix）に準拠。client content を分離し、`useSearchParams`/初期状態が絡むものは `Suspense` + `Loading` fallback で包む（workflow / pharmacy-cooperation / tasks / search）。全画面マウント系は `variant="bare"`（workbench 4画面 / handoff / clerk-support / views）。
- **error boundary の共通ファクトリ**: colocated `error.tsx` は全て `createRouteErrorBoundary('XxxError')`（`src/components/ui/route-error-boundary.tsx`。PHI を telemetry に載せない設計・digest 表示・retry+ダッシュボード退避導線）。dispense/audit/set/set-audit/workflow/tasks/handoff/search が採用。
- **false-empty fail-close**: 取得失敗を空一覧/0件に潰さない方針がコメント付きで明文化されている — workbench（dispensing-workbench.tsx:351,414,432）、tasks（tasks-content.tsx:975-977 SegmentError）、workflow（workflow-dashboard-view.tsx:268 detail 文言）。memory の FE False-Empty パターンがこのバケットでは概ね徹底。
- **org スコープの定型**: `buildOrgHeaders(orgId)` + queryKey `[..., orgId]` が全 fetch 系で一貫（clerk-support:102、handoff、views、search、pharmacy-cooperation）。
- **URL query state の復元**: workflow / tasks は `read*State(searchParams)` モジュール（workflow-query-state.ts / tasks-query-state.ts）で server 側パース→initial props 注入という同型パターン（テスト付き）。search も page.tsx 内で同等の validation を行うが専用モジュール化していない（軽微な流儀差）。

### 逸脱・要監査ポイント

1. **/views の error 表示が SSOT 逸脱**: viewsQuery.isError が「取得できませんでした」の素テキストのみで retry 導線なし・`ErrorState` 不使用（saved-views-content.tsx:392-393）。同バケット他画面（clerk-support/tasks/handoff/workflow）は ErrorState/SegmentError + retry が標準。preferences 側の degrade 文言（:170-173）は意図的 graceful degrade の可能性があり要判定。
2. **E2E 空白ルート**: `/views` `/search` `/clerk-support` は tools/tests 配下に参照が 0 件（grep 確認）。特に search は 8 API 並列+部分失敗 UI という複雑度の割に E2E なし。
3. **状態バッジ実装の多流儀**: `StateBadge`（tasks）/ `STATUS_TOKENS`（workflow）/ 素の `Badge variant`（dispense-work-queue.shared.tsx の QueuePriorityBadge: emergency=destructive 等の独自マップ）/ workbench 独自 `--wb-*` 色 — 4 方式が同一バケット内に並存。clerk-support は T4（非6軸色混在）として台帳記録済み。
