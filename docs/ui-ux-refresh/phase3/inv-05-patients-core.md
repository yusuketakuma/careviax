## De facto パターン・逸脱・重複実装（Phase 5 監査の種）

- **確立パターン**: page.tsx は薄いサーバコンポーネント（PageScaffold + WorkflowPageIntro もしくは Suspense+Loading）、状態管理は colocated な `*-content.tsx` クライアントコンポーネントに集約。テストも page.test（構造）+ content.test（状態）+ `.shared.test`（純関数）の3層が標準形。
- **患者文脈ヘッダの3流儀混在**: PatientHeader（[id], safety-check）/ WorkflowPageIntro の戻る導線またはcontent内見出しのみ（edit, consent, share, mcs, residual-adjustment, medication-calendar, medications, prescriptions）/ bare（collaboration, compare）。同じ患者サブページ群で患者識別表示の方式が不統一（PatientHeader は SSOT のはずだが2画面のみ）。safety-check は header-summary API に乗らず root fetch している（メモリ記録とも整合）。redirect-only 2件とprint 3件は別縮尺の患者文脈契約が必要。
- **エラーUIの逸脱**: mcs-content.tsx:1127-1138 は共通 `ErrorState` を使わず独自 destructive ボックス。edit（patient-edit-content.tsx:228-239）は取得エラーと未存在を同一の `EmptyState` に畳み retry 導線なし — FE false-empty fail-close パターン（ErrorState+refetch）からの逸脱候補。
- **loading.tsx の aria-label コピペ疑い**: mcs/loading.tsx:6 が「服薬指導を読み込み中」（MCS 画面なのに別画面の文言）。
- **状態色 SSOT の未収束**: patients-board は STATUS_TOKENS 直参照、card-workspace は生 Badge で、StateBadge/StatusDot を使うのは consent のみ。UI_AUDIT_MATRIX T4 の既知課題と一致。
- **stale-refetch 保持の実装格差**: patients-board のみ「背景 refetch 失敗時に前回 data を保持し警告」を明示実装（patients-board.tsx:789-796）。他画面は初回エラー処理のみ。
- **colocated loading.tsx の有無がまちまち**: patients/, [id]/, consent/, share/, mcs/ に有り。new/compare/collaboration/safety-check/residual-adjustment は親境界依存（compare/collaboration/safety-check は Suspense fallback で代替）。error.tsx は patients/ 直下の1枚のみ。
- **E2E 空白ルート**: compare / safety-check / residual-adjustment / medication-calendar は tools/tests に該当 spec なし。特に safety-check は患者安全系でありながら E2E 未カバー。
- **内部印刷境界の未収束**: management-plan/print、medications/print、visit-records/print はdata readyだけで`window.print()`を自動発火し、UI/UX SSOTの「PHI出力は明示操作」契約に未適合。medication-calendarの印刷は明示buttonだが、calendar/medications/visit-recordsの印刷表現はpurpose-built safe display IDではなくraw route/DB patient IDを表示する。
- **監査台帳の stale 記述**: §141「patients/[id]/consent(Skeleton無)」は consent/loading.tsx 実在と矛盾。§94 の「段階展開が望ましい」も patient-form.tsx:358 で段階表示実装済み。
