## De facto パターン・逸脱・重複実装（Phase 5 監査の種）

- **ページ骨格の 4 流派が混在**:
  1. 標準形 `page.tsx(Server) = PageScaffold + AdminPageHeader + Suspense(Loading) + *Content`（analytics / audit-logs / jobs / metrics / uat）
  2. page.tsx 素通し・content 側が PageScaffold/AdminPageHeader を内包（data-explorer / operations-insights）
  3. `'use client'` の page.tsx がそのまま画面本体（Suspense 無し）（dispense-audit-stats / performance / realtime）
  4. Server page + content だが Suspense 無し（incidents）
  同じ admin バケット内で責務分割が不統一。performance/page.tsx は 35KB・realtime/page.tsx は 20KB の単一クライアントファイル。
- **loading.tsx 欠落 3 ルート**（dispense-audit-stats / incidents / operations-insights）。親 `(dashboard)/loading.tsx` 継承で機能はするが、他 8 ルートは骨格に合わせた専用 skeleton を持つため初期表示の質が非対称。colocated error.tsx は全 11 ルートで無し（親 `(dashboard)/error.tsx` 依存で統一されている点は一貫）。
- **KPI/統計カードの重複実装**: performance の `KpiCard`（page.tsx:551）、metrics のローカルカード、analytics のローカル指標カード、realtime の手書き KPI div、一方 operations-insights のみ共通 `@/components/ui/stat-card` を使用。共通 StatCard への集約余地が大きい。
- **false-zero/false-empty fail-close は概ね普及**（ErrorState/SegmentError + retry、realtime は '—' 表示、コメントで意図明記）だが、**performance 最上段「今すぐ見る要対応シグナル」帯だけは error/loading 中も `?? 0` 描画で false-zero が残存**（page.tsx:736-773。承認待ち/緊急影響が「0=問題なし」に見える偽 all-clear リスク。realtime は同型の帯を '—' 化済みなので画面間で非対称）。
- **UI_AUDIT_MATRIX の stale 記載**: §3「dispense-audit-stats(error無)」「performance(isError未描画)」は現コードでほぼ解消済み（前者は ErrorState 実装済み、後者も大部分に ErrorState あり）。台帳更新が必要。
- **44px タッチターゲットの実現手法がバラバラ**: `!h-11 !min-h-[44px]` 直付け（dispense-audit-stats/jobs 行ボタン）、`sm:h-11 sm:min-h-[44px]`（audit-logs、institutions test-locked 契約に準拠）、ルート/ラッパーからの CSS 子孫一括上書き `[&_button]:!min-h-[44px]`（jobs:382）・`[&_button]:!h-11 [&_input]:!h-11`（performance:1067 の StaffKpiPanel ラッパー、analytics:265）。Button variant 契約（coarse44px encode）で吸収すべき箇所が force override で散在。
- **折りたたみ UI にネイティブ `<details>/<summary>` を採用**（audit-logs フィルタ、jobs ヘルプ）— shadcn Accordion 不使用の de facto パターン。marker 非表示ハックが両所で重複。
- **自動更新の実装 2 系統**: `useRealtimeQuery`(SSE + fallbackRefetchInterval)（realtime / performance）と素の `refetchInterval` ポーリング（jobs / performance の runtime クエリ）。ジョブ監視こそ SSE 恩恵が大きい候補。
- **チャート系列色は `--chart-*` トークン + ファイルローカル色マップ**（dispense-audit-stats `CODE_COLORS`、operations-insights `BAR_COLORS`/`PROCESS_COLORS`）。「状態色ではなくデータ可視化なので chart トークン」というコメント規約が 2 ファイルで独立に記述されており、規約の SSOT 文書化候補。
- **権限モデル**: /admin 配下なのに画面・API とも管理者 capability ゲートが（確認した範囲で）無い。org-wide アクセスの意図的仕様だが、「admin という URL セグメント ⇔ 実際は全メンバー可視」のギャップは監査観点で明文化が必要。incidents のステータス変更 canAdmin ゲートが唯一の画面内 permission-denied 分岐。
- **jobs の再実行は確認ダイアログ無しの即時 mutation**（toast のみ）。破壊的ではないが再実行コスト次第では確認 UX の検討余地。
