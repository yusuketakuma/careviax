## このバケットで気づいた De facto パターン・逸脱・重複実装(Phase 5 監査の種)

### De facto パターン(慣行として定着)

- **admin 標準骨格**: `page.tsx`(server, Metadata) → `PageScaffold` + `AdminPageHeader`(title/description/shortcuts, `data-page-header`) + `Suspense fallback=<Loading label=…>` + `*-content.tsx`('use client')。14 admin ルート中 11 が準拠(src/app/(dashboard)/admin/*/page.tsx)。
- **エラー境界の2層**: セグメント共通 `admin/error.tsx`(createRouteErrorBoundary)+ 画面内はクエリ別 SegmentError/ErrorState(variant=server, onRetry)。false-empty 対策コメントが vehicles-content L587 に明文化。
- **状態色**: StateBadge/StatusDot の 6軸 role(readonly/confirm/done/blocked/hazard/alert)が広く採用(external-professionals x23, vehicles x17, master-hub の StatusRole マップ)。
- **破壊的操作**: 削除/セッション終了は `ConfirmDialog`(business-holidays, service-areas, vehicles, shifts, notification-settings, break-glass)。
- **platform 系**: 独立 layout で PlatformOperator ゲート(`unauthorized()`/`forbidden()`)、AppShell 非再利用+常設 role="alert" 監査バナー、という意図設計(platform/layout.tsx, platform-shell.tsx のコメント)。

### 逸脱

1. **platform 配下に loading.tsx / error.tsx が一切ない**(src/app/platform/ 配下)。初回遷移のフォールバックと render エラー境界が root 依存。admin 側の2層構えと非対称。
2. **`/admin/service-areas` は page.tsx 単体の 'use client' 巨大実装**(content 分離・server page なし、Metadata なし)。admin 標準骨格から逸脱(src/app/(dashboard)/admin/service-areas/page.tsx)。
3. **`/admin/notification-settings` は Suspense fallback なし・PageScaffold/AdminPageHeader を content 側が内包**(page.tsx は content を直 render)。骨格の責務配置が他と逆。加えて UI_AUDIT_MATRIX T4(L122)で状態色の非6軸混在の個別指摘あり。
4. **`/admin/operating-hours` のみ colocated loading.tsx なし**(親 admin/loading.tsx 継承)。他 13 admin ルートは保有。
5. **`/admin/professionals` は redirect alias なのに loading.tsx が同居**(死にファイル)。
6. **クライアント側 capability ゲートの不統一**: pharmacy-sites だけ `useAuthStore`+`hasPermission('canAdmin')` で編集 UI を出し分け(content L169-170)。users/settings/shifts/vehicles 等は API 403+toast 依存で、権限なしユーザーにも操作 UI が見える。
7. **users-content の「停止」行アクションに確認ダイアログの形跡なし**(ConfirmDialog/AlertDialog import なし、Dialog は招待用)。破壊的操作の確認ダイアログ必須方針(CLAUDE.md)との突合が必要 — 未確認(実挙動はテスト/実機で要検証)。
8. **E2E の stale selector**: tools/tests/helpers/design-screen-map.ts が待機する `data-testid`(`staff-master-editor` / `vehicle-master-editor` / `external-professionals-master-editor`)は src に存在しない(grep 0件)。ui-design-fidelity の該当エントリはタイムアウト前提の死んだ定義の可能性。
9. **E2E カバレッジの薄さ**: 本バケット 16 ルート中、spec に route が現れるのは `/admin`(ハブ)と design-screen-map 経由の staff/vehicles/external-professionals のみ。platform 2 ルート・users・settings・shifts 等はゼロ。
10. **エラー表示部品の混在**: SegmentError(settings, service-areas, pharmacist-credentials, master-hub)と ErrorState(operating-hours, pharmacy-sites, notification-settings, shifts, vehicles KPI, platform 系)が画面ごとに揺れる。使い分け基準の明文化候補。

### 重複実装・共有

- **`/admin/staff` と `/admin/users` は同一 UsersContent を表示**(staff は StaffKpiPanel を上に足すだけ)。1画面2ルートの実質重複。ナビ導線と役割の整理候補(src/app/(dashboard)/admin/staff/page.tsx が `../users/users-content` を import)。
- **`/platform/tenants/[orgId]` に専用 detail API なし**: ディレクトリ一覧クエリ(`/api/platform/tenants`, 同一 queryKey)を再利用しクライアント側 `find()` で対象テナント抽出、不在時 ErrorState not-found(tenant-detail-content.tsx L17-54)。テナント数増加時の設計論点。
- **tenant-detail-content 自体のユニットテストなし**(パネル3種のみテストあり)。
