## De facto パターン・逸脱・重複実装（Phase 5 監査の種）

### 構成パターンと逸脱
- **標準形**: server `page.tsx`（PageScaffold + AdminPageHeader + Suspense fallback Loading）+ client `*-content.tsx` + colocated `loading.tsx`。contact-profiles / facilities / facility-standards / institutions / packaging-methods / pca-pumps / pharmacy-cooperation が準拠。
- **逸脱1（クライアントモノリス）**: alert-rules（612行）と billing-rules（649行）は page.tsx 自体が `'use client'` でヘッダ込み全実装。Suspense 分割なし（UI_AUDIT_MATRIX.md:140 の「alert-rules(Suspense未包)」指摘と一致）。
- **逸脱2（ヘッダの持ち場所）**: capacity / inventory-forecast は PageScaffold(variant="bare") + AdminPageHeader を **content コンポーネント側** に持つ（SYS-3 統一コメント付き）。他は page 側。ヘッダ責務の置き場が 2 流派。
- **巨大共用 content**: drug-master-content.tsx（2,506行）が /admin/drug-masters と /admin/formulary の 2 ルートを variant prop で兼務。UI_AUDIT_MATRIX.md:131 の見出し階層・行数指摘が現存。pca-pumps（1,585行）、pharmacy-cooperation（1,678行）、facilities（1,365行）も単一ファイル大型。

### 状態表示の重複実装（同一ポリシーに4系統）
- false-empty/false-judgement 回避というポリシーは全画面共通（コメントでも明文化）だが、実装が分裂:
  1. `SegmentError`/`SegmentLoading`（alert-rules, contact-profiles, document-templates, packaging-methods, pharmacy-cooperation）
  2. `ErrorState`（capacity, drug-masters, facility-standards, inventory-forecast, pca-pumps）
  3. `DataTable` 内蔵 errorMessage/onRetry/emptyMessage（billing-rules, facilities, institutions, pca-pumps, document-templates）
  4. `Alert variant="destructive"`（facilities は DataTable errorMessage と**二重表示**）
- 空状態も 3 流派: `EmptyState` コンポーネント（pharmacy-cooperation のみ）/ DataTable emptyMessage / 素の `<p>` 文言（alert-rules, contact-profiles）。
- loading 判定も `isLoading` 派と `isPending` 派（packaging-methods, signal-tuning-panel は「orgId 未解決 = disabled query を loading 扱い」の理由コメント付きで isPending）。この根拠コメントは良い先例だが横展開されていない。

### loading.tsx の穴と文言ドリフト
- loading.tsx 欠落: capacity / inventory-forecast / packaging-methods / pharmacy-cooperation の 4 ルート（前 2 者はコンポーネント内 skeleton で代替、後 2 者は Suspense fallback のみ）。
- 存在する loading.tsx は全て同一 Skeleton+SkeletonRows テンプレのコピペ（内容は良いが共通化されていない）。contact-profiles の loading aria-label「連絡先プロファイル」が画面名「連携先プロファイル」とズレ。
- colocated error.tsx は 0/14。

### 確認ダイアログの分裂
- 削除確認: 共通 `ConfirmDialog`（alert-rules, document-templates, drug-masters, facilities, institutions）に対し、billing-rules だけ Radix `AlertDialog` プリミティブ直組み。
- 破壊的でない運用停止（packaging-methods の is_active、pca-pumps のステータス変更）は確認なし — ポリシーとして明文化されていない。

### 権限表示の非対称
- institutions（および隣接の pharmacy-sites / incidents）は「非管理者に常時 403 になるボタンを出さない」FE ゲートを実装済みだが、**同種の書き込み UI を持つ facilities / billing-rules / document-templates / pca-pumps / pharmacy-cooperation 等は全員に表示**。同じ admin バケット内で permission-denied の扱いが不統一（Phase 5 で API 側ガードとの突合要）。

### 44px タッチターゲットの記法分裂
- 一括強制セレクタ `[&_button]:!h-11 [&_input]:!min-h-[44px]`（alert-rules, pharmacy-cooperation）/ 個別 `!h-11 !min-h-[44px]`（facilities, institutions）/ `min-h-11 sm:h-11`（billing-rules, facility-standards）/ `h-11 sm:h-11 sm:min-h-[44px]`（pca-pumps）。sm: 付きは意図的契約（Button variant contract）だが 4 記法混在は監査コスト要因。

### その他の先例・良パターン
