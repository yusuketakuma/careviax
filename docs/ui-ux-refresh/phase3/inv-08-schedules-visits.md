## De facto パターン・逸脱・重複実装（Phase 5 監査の種。ここでは修正しない）

- **De facto 標準パターン**: `page.tsx`（server, searchParams 解決）→ PageScaffold（一覧/没入系は `variant="bare"`）→ client content（TanStack Query, isLoading→Skeleton / isError→ErrorState+onRetry / empty→EmptyState / mutation confirm→ConfirmDialog or AlertDialog）。本バケット14ルート中11ルートがこの形に収束している。
- **PageScaffold 逸脱 2件**: `visits/[id]/capture/page.tsx`（意図的: 最小シェル没入 UI、layout.tsx:99 に専用分岐あり）と `visits/[id]/facility-packet/page.tsx`（意図不明: PageScaffold も WorkflowPageIntro もなく content 直 return。同じ「訪問中サブ画面」の brief/voice-memo は bare PageScaffold を使っており不整合）。
- **確認ダイアログの二重実装**: 同系操作で ConfirmDialog（team-board, emergency-route, route-compare）と AlertDialog 直使用（proposals, weekly-optimizer）が併存。確認 UI の SSOT が割れている。
- **エラー提示の3流派**: ①ErrorState+retry（大半）②SegmentError/SegmentStaleBanner+useStaleAfterRefetchError（visits-today のみ。stale-after-refetch まで扱える上位互換）③toast のみ（voice-memo。取得系がなく録音デバイス系エラーだが、ErrorState 系を一切使わない唯一の画面）。stale-refetch 提示は visits-today が最も整備されており、他画面へ横展開されていない。
- **患者安全ヘッダの実装分裂**: 患者文脈画面で PatientHeader コンポーネント使用は visits/[id] のみ。capture/record は `PatientHeaderSummary` 型＋`VisitHeaderSafetyTags`（record/visit-step-nav 由来）で独自バナーを再実装。brief は3カラム内独自表示。MEMORY の「PatientHeader は患者識別 SSOT」方針に対し、訪問中モバイル系は別系統が事実上の標準になっている。
- **サーバ側権限 gate の非対称**: capture だけが page.tsx で auth+canVisit+assignment scope+PHI read audit を実装（Wave1 対応）。同じく PHI を出す brief/record/facility-packet/voice-memo の page.tsx には同等の SSR gate がない（API 側で担保かは未確認）。my-day は client 側 canAdmin gate のみ。
