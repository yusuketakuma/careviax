# High ROI Functional Proposals

作成日: 2026-06-18

対象は、現行 CareViaX の実機能に直接つながる改善だけに絞る。新規の大規模 redesign ではなく、既存 API / BFF / UI / service / test の接続を強める前提。

| #   | 領域           | 提案                                                                                      | 根拠となる現行機能                                                                                | 最小実装スライス                                                                                      |
| --- | -------------- | ----------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| 1   | 訪問準備       | 訪問 ready 判定を 1 つの server helper に統一                                             | `visit-preparation-readiness.ts`, visit-preparation GET/PUT, day-board, schedule PATCH ready gate | `buildVisitReadyContext` を export し、day-board / prep pack / ready PATCH のテストを同じ期待へ寄せる |
| 2   | 訪問準備       | 「準備保存」と「ready 化」を atomic action にする                                         | `schedule-day-preparation.ts` は保存後に別 PATCH で ready 化                                      | PUT に `mark_ready` を追加し、ready 失敗時に成功 toast が出ないようにする                             |
| 3   | 訪問準備       | 準備 pack の過去 contact log を PHI 最小表示にする                                        | `visit-preparations/[scheduleId]/route.ts` は contact name / phone / note を扱う                  | method / outcome / due / has_note のみ返し、詳細は権限付き detail 取得へ分離                          |
| 4   | 日程ボード     | 同姓同名患者の識別 chip を day-board / 提案カードへ追加                                   | day-board DTO は `patient_name` 中心                                                              | kana / masked DOB / 施設 unit の compact chip を追加                                                  |
| 5   | 日程提案       | `change_requested` から focused reschedule task を起こす                                  | contact result は `reschedule_pending` に残るが task 化は未整備                                   | contact result 記録時に再提案 task を upsert し、候補詳細へ deep link                                 |
| 6   | 日程変更       | reschedule API の date-key を厳格化                                                       | reschedule route の日付境界が route ごとに散る                                                    | 既存 `visitScheduleDateKeySchema` / strict date helper を再利用                                       |
| 7   | ルート確定     | route confirmation に OCC / freshness を入れる                                            | route snapshot と vehicle assignment が stale でも通る余地                                        | prep GET に schedule version を返し、route confirm PUT で expected version を要求                     |
| 8   | 日程取消       | 日程 cancel 時に関連 task / proposal を掃除する                                           | schedule DELETE は schedule/audit 中心                                                            | visit_preparation / contact_followup task 解決、open proposal supersede を同 transaction へ           |
| 9   | 訪問記録       | handoff extraction を fire-and-forget から retryable task へ                              | visit-record 作成後の extraction 失敗が console warn 止まり                                       | extraction failure を task / status に保存し、retry endpoint を追加                                   |
| 10  | 訪問記録       | next-visit suggestion から日程 proposal を直接生成                                        | visit record は follow-up task を作るが proposal draft には直結しない                             | task action で `source_visit_record_id` 付き proposal 生成                                            |
| 11  | 残薬           | 残薬 issue の dedupe key を薬品コード / 単位込みにする                                    | drug name 中心 dedupe は同名・規格違いに弱い                                                      | residual schema と issue metadata に drug_code / unit を追加                                          |
| 12  | 処方差分       | 訪問準備の処方差分に line id / code / date delta を含める                                 | prep pack は display name 中心                                                                    | DTO に evidence fields を追加し、UI は compact 表示のまま保持                                         |
| 13  | 日程ボード     | schedule operational task count を当日関連に scope する                                   | board 上の task count が org-wide open task に寄りやすい                                          | due-today / overdue / schedule-related group count を追加                                             |
| 14  | 施設訪問       | facility parallel context を redacted packet summary と full packet に分ける              | prep pack が同施設文脈を広く扱う                                                                  | redacted summary DTO を day-board、full detail を facility-packet route に限定                        |
| 15  | ブラウザ検証   | 訪問準備 -> ready -> contact follow-up -> 再提案の route-mocked E2E を追加                | 単体テストは増えたがブラウザ proof が薄い                                                         | Playwright route-mock で schedule board の実導線を固定                                                |
| 16  | 調剤UI         | workbench の title/menu 旧 chrome を撤去し、PhaseTabs と前回処方/新規グループ作成だけ残す | `dispensing-workbench.tsx` にタイトル専用 bar と span menu が残る                                 | 旧 titleBar/menuBar を削除し、主操作を既存 toolbar へ再配置                                           |

## 先に実装したい順

1. 患者詳細・訪問準備・日程ボードにまたがる安全/請求 blocker の接続強化。
2. PHI が一覧 cache に広く乗る箇所の projection 縮小。
3. contact / report / external self-report の idempotency と partial failure 対策。
4. ready / review / close などの状態遷移に OCC を追加。
5. route-mocked E2E で、訪問準備から報告・算定までの実導線を固定。
