### 明示的な非ゴール（yrese業務Domain / 周辺システムの責務）

- フル在庫管理（発注・仕入・棚卸し・在庫評価）→ PH-OSは在庫医薬品マスタ（採用薬フラグ+引当フラグ）の薄い層のみ
- 麻薬管理帳簿・毒薬劇薬受払い簿 → yreseまたは批准済み法定帳票システムが担う
- 領収書・調剤報酬明細書の発行 → yreseの中核機能（二重入力回避）
- 会計・一部負担金の収納管理 → yrese / 会計システム
- POS・仕入・発注 → 在庫管理専用システム

### 実装優先原則（今回レビュー反映）

- MVPは「訪問日次運用 + 報告送付 + 最低限の処方差分/持参判定」を最優先にし、重いマスタ/処方安全チェック/請求自動化は後段に寄せる
- `MedicationCycle` は「処方起点の1運用サイクル」を維持する。MVPでも訪問予定は処方差分・持参可否・未解決課題と切り離さない
- PH-OS / yrese / 電子薬歴 / 在宅支援システムの責任分界を先に固定し、二重入力を避ける
- 公開情報ベースの市場比較では、既存製品は「訪問記録・計画書/報告書作成・FAX/メール送付・現場共有」に強い。初期価値は最適化機能より、現場記録/連携/持参漏れ防止に置く

### yrese・PH-OS FHIR / JP Core Native 相互運用・完全置換計画 v0.5（2026-07-15）

#### 結論と固定境界

- yrese・PH-OSの患者・処方・調剤・在宅訪問・服薬実態の正本は、**FHIR Resourceそのもの**とする。FHIR / JP Coreは外部出力形式や既存独自Clinical Modelのprojectionではなく、両製品のCanonical Clinical Coreである。
- 採用版は **FHIR R4 4.0.1 + JP Core 1.2.0** にpackage lockする。JP Core公式履歴で1.2.0がcurrent、1.3.0-devは開発版であるため、開発版を本番適合基準にしない。Phase 0開始時と各release前に公開版を再確認する。
- yreseは処方・調剤のAuthoritative FHIR Server、PH-OSは在宅訪問・服薬実態のAuthoritative FHIR Serverとする。双方は相手所有Resourceをread-only replicaとして保持できるが、同じlogical Resourceを双方が更新するmulti-masterにはしない。
- `yrese -> PH-OS`はPatient / Coverage / Medication / MedicationRequest / MedicationDispense等、`PH-OS -> yrese`はEncounter / MedicationStatement / Observation / DetectedIssue candidate / CarePlan / Task / Communication / DocumentReference等をFHIR REST / transaction Bundle / Subscription / historyで同期する。受信通知だけを正本や同期完了証跡にしない。
- JAHIS、NSIPS、電子処方箋Official IF、資格確認、PMHはAdapter Planeへ隔離する。外部CSV、文字コード、共有folder、record順、official transport状態をFHIR Clinical Data Planeへ漏らさない。JAHIS電子版お薬手帳QRからMedicationRequestを生成・確定しない。
- 調剤報酬算定、Calculation Trace、一部負担金、公費按分、会計、未収、領収証、電子レセプト、月次締め、返戻・再請求はyrese固有Domainに残す。ただし全entityがPatient / MedicationRequest / MedicationDispense等のFHIR Referenceを持つ。
- **後方互換性は提供しない。** 旧Clinical Model、旧臨床API、独自clinical DTO、runtime alias、dual-write、dual-read、legacy fallback、source別canaryは作らない。旧データは検証可能な一方向変換でFHIR Resourceへ移行し、書込停止後の最終差分変換と全件照合を通して単一releaseでハードカットオーバーする。
- write再開前のabortは旧runtimeとの互換層ではなく、release全体とDB / Resource history / object・raw vault / audit chain / IG・terminology package / key availability / adapter cursor・manifestを含むfull recovery-set snapshotの整合した復元で行う。write再開を不可逆commit pointとし、それ以後はFHIR-native recovery set + append-only Resource history / accepted-write journal / ingress journal replayまたはforward-fixだけで回復し、旧releaseへrollbackしない。production migration、書込停止、snapshot取得、cutover、旧table/drop、外部送信enableはそれぞれhuman gateとし、dry-run / restore rehearsal / 件数・hash・臨床重要field・reference照合が揃うまで実行しない。

Program umbrellaは`FHIR-NATIVE-INTEROPERABILITY-V05-001`とする。実装入口は`FHIR-NATIVE-P0-FOUNDATION-001`→`FHIR-NATIVE-PHOS-SERVER-001` / `FHIR-NATIVE-CONFORMANCE-001`→`FHIR-NATIVE-ADAPTER-PLANE-001` / `FHIR-NATIVE-YRESE-SYNC-001`→`FHIR-NATIVE-LEGACY-MIGRATION-001`→`FHIR-NATIVE-CUTOVER-001`→`FHIR-NATIVE-UI-DOGFOOD-001` / `FHIR-NATIVE-OFFLINE-EDGE-001`→`FHIR-NATIVE-OPEN-ECOSYSTEM-001`の10 sliceとする。v0.2〜v0.4の旧実装計画はv0.5で置換し、実装契約として参照しない。

`source_type`と`trust_level`は自由な2 enumではなく、次の合法組合せmatrixをserver-side SSOTとする。`trust_level`は取込時点の由来を表すimmutable evidenceであり、後日の薬剤師確認で書き換えない。薬剤師確認、原本照合、FHIR validation、claimability、yrese commitは別のversioned review/processing stateにする。

`RawIngressEnvelope`はAdapter / Technical Control PlaneのartifactでありClinical Coreではない。最低限`ingress_id / tenant_id / pharmacy_id / source_type / trust_level / transport_type / payload_format / character_encoding / specification_name+version / adapter_id+version / mapping_version / raw_payload_hash / raw_payload_storage_ref / received_at+by / device_id / idempotency_key / consent_ref / processing_status`を持つ。`raw_payload_storage_ref`はopaque artifact IDだけとし、内部URL、presigned URL、payload本文をResource、queue metadata、logへ複製しない。

| Direction         | FHIR Data Plane                                                                                                                                                             | Owner / receiver behavior                                                                                                                                                            |

Technical Control Planeのdelivery recordは`delivery_id / subscription_id / origin_system / tenant+pharmacy / correlation_id / resource type+id+version+hash / purpose / consent reference / cursor / retry state`だけを保持する。FHIR Resource / Bundle本文、Patient名、薬剤名を独自event envelopeへコピーしない。配送順は保証を前提にせずauthoritative versionで判定し、duplicateは同じ結果、out-of-orderはhistory pull、同version異hashはconflictへ送る。

FHIR R4の`meta.versionId`、ETag、`If-Match`をversion-aware updateへ利用する。push通知の欠落・delete・条件外更新を補うため、resource historyまたは批准済みdelta cursorによる定期reconciliationを持つ。訂正、取消、entered-in-error、削除相当の表現はresource別にPhase 0で固定し、物理deleteや「最新timestamp勝ち」で消さない。

#### canonical ingress / sync data flow

```text
JAHIS / NSIPS / Official IF / Scanner / File
  -> transport preflight + RawIngressEnvelope
  -> versioned Adapter Registry / Anti-Corruption Layer
  -> syntax / terminology / patient identity / duplicate validation
  -> FHIR Resource candidate + DocumentReference + Provenance

FHIR Client / yrese FHIR Server / PH-OS FHIR Server
  -> OAuth2/OIDC + tenant/purpose/consent/authz
  -> FHIR REST / transaction Bundle
  -> pinned package/profile/terminology/reference validation
  -> quarantine/review OR authoritative immutable Resource Version commit
  -> current head + rebuildable Search Projection + local Provenance/AuditEvent
  -> Subscription notification with Resource reference
  -> receiver read/vread/history pull + read-only replica commit
  -> periodic history/delta reconciliation
```

Patient matchingはstable yrese patient identifier、official prescription identifier、source-system namespaceを優先し、氏名/生年月日/性別だけの一致を自動linkに使わない。mismatch、missing identifier、複数candidateはfail-closedで、Resource保存前またはquarantine内に留める。

#### validation、性能、運用contract

- ingress同期gateはpayload byte/resource count、JSON/resourceType、tenant/source identifier、signature/hash、profile allowlist、patient mismatch等の安価なpreflightに限定する。full profile/terminology/reference validationはcontent hash + FHIR package version + terminology snapshotで結果をcacheし、bounded workerで実行する。
- full validationは高コストかつ不完全なhistorical dataを拒否し得るため、受信した臨床情報を捨てない。`valid / invalid / unsupported_profile / terminology_review / identity_review / reference_unresolved`をquarantineで保持し、患者安全上の実行を止めつつreview/replay可能にする。
- 検索は暗号化canonical payloadの全scanではなく、批准済みSearchParameterのtenant-scoped projection indexを使う。patient、identifier、date、status、category、code、subject、based-on、part-of、performer、organization、encounterをresource別に定義し、`_include`/`_revinclude`/chainはquery budgetと許可listなしに公開しない。
- Bundleはbytes、entry数、reference depth、contained size、attachment size、timeoutを明示上限化する。transaction Bundleの原子性を用途ごとに定義し、巨大Bundleを一つのDB transaction/worker leaseで処理しない。
- QR/file inputはuntrustedとして、総byte/分割数/行数/列数/field長/文字コード/制御文字を上限化し、parser timeoutとresource budgetを持たせる。CSV式prefixは表計算exportでneutralizeし、adapterを単なるinterface呼出しだけで「sandbox済み」と表現しない。worker/process/WASM等の分離方式と失敗境界はPhase 1で批准する。
- SLO/metricsはingress accepted/quarantined/rejected、validation latency/cache hit、inbound/outbound queue lag/retry/dead-letter、ack stage/lag、reconciliation drift、echo suppression、unresolved references、patient match review、ConceptMap miss、adapter/version別failure、read source、rollback flagをtenant/source/resource単位でPHIなしに持つ。
- v0.3で提示された`QR decode 300ms / JAHIS parse 300ms / FHIR mapping 500ms / validation 500ms / 通常取込全体1.5s / duplicate search 300ms`は**provisional laboratory budget**としてのみ継承する。実機scanner、低spec薬局端末、分割QR、LOCAL_ONLY、cold/warm cache、payload分布を固定して測定するまでrelease SLOや同期API timeoutへハードコードしない。
- 必須test matrixはofficial example + synthetic boundary fixture、JAHIS両familyの識別/byte/record/split round-trip、電子お薬手帳からMedicationRequestを作らないnegative、source×trust legality、RawIngress lifecycle、FHIR package/IG/terminology validator、ConceptMap、FHIR REST/transaction/history/Subscription、双方向version/loop suppression/reconciliation/correction/cancellation、RLS/cross-tenant、role/qualification/purpose/Consent、one-way conversion/full cutover/full snapshot restore、offline expiry/revoke/conflict、CapabilityStatement contractとする。
- validator fixtureにはFHIR primitive / `_element`配列のnull placeholder、JP Core Patientの正確なcardinality、OperationOutcomeのseverity/code/expression、validator/package fingerprintを含める。local preflightやcallerのrequested statusは最終`valid`を付与できない。

#### 明示的な非ゴール / 停止条件

- FHIRを電子処方箋管理サービス、JAHIS、NSIPS、レセプト、会計、請求の代替仕様にしない。PH-OSから電子処方箋Official IFへ非公式接続しない。
- [JAMI FHIRポータル](https://std.jpfhir.jp/)の処方情報FHIR仕様やJP Core profileを、電子処方箋管理サービスのONS transport/record-condition/署名仕様そのものと扱わない。
- `PrescriptionIntake`を一括renameしてFHIR-nativeと呼ばない。canonical Resource/version、標準validator、profile/terminology、CapabilityStatement、round-trip/negative testが揃う前に準拠を表示しない。
- 旧Clinical Model、独自臨床API、独自clinical DTO、compatibility alias、dual-write、dual-read、legacy fallbackを残さない。移行中の検証用read-only snapshot比較は許可するが、production requestを旧経路へ流す互換動作は作らない。
- Control PlaneへFHIR Resource / Bundle本文や臨床情報を複製しない。臨床payloadを独自event/queue DTOだけで同期し、FHIR APIを装飾にする実装は停止する。
- JAHIS/NSIPS仕様を無許諾でコピー・推測・reverse engineeringしない。NSIPSの利用主体・scope・加入/審査が未承認ならadapter実装を停止する。
- 患者持参のお薬手帳情報、AI生成、OCR、曖昧コード変換、demographic-only matchを確定MedicationRequest/MedicationDispenseへ自動昇格しない。
- 双方向同期を理由にyrese-owned ResourceとPH-OS-owned Resourceを同じfield-level multi-masterへしない。origin/source eventなしの再送、acceptedだけのSYNCED表示、timestamp last-write-wins、自動上書き、訂正/取消のsilent deleteは禁止する。
- `source_type`と`trust_level`をclient自由入力にせず、薬剤師確認済み状態をtrust levelの上書きで表現しない。Provenance targetはversion-specificに追跡し、raw payload hash、adapter/spec/mapping versionの欠落時はcommitを停止する。
- CareTeamのauthoritative owner、profile、identifier namespace、replica directionが未批准のままcurrent head、search、write、同期、CapabilityStatementを実装・宣言しない。
- exact source commit/dirty manifest、FHIR/JP Core/IG/terminology/validator package digest、validation authorityが未固定ならschema/API実装へ進まない。
- migration profileで親FHIR/JP Core profileの制約を緩和しない。適用不能なdata-absent-reason、誤ったentered-in-error、必須値の捏造、exceptionをJP Core準拠として扱う変換は停止する。
- DocumentReference/Binary owner、object access/retention/key、FHIR R4 `Attachment.hash`のSHA-1と独立manifest SHA-256が未定義なら添付移行・公開を停止する。
- cutoverは全ingressだけでなく全clinical outbound/export/Subscription deliveryもfenceする。signed cutover/recovery manifest、adapter ack/replay、accepted-write journalがなく、write-unfreeze後のacknowledged clinical write RPO=0（または定量化RPOと患者安全の明示承認）を証明できない場合はfreezeを解除しない。
- schema/DML apply、production conversion、write freeze、external send、eRx接続、NSIPS申請/実装、Bulk Export、offline PHI持出し、cutover、legacy dropはそれぞれhuman gateとする。後方互換性不要という決定はproduction mutation/deploy/dropの承認を省略するものではない。

**Account model v1 — authoritative role SSOT（2026-07-15）**:

- 公開ロールは固定5種: global `owner`、global `supporter`、tenant `admin`、tenant `pharmacist`、tenant `clerk`。tenant内の管理責任上の序列は `admin > pharmacist > clerk` だが、これは全actionの包含関係ではない。専門actionはrole順位で自動許可せず、action classificationと資格を別々に評価する。legacy enum名を公開ロールとして延命しない。
- `owner` は全tenant/system/accountと監査記録の統括・閲覧を扱える。薬歴記載・報告書作成・調剤監査/set監査等の薬剤師専用actionを実行する場合だけは、明示`target_org`内のaction-capable classificationと下記canonical薬剤師資格gateを別途必須とする。tenant PHIへ入る各requestは`target_org`へRLS pinし、purpose、step-up、read/write auditを残す。BYPASSRLSやunscoped queryを「full access」の意味にしない。
- `supporter` は明示assignmentされたtenantだけread-only。通常supportと30分MFA break-glassは別modeとするが、どちらでもsupporter identityはread-onlyのままであり、emergency/session modeからread-writeへ昇格しない。assignmentなし閲覧、write、export、external share、secret accessを許可しない。
- tenant actionはmethod/purpose単位で分類する。v1で薬剤師専用とする実行クラスは薬歴記載・報告書作成・監査実行で、action-capable account classificationはtenant `admin` / `pharmacist`（global `owner`は明示target tenant文脈のみ）。それ以外の非薬剤師運用actionは`admin + pharmacist + clerk`。`clerk`は権限内tenantデータと薬剤師業務情報を全件readできるが、上記3クラスは実行不可。read、govern/oversight、assign、notification recipient、external outputは実行capabilityから分ける。
- role capabilityと法的資格を分離する。上記3クラスの実行は、action-capable account classification **AND** actor本人のcanonical薬剤師資格がverified/current **AND** target tenant・assignment・purpose・職務分離条件を満たす場合だけ許可する。admin/global ownerも資格未確認なら不可、pharmacistも資格欠落・未検証・期限切れ・停止・取消なら不可、clerk/supporterは資格情報が存在しても不整合provisioningとしてfail-closedにしrole修正を要求する。role名、legacy claim、任意credential文字列を資格証明に使わず、単一のmutable booleanだけを法的証跡にしない。
- identityはglobal principal、tenant role assignment、site assignment、supporter tenant assignment、canonical qualification、session/claim projectionへ分離する。資格正本は少なくともsubject、qualification type/jurisdiction、`pending/verified/suspended/revoked/expired`、verified source/actor/time、effective/expiry/invalidated time、record/versionを保持し、`is_licensed_pharmacist`はその時点のderived projectionとする。生の免許番号をclaim・通常log・認可decision evidenceへ複製しない。
- service/machine identityは固定5 human roleへ偽装せず、専用principal type、最小scope、credential rotation、owner、expiry、auditを持たせる。nonclinical global/tenant jobは批准済みservice capabilityだけで実行し、薬歴記載・報告書作成・監査実行のactorにはならない。
